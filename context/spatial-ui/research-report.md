# Spatial UI — Deep Research Report

_Generated by the `spatial-ui-architecture` workflow (10 parallel web-research agents, primary-source-cited). Background/best-practice corpus that the architecture is built on._

## RealityKit Entity-Component-System architecture for building a large interactive 3D UI on visionOS

RealityKit is an ECS: Entities are lightweight containers, Components are pure data attached per entity, and Systems hold per-frame behavior. The single most important finding for this app is that the RealityView `update:` closure is NOT a per-frame tick — it fires only when SwiftUI-observed state it reads changes — and Apple (WWDC25 274) explicitly warns that mutating an Observed entity property inside it causes a recursive loop. That is exactly the drag snap-back bug: the current `reconcile()` (called from `update:`) re-asserts every card's transform from stale model data each pass, fighting the live gesture, and the `@State draggingID` guard can't work because @State mutated in a RealityKit gesture isn't reliably visible to the next update pass. The fix is a real transform-ownership model: encode ownership in a Component a System respects, mutate transforms only in gesture closures / Systems / `Entity.animate()`, and restrict reconcile to entity identity (add/remove/attach) — never live transforms. The "x" control not highlighting is a geometric/hierarchy problem: the parent card's collision box (depth max(z,0.03)) encloses the child x at +0.006 z so the parent swallows hover/input, and a SwiftUI `.hoverEffect` inside a ViewAttachment is not a reliable RealityKit hover signal — use HoverEffectComponent (.highlight/.shader) on the control entity with its own non-enclosed collider and input/hover-group isolation. visionOS 26 ships turnkey APIs that make grab-drag near-trivial (ManipulationComponent with releaseBehavior=.stay), per-entity gestures (GestureComponent) that retire the brittle name-prefix parsing, Observable entities for clean scene→SwiftUI flow, and implicit Entity.animate for smooth motion. All of this preserves the locked "SwiftUI renders content, RealityKit places it" paradigm via ViewAttachmentComponent / PresentationComponent.

### Core principles
- **The RealityView update: closure is NOT a per-frame tick — it runs only when SwiftUI-observed state it reads changes.** — "This update closure is not like an update or tick function in a game engine — those run every frame, whereas this only runs when the state it references changes." Because the current reconcile() is called from update: and reads viewModel.renderedEntities plus other @Observable fields, it re-fires on ANY observed change and rebuilds the whole scene (re-adds all edges/ports, re-sets every transform and collision box). Continuous/runtime behavior must NOT live here. This is the root of bug #1: the closure re-asserts each card to its stored (stale-during-drag) position every pass. _(src: Step Into Vision 'RealityKit Basics: update closure'; Apple Developer Forums thread 733020; WWDC23 10080)_
- **Never mutate observed entity state inside the update closure — it creates recursive feedback loops, the same failure mode as drag snap-back.** — WWDC25 274 is explicit: entities are now Observable (position/scale/rotation/children/components), so reading them in a view body creates a dependency and writing them in update: re-triggers update: -> infinite loop. The session's stated SAFE places to mutate a transform are (a) gesture closures, (b) custom Systems, and (c) Entity.animate(...). The UNSAFE place is the update closure. Transform ownership must be explicit per instant; today the model (via reconcile) AND the gesture both write the same transform, so they fight. Make exactly one authority own an entity's transform at any moment. _(src: WWDC25 'Better together: SwiftUI and RealityKit' (session 274))_
- **Per-frame / continuous behavior belongs in a custom System, queried with EntityQuery and ticked via update(context:).** — Pattern: struct FooSystem: System { static let query = EntityQuery(where: .has(FooComponent.self)); init(scene: Scene){}; func update(context: SceneUpdateContext){ for e in context.entities(matching: Self.query, updatingSystemWhen: .rendering){...} } }, then register once at launch with FooSystem.registerSystem(). updatingSystemWhen: takes .rendering (sync to display refresh — default choice) or .always; it exists because visionOS has high/variable refresh rates. Registered systems run automatically on all RealityKit content on the MainActor render loop. This is where jitter-on-highlight, smooth glide-to-layout-target, edge re-derivation, and drag inertia belong — not in reconcile. _(src: WWDC23 10080; swiftorbit.io ECS floating brick; Apple 'Implementing systems for entities in a scene'; daniloc/MyFirstECS)_
- **Decide behavior placement by lifetime: one-shot intent -> gesture handler; reactive identity/topology -> SwiftUI declarative; continuous/runtime -> System + Component.** — Rule for this codebase: (1) Gesture handlers express user INTENT and write a target into a Component (set DragStateComponent.isDragging, or LayoutTargetComponent.target). (2) SwiftUI/RealityView reconcile owns ONLY entity existence/identity (add/remove cards, attach the right ViewAttachment, set name) — it must NOT touch live transforms. (3) Systems own continuous transform evolution (glide toward layout target unless dragging; apply jitter; recompute edges from live positions). This split is what makes grab-drag trivial: the drag System/gesture owns the transform while dragging, reconcile is forbidden from overwriting it, and on release a one-shot persists to the model. _(src: WWDC25 274 (safe mutation sites); WWDC23 10080 (systems for continuous behavior))_
- **Drive SwiftUI from RealityKit through Observable entities/components, not by polling; keep model-owned data flowing the other way through your @Observable view model.** — visionOS 26 makes Entity and components Observable: reading entity.position or entity.components[Foo.self] in a view body (or via entity.observable.components[...]) creates a reactive dependency, so a System mutating that component auto-updates SwiftUI with no glue. Use withObservationTracking { } onChange: for imperative observation. Clean bidirectional contract: model->scene via your view model + reconcile (identity only); scene->SwiftUI via Observable components (e.g. a HUD reading drag state/selection). New CoordinateSpace3D / GeometryProxy3D.coordinateSpace3D() replaces manual points-to-meters/axis math. _(src: WWDC25 274; swiftorbit.io 'Sync RealityKit ECS with SwiftUI for Collision Events')_
- **InputTarget and Collision are hierarchical; child controls need explicit input/hover isolation so the parent card doesn't swallow them.** — InputTargetComponent propagates to descendants with CollisionComponents: a child IS independently targetable, BUT a descendant's allowedInputTypes overrides ancestors and isEnabled=false hides a subtree from gestures. The current bug is geometric: the parent card collider has depth max(e.z,0.03) and the x sits only +0.006 z in front, so the parent physically encloses the child and gaze/taps resolve to the parent. Fixes: flatten the parent collider z, push the control clearly outside the parent box, and/or give the control its own non-enclosed collider. HoverEffectComponent also applies to a hierarchy and propagates broadly without an explicit group. _(src: Apple 'Responding to gestures on an entity'; Apple Developer Forums thread 747256; Apple HoverEffectComponent reference)_
- **Hover/highlight must be expressed with HoverEffectComponent styles (RealityKit-visible), not a SwiftUI .hoverEffect buried inside a ViewAttachment.** — HoverEffectComponent (visionOS 1) gained styles in visionOS 2: .spotlight (default), .highlight (uniform tint via HoverEffectComponent.HighlightHoverEffectStyle(color:strength:)), and .shader (drives a ShaderGraphMaterial 'Hover State' node with an animatable 0->1 intensity — the right tool for playful jitter/glow). Hover grouping (HoverEffectGroup / GroupID, visionOS 2) controls whether grouped entities activate together; without an explicit group hover propagates broadly. The current DismissBadge uses SwiftUI .hoverEffect { effect.scaleEffect } inside a ViewAttachmentComponent — that scales the SwiftUI view in its own 2D space and is not a dependable RealityKit hover here. Use a .shader or .highlight HoverEffectComponent on the control entity (or a System reading HoverState applying jitter via a JitterComponent). _(src: WWDC24 session 10103; WWDC24 'Create custom hover effects in visionOS' (10152); WWDC25 'What's new in RealityKit' (287))_
- **For grab-and-drag specifically, prefer the built-in ManipulationComponent over a hand-rolled DragGesture+reconcile.** — visionOS 26's ManipulationComponent.configureEntity(entity, collisionShapes:[...]) auto-adds InputTarget + Collision + HoverEffect + Manipulation and gives pick-up, telescoping move, one/two-hand rotate, scale, and hand-off for free. Crucially the default releaseBehavior = .reset is exactly the snap-back symptom — set releaseBehavior = .stay to keep the moved position. Lifecycle via ManipulationEvents (WillBegin, DidUpdateTransform, WillRelease, WillEnd, DidHandOff); dynamics exposes inertia, scalingBehavior, secondaryRotationBehavior. Persist to backend on WillEnd/WillRelease. If you keep a custom gesture, the ownership rule still holds: while the manipulator owns the transform, reconcile must not write it. _(src: WWDC25 'What's new in RealityKit' (287); Step Into Vision 'Deep Dive into Manipulation on visionOS')_
- **Use Entity.animate() and content.animate {} for smooth transitions; update topology incrementally, not from scratch.** — visionOS 26 adds implicit animation for TransformComponent (and Audio/Model/Light color) via entity.animate(.easeInOut(duration:)) { entity.transform = ... } and a content.animate {} block usable in update. This replaces per-frame manual lerping for layout transitions and gives smooth 'fly to type-cluster slot' motion. Separately, the current reconcile deletes and re-adds ALL edges/ports every pass — for a scalable scene, edges should be persistent entities updated in place (or owned by an EdgeSystem repositioning a stable line/port entity per dependency), keyed by a stable id, so the scene doesn't thrash. _(src: WWDC25 274 (Entity.animate, content.animate); WWDC23 10080 (systems repositioning entities each frame))_
- **SwiftUI-renders-content / RealityKit-places-it is fully preserved via ViewAttachmentComponent + PresentationComponent + GestureComponent (all visionOS 26).** — ViewAttachmentComponent(rootView:) makes a SwiftUI view an entity (already used); PresentationComponent(presentation:.popover, binding:$x){...} attaches modals/popovers to entities (cleaner than .sheet on the volume for in-scene editing); GestureComponent(TapGesture()...) attaches SwiftUI gestures directly to an entity, removing the brittle name-prefix parsing ('ctl::role::owner', 'edge::') used today. This is the scalable replacement for the single top-level .targetedToAnyEntity() gesture + string dispatch. _(src: WWDC25 274 (ViewAttachmentComponent, PresentationComponent, GestureComponent); WWDC25 287)_

### Key APIs
- **`EntityQuery / SceneUpdateContext.entities(matching:updatingSystemWhen:)`** _(min visionOS 1.0)_ — Selects the entities a System operates on by component composition (e.g. .has(DragStateComponent.self)).
  - When: Inside a System's update to iterate matching entities; scene.performQuery(query) for ad-hoc queries outside a tick.
  - ⚠️ Predicates compose with .has, &&, ||, !. Prefer marker components (DragStateComponent, LayoutTargetComponent, JitterComponent) so queries are cheap and intent is explicit.
- **`System protocol + registerSystem() + update(context: SceneUpdateContext)`** _(min visionOS 1.0)_ — Hosts per-frame/continuous behavior; the right home for transform glide, jitter, inertia, edge updates.
  - When: For anything that must run over time rather than once on a state change. Register once at app init.
  - ⚠️ init(scene:) required; runs on MainActor render loop. updatingSystemWhen: .rendering (default) vs .always. SystemDependency orders systems relative to each other.
- **`custom Component (struct: Component [, Codable])`** _(min visionOS 1.0 (Observable components for SwiftUI: visionOS 26))_ — Pure-data runtime state owned by the SCENE, attached per entity — the clean separation from model-owned data.
  - When: Mark transform ownership (e.g. TransformOwner enum .model/.gesture/.animating), store layout targets, hover/jitter state, selection — anything a System reads/writes.
  - ⚠️ One component per type per entity. Mutate by copy-back: var c = e.components[T.self]; c.x=...; e.components[T.self]=c. registerComponent() auto for in-target Swift types; call explicitly for RCP/cross-module types.
- **`ManipulationComponent + ManipulationComponent.configureEntity(_:collisionShapes:)`** _(min visionOS 26 (visionOS 2))_ — Turnkey pick-up / move / rotate / scale / hand-off — the 'near-trivial grab-and-drag' the developer wants.
  - When: Default choice for movable cards/nodes; replaces the hand-rolled DragGesture + manual coordinate conversion.
  - ⚠️ Default releaseBehavior = .reset causes snap-back; set .stay. Persist on ManipulationEvents.WillEnd/WillRelease. dynamics can disable scaling/secondary rotation for move-only. Auto-adds InputTarget/Collision/HoverEffect.
- **`HoverEffectComponent (.spotlight / .highlight / .shader) + HoverEffectGroup`** _(min HoverEffectComponent visionOS 1.0; .highlight/.shader styles + GroupID visionOS 26)_ — RealityKit-visible hover/highlight; .shader drives a ShaderGraph HoverState node (0->1 intensity) for glass glow + jitter.
  - When: On the card AND independently on the x/control entity. .highlight for simple tint; .shader for playful animated effects; group with GroupID to control co-activation.
  - ⚠️ A SwiftUI .hoverEffect inside a ViewAttachment is NOT a reliable RealityKit hover. Without an explicit group, hover propagates across the hierarchy. Child needs its own collider not enclosed by the parent's.
- **`InputTargetComponent (isEnabled, allowedInputTypes) + CollisionComponent`** _(min visionOS 1.0)_ — Hit-testing + gesture targeting. Hierarchical: descendants are independently targetable.
  - When: Per interactive entity. Give the x-control its own non-enclosed collider; use child allowedInputTypes/isEnabled to override the parent so the card doesn't swallow the control.
  - ⚠️ Parent collider depth max(z,0.03) enclosing a child at +0.006 z routes input to the parent. CollisionFilter (group/mask, 32 groups) controls collision pairing, but input precedence is geometric + hierarchy flags, not just filters.
- **`Observable Entity + entity.animate(_:) / content.animate { }`** _(min visionOS 26)_ — Read entity/component state into SwiftUI reactively; animate transforms implicitly for smooth motion.
  - When: scene->SwiftUI HUD/state without polling; smooth glide to layout slots; release animations after a drag.
  - ⚠️ Reading entity.position in a view body creates a dependency; writing it in update: causes an infinite loop. Animate only inside gesture/system/event closures or via animate().
- **`GestureComponent + ViewAttachmentComponent + PresentationComponent`** _(min visionOS 26)_ — Per-entity SwiftUI gestures, SwiftUI-view-as-entity, and entity-anchored modals/popovers — preserves 'SwiftUI renders, RealityKit places.'
  - When: GestureComponent to drop name-prefix gesture parsing; ViewAttachmentComponent for all cards (already used); PresentationComponent for in-scene edit popovers instead of volume-level .sheet.
  - ⚠️ ViewAttachmentComponent replaces visionOS 1's builder attachments API (attachments aren't ready in make: — the current setup-race workaround). Gesture values are in the entity's coordinate space by default.

### Anti-patterns
- **Letting two authorities (model/reconcile AND gesture) write the same entity transform, then gating with a SwiftUI @State flag like draggingID.** — @State mutated inside a RealityKit gesture .onChanged is not reliably visible to the update closure before the next pass (confirmed by the failed fix). Transform ownership must be encoded in the scene (a Component a System respects), not in SwiftUI view state.
- **Mutating an Observable entity property (position/components) inside the update: closure.** — Writing observed state inside update: re-triggers update: -> recursive/infinite loop (explicitly warned in WWDC25 274). Mutate transforms only in gesture closures, Systems, or via Entity.animate().
- **Encoding a parent collider deep enough to enclose its child control, then expecting the child to receive hover/taps.** — Input/hover resolves geometrically up the hierarchy; an enclosing parent collider swallows the child. The x at +0.006 z inside a max(z,0.03) box can never reliably get its own hover. Needs a non-enclosed child collider and/or input/hover-group isolation.
- **Driving 3D highlight via a SwiftUI .hoverEffect placed inside a ViewAttachmentComponent's rootView.** — That effect operates in the attachment's 2D SwiftUI space and does not translate to a dependable RealityKit hover for an entity-targeted control. Use HoverEffectComponent (.highlight/.shader) on the entity, or a System reading HoverState.
- **Disambiguating gestures by parsing entity names ('ctl::role::owner', 'edge::') under a single top-level .targetedToAnyEntity() gesture.** — Stringly-typed dispatch doesn't scale and is fragile. Per-entity GestureComponent (visionOS 26) or typed marker components + EntityQuery give type-safe, localized handling.
- **Deleting and re-adding all edges and port spheres from scratch on every reconcile pass.** — O(edges) churn per observed change causes visible flicker and allocation pressure as the graph grows. Edges should be persistent, stably-keyed entities updated in place by an EdgeSystem reading live positions.
- **Relying on the default ManipulationComponent releaseBehavior (.reset).** — .reset returns the entity to its original transform on release — indistinguishable from the snap-back bug. For a persistent placed workspace set releaseBehavior = .stay and persist on WillEnd.

### Recommendations for our app
- Adopt an explicit transform-ownership model. Add a scene-owned component (e.g. TransformAuthorityComponent { case model, gesture, animating }). reconcile() may set a card's transform ONLY when authority == .model and the entity is newly added; while a drag/manipulation owns the entity it is .gesture and reconcile is forbidden from writing position. This is the root fix for the drag snap-back, replacing the unreliable @State draggingID guard.
- Shrink reconcile() to identity-only. It should add/remove card entities, attach the correct ViewAttachmentComponent, set the stable name/id, and set persistent components ONCE on creation. Remove the per-pass card.position re-assert (lines ~245-250 of SpatialSceneView.swift) and the per-pass CollisionComponent rebuild from visualBounds — set collision once and update only on real geometry change.
- Replace the hand-rolled DragGesture with ManipulationComponent.configureEntity(card, collisionShapes:[...]) and set releaseBehavior = .stay. Subscribe to ManipulationEvents.WillEnd to call viewModel.endDrag and persist. Disable scaling/secondary-rotation via dynamics if you want move-only. This makes grab-drag near-trivial and eliminates manual coordinate conversion.
- Introduce custom Systems for continuous behavior: a LayoutSystem that glides un-hand-moved entities toward their type-cluster layout target (LayoutTargetComponent) using Entity.animate or per-frame lerp, skipping entities whose authority is .gesture; an EdgeSystem that repositions persistent, stably-keyed edge/port entities from live positions instead of delete-and-re-add; and a JitterSystem that reads HoverState to apply the playful jitter on highlight.
- Fix the x control hover at the root: give the control its own entity with a small, non-enclosed CollisionComponent placed clearly in front of the card, set the parent card collider to a shallow z (don't inflate to max(z,0.03) so it doesn't enclose the child), and use a HoverEffectComponent (.highlight or .shader) ON the control entity. Use HoverEffectGroup/GroupID to isolate the control's hover from the card. Remove the SwiftUI .hoverEffect inside DismissBadge — it doesn't drive RealityKit.
- Retire name-prefix gesture parsing. Move to per-entity GestureComponent (visionOS 26) so each card/control/volume carries its own tap/drag handler, and use typed marker components instead of 'ctl::', 'edge::' string prefixes for dispatch and EntityQuery filtering.
- Wire scene->SwiftUI via Observable entities/components rather than polling: a SelectionComponent or DragStateComponent read by SwiftUI HUD/banners auto-updates with no glue. Keep model->scene flowing through the existing @Observable view model + identity-only reconcile. Use CoordinateSpace3D/GeometryProxy3D.coordinateSpace3D() for any remaining coordinate conversions.
- Use PresentationComponent(.popover, binding:) anchored to the tapped entity for in-scene edit/note editing instead of volume-level .sheet, keeping editing spatially attached to its card. Keep the locked SwiftUI-renders / RealityKit-places paradigm via ViewAttachmentComponent for all card visuals and ShaderGraphMaterial for glass + hover glow.
- Keep the pure nonisolated SpatialLayoutEngine exactly as-is — it computes layout targets that a MainActor LayoutSystem applies. The engine stays Foundation/simd-only and unit-tested; the System is the only thing that touches entities. This satisfies SWIFT_DEFAULT_ACTOR_ISOLATION=MainActor while keeping the engine testable.
- Verify with xcodebuild -scheme TaskPlannerVision -destination 'generic/platform=visionOS Simulator' (SourceKit reports false errors). Note these are visionOS 26 APIs (ManipulationComponent, GestureComponent, PresentationComponent, Observable entities, .highlight/.shader hover, HoverEffectGroup) — fine for the 26.2 target; the System/Component/EntityQuery/HoverEffectComponent/InputTarget foundations are battle-tested since visionOS 1.

## Reconciling a declarative data model into a retained 3D scene graph without a redraw-everything-every-pass loop (visionOS 26 / RealityKit, with cross-engine prior art from React-Three-Fiber, Unity, Bevy, Needle)

The current architecture has a structural inversion: a single `reconcile()` runs on EVERY RealityView `update` pass and re-asserts every property of every entity from scratch (position, collision, controls, all edges), which makes it the authority over transforms it does not actually own during a live gesture or animation. This is exactly the failure mode Apple now warns about: in visionOS 26 (WWDC25 "Better together: SwiftUI and RealityKit", session 274) entities are Observable, so reading entity state in `update` creates a SwiftUI dependency and mutating entities there can loop; Apple's explicit guidance is "Don't modify observed state in the update closure — modify in gesture closures, custom Systems, or the make closure." Every retained-scene reconciler studied (React-Three-Fiber, Unity DOTS Hybrid Renderer, Bevy ECS, Needle) shares one principle: diff and write ONLY the properties that changed, and never re-touch a value that another authority (gesture/physics/animation) currently owns — R3F's `applyProps` even filters identical props before writing. The canonical fix for the drag snap-back is to stop letting the reconciler own transforms: adopt visionOS 26's `ManipulationComponent` (which owns the transform during manipulation and keeps it on release via `releaseBehavior = .stay`), or at minimum cache the start transform on first `onChanged` and gate the reconciler with a per-entity "owned-by-gesture" dirty flag — not view `@State`, which isn't reliably visible to the update closure mid-gesture. The x-control hover bug is a hierarchical-collision/input-precedence problem: the parent card's CollisionComponent encloses the child badge (only +0.006 m away) and swallows its hover/tap; the fix is per-entity `GestureComponent` + `HoverEffectComponent` on the badge with non-overlapping collision (and the card itself should not enclose its controls). visionOS 26 provides first-class primitives — Observable entities, `GestureComponent`, `ManipulationComponent`, `PresentationComponent`, `Entity.animate`/`content.animate` — that let you replace the imperative reconcile loop with targeted, ownership-respecting updates plus an interpolation layer (lerp/critically-damped spring) between the data-model target transform and the presented transform.

### Core principles
- **A reconciler must write ONLY the properties it owns, and skip any property currently owned by a live gesture, physics, or animation** — This is the root cause of the snap-back. Our reconcile() re-sets card.position from the view-model's STORED position on every update pass; during a drag that stored value is stale (written only on release), so the card is yanked back to its tray slot each frame and jerks back on release. Every studied engine obeys ownership: in Unity DOTS the transform is synced from the data component (LocalToWorld) only when that component is the source of truth, and the guidance is to 'avoid writing anyways even if the value is the same' so change-versioning stays meaningful; R3F's applyProps filters identical props and skips no-op writes. The transform of an entity has exactly ONE authority at any instant — model, gesture, or animation — and the reconciler must defer to the live one. _(src: WWDC25 session 274 'Better together: SwiftUI and RealityKit'; Unity DOTS Hybrid Renderer change-filter docs; R3F applyProps technical breakdown (codyb.co))_
- **In visionOS 26 the RealityView update closure is part of SwiftUI body evaluation; reading observed entity state there creates a dependency and mutating entities there causes loops** — Apple's session 274 states this directly and shows the anti-pattern: setting entity.position inside `update` when entities are Observable creates an infinite re-evaluation loop. The recommended places to mutate entities are: the make closure (one-time setup, no dependency), gesture closures, and custom Systems. This is the framework-level statement of why a redraw-everything reconcile loop is the wrong foundation on visionOS 26. _(src: WWDC25 session 274 'Better together: SwiftUI and RealityKit'; tomkrikorian visionOS 26 AGENTS.md ('Do not implement reconciliation loops in the update closure'))_
- **Stable, keyed entity identity is what makes minimal-mutation diffing possible — match by stable id, mutate in place, never recreate** — React reconciliation uses stable keys so it can tell which items were added/removed/moved and preserve instance state instead of tearing down and rebuilding. R3F leans on this: <mesh/> maps to a persistent THREE.Mesh, and only commitUpdate mutates the existing instance — geometries/materials are never recreated on a prop change. Our code already has stable ids (SpatialEntity.id == attachment id == entity.name), which is the right foundation; the problem is we rebuild side-effects (collision, controls, ALL edges) every pass instead of diffing against what's already in the scene. _(src: React legacy reconciliation docs; R3F technical breakdown (codyb.co); pmndrs/react-three-fiber)_
- **Dirty-flag / change-detection systems exist precisely so you process only what changed, not the whole world every frame** — Bevy's Changed<T> query and Ref<T> yield only entities whose component was mutated (flags double-buffered across two frames); Unity DOTS uses chunk-level change versions bumped on write so a filtered query touches only modified chunks. The lesson for us: maintain an explicit dirty set (e.g. ids whose model transform changed since last applied, plus an edges-dirty flag) so a reconcile pass writes a handful of transforms and rebuilds edges only when topology/positions actually moved — instead of O(n) full rewrite + full edge teardown every observed change. _(src: Bevy Unofficial Cheat Book (change-detection); Bevy DetectChanges docs.rs; Unity DOTS chunk versioning (5argon))_
- **Put an interpolation layer between the target transform (data model) and the presented transform (scene)** — Engines separate 'where it should be' from 'where it is drawn'. Instead of hard-setting card.position to the model value each pass (which fights gestures and looks jerky), drive the presented transform toward the target with a frame-rate-independent damped follow (SmoothDamp / critically-damped spring) so layout changes and returns-to-tray animate smoothly and never overshoot. visionOS 26 gives this natively via Entity.animate(to:relativeTo:animation:) and content.animate { } — set the TARGET, let RealityKit interpolate. This also cleanly decouples the auto-layout engine output (target) from live manipulation (presented), removing the snap-back by construction. _(src: Rory Driscoll 'Frame Rate Independent Damping using Lerp'; theorangeduck 'Spring-It-On'; Unity Vector3.SmoothDamp; WWDC25 session 274 (Entity.animate / content.animate))_
- **Hierarchical collision and input precedence must be designed, not accidental — a parent collider that encloses a child swallows the child's hover/tap** — HoverEffectComponent applies to a hierarchy and requires the collider it intersects to belong to the intended entity. Our x-badge sits only +0.006 m in front of a card whose CollisionComponent depth is max(z,0.03), so the parent box geometrically contains the badge and intercepts the gaze ray / pinch first; the badge's SwiftUI .hoverEffect never fires. Fix: give controls their own non-overlapping collision sitting clearly in front of the card, drive the highlight with a RealityKit-visible HoverEffectComponent (not only a SwiftUI scaleEffect inside the attachment), and scope each control's behavior with its own GestureComponent rather than name-parsing one global gesture. _(src: Apple HoverEffectComponent reference; askwwdc hover-effect; stepinto.vision GestureComponent article; WWDC25 session 287 'What's new in RealityKit')_

### Key APIs
- **`RealityView make / update / attachments closures`** _(min visionOS 1.0 (RealityView); Observable-entity loop caveat is visionOS 26)_ — make = one-time setup (no SwiftUI dependency); update = react to discrete state changes only; attachments = declare SwiftUI views surfaced as ViewAttachmentEntity
  - When: Use make for initial entity creation/placement. Use update SPARINGLY for genuinely changed structure. Do NOT run a full reconcile-everything pass here on visionOS 26.
  - ⚠️ On visionOS 26 the update closure participates in body evaluation; reading Observable entity state creates a dependency and mutating entities here can loop. This is the framework's own warning against our current pattern.
- **`ManipulationComponent + ManipulationComponent.configureEntity(_:hoverEffect:allowedInputTypes:collisionShapes:)`** _(min visionOS 26 (2.6))_ — One-component drag/rotate/scale/two-hand/hand-off with built-in audio and hover; OWNS the entity transform during manipulation
  - When: This is the canonical fix for grab-and-drag. Replace the hand-rolled DragGesture + transform caching for moving nodes. Set releaseBehavior = .stay to keep the entity where dropped.
  - ⚠️ Default releaseBehavior is .reset (snaps back) — you MUST set .stay. configureEntity auto-adds Collision+InputTarget+HoverEffect. Persist via ManipulationEvents.WillEnd; constrain (clamp to bounds) via DidUpdateTransform — constrain, don't replace. Other events: WillBegin, WillRelease, DidHandOff. dynamics lets you disable rotation/scaling if you only want translation.
- **`GestureComponent(_:) on an entity`** _(min visionOS 26)_ — Attach a SwiftUI gesture (Tap/Drag/etc, targetedToEntity) directly to a specific entity so each entity carries its own handler
  - When: Replace the two global .gesture(...).targetedToAnyEntity() + name-parsing ('ctl::', 'edge::') dispatch. Give the card, the x-badge, ports, and volumes their own gestures.
  - ⚠️ Entity still needs CollisionComponent + InputTargetComponent. Eliminates the brittle string-prefix routing entirely.
- **`Entity is Observable / entity.observe(_:) / entity.observable`** _(min visionOS 26)_ — Two-way data flow: SwiftUI views re-render on entity property changes; code can observe entity property changes
  - When: Use to surface live entity transforms back into SwiftUI (e.g. inspector showing live position) and to let small self-contained views depend on a single entity rather than the whole scene.
  - ⚠️ The flip side of the loop warning: reading entity.position inside the update/body closure creates a dependency. Keep observation in small leaf views and out of the reconcile path.
- **`Entity.animate(to:relativeTo:animation:speed:) and content.animate(animation:) { }`** _(min visionOS 26 (Entity.animate); content.animate also 26)_ — Animate Transform/audio/model/light component changes with SwiftUI-style animations (e.g. .bouncy) — set the target, RealityKit interpolates
  - When: This is the built-in interpolation layer: animate layout-engine placements, return-to-tray, and pop-out instead of hard-setting positions each pass.
  - ⚠️ move(to:) / Entity.animate is preferred over animating inside the update closure; animating in update can still interact with the body-evaluation/observation rules.
- **`HoverEffectComponent(.highlight / .spotlight / .shader)`** _(min visionOS 1.0 (HoverEffectComponent); spotlight/shader styles visionOS 2.0)_ — RealityKit-native gaze/hover feedback applied across an entity hierarchy (highlight tint, spotlight, or shader-graph-driven intensity)
  - When: Drive the x-badge and node highlight here (plus a 'jitter' via Entity.animate on hover) so the effect is RealityKit-visible, not a SwiftUI scaleEffect buried in an attachment that doesn't render.
  - ⚠️ Effect targets the entity whose collider the gaze ray hits FIRST — an enclosing parent collider steals it. spotlight/shader styles are richer than the default .highlight. Custom hover-state shader needs a shader-graph hover node.
- **`ViewAttachmentComponent(rootView:)`** _(min visionOS 1.0 (attachments builder); ViewAttachmentComponent visionOS 2.6)_ — Render a SwiftUI view as an entity in 3D (the locked 'SwiftUI renders content' paradigm)
  - When: Keep using it for cards/panels/notes/controls. In visionOS 26 you can create attachments imperatively as components rather than only via the attachments builder, giving finer control over when/where they appear.
  - ⚠️ Was the source of the 'setup race' (attachments not ready in make). The component form (visionOS 2.6+) avoids relying on RealityViewAttachments timing.
- **`PresentationComponent(isPresented:configuration:content:)`** _(min visionOS 26)_ — Present SwiftUI popovers/sheets anchored to a RealityKit entity from within the scene
  - When: Consider for in-scene node editing / context menus instead of routing everything through top-level .sheet bindings keyed off selectedEntityID.
  - ⚠️ New API; validate against your volumetric window before committing UX to it.
- **`DragGesture caching pattern (start transform on first onChanged, apply translation as delta, clear on onEnded)`** _(min visionOS 1.0)_ — The pre-26 canonical drag: store entity start position once, add value.convert(translation) delta, never re-read live position mid-drag
  - When: Fallback if you cannot adopt ManipulationComponent. Combine with a per-entity 'gesture owns transform' flag so the reconciler skips that entity.
  - ⚠️ Caching the START is essential — reading live position each frame compounds error. The flag MUST live where the update closure can read it reliably (entity component or @Observable model field), NOT view @State, which is not reliably visible to the update closure between a gesture onChanged and the next reconcile (our failed fix).

### Anti-patterns
- **A single reconcile(content:attachments:) that re-asserts EVERY property of EVERY entity on every update pass** — Update fires on any observed @Observable change, so this is effectively a full redraw loop. It makes the reconciler the de-facto owner of transforms it doesn't own during gestures/animations (root cause of drag snap-back), and on visionOS 26 reading/writing Observable entities here risks infinite body re-evaluation. Apple explicitly says not to modify observed entity state in update.
- **Re-setting card.position from the data model's STORED position while a drag is in progress** — The stored model position is stale until drag-end, so each pass yanks the card back to its tray/wall slot and it jerks back on release. Transform ownership must transfer to the gesture for the duration of the drag; the reconciler must defer.
- **Using a view @State flag (draggingID) inside the update closure to skip the dragged entity** — @State mutated inside a RealityKit gesture .onChanged is not reliably visible to the update closure before the next reconcile pass — the dirty/ownership state must live in the entity (a component) or the @Observable model, not transient view state.
- **Deleting and re-adding ALL edges and ports from scratch every pass** — O(E) teardown/rebuild per observed change causes churn, flicker, and lost per-edge state, and defeats any change-detection. Edges should be diffed (add/remove/move only what changed) and their endpoints driven by live entity transforms, ideally rebuilt only when an edges-dirty flag is set.
- **A parent card CollisionComponent (depth max(z,0.03)) that encloses its child x-badge sitting only +0.006 m in front** — The gaze ray / pinch hits the enclosing parent collider first, so the badge never receives hover or tap; its SwiftUI .hoverEffect scaleEffect never fires. Hierarchical collision/input precedence must be designed so controls have their own clearly-separated, front-facing colliders.
- **Driving highlight only via SwiftUI .hoverEffect { scaleEffect } inside a ViewAttachmentComponent rootView** — That SwiftUI hover state does not reliably translate to a RealityKit-visible transform on the placed entity. Use a RealityKit HoverEffectComponent (highlight/spotlight) and/or an Entity.animate jitter so the effect is visible in 3D.
- **One global gesture with targetedToAnyEntity() dispatched by parsing entity names ('ctl::role::owner', 'edge::')** — String-prefix routing is brittle and centralizes all interaction logic. visionOS 26 GestureComponent lets each entity carry its own scoped gesture, removing the parser and the name-as-protocol coupling.
- **Hard-setting transforms to the layout-engine output with no interpolation layer** — Snaps and fights live manipulation. Separate target (model/layout) from presented (scene) and interpolate via Entity.animate or a damped follow so layout, return-to-tray, and pop-out are smooth and never overshoot.

### Recommendations for our app
- ADOPT visionOS 26 ManipulationComponent for all movable nodes/volumes/notes. Use ManipulationComponent.configureEntity(entity, hoverEffect:..., collisionShapes:...) and set releaseBehavior = .stay. Persist the new transform in ManipulationEvents.WillEnd (call viewModel.endDrag), and clamp to VolumeMetrics bounds in DidUpdateTransform by constraining the value, not replacing it. This deletes the hand-rolled DragGesture, the start-position caching, and the draggingID problem outright — grab-and-drag becomes near-trivial as the developer expects.
- INVERT the reconciler from 'write everything every pass' to 'diff and write only what changed'. Maintain explicit dirty sets in the @Observable model: addedIds, removedIds, transformDirtyIds (model-driven moves only), and an edgesDirty flag. In update: add new attachments, remove gone ones, and for transformDirtyIds animate the entity toward the target via Entity.animate(...). Never touch an entity currently owned by ManipulationComponent (check for an in-progress manipulation / a per-entity 'manipulating' component flag).
- MOVE transform-ownership state out of view @State and into the entity/model. The single source of truth for 'who owns this transform right now' should be a RealityKit component on the entity (e.g. a small TransformOwner component: .model / .gesture / .animating) or an @Observable model field — never SwiftUI @State, which the failed draggingID fix proved is not reliably visible to update mid-gesture.
- SEPARATE target transform from presented transform. The SpatialLayoutEngine produces TARGETS; the scene interpolates to them via Entity.animate (built-in) or a frame-rate-independent critically-damped follow. This makes auto-layout, return-to-tray, and pop-out animate smoothly and structurally removes any fight between layout output and live manipulation.
- FIX the x-control with proper hierarchical collision + per-entity input. Give the badge its own collider clearly in FRONT of the card (enough z-separation that the card's collider does not enclose it, or shrink the card collider so it excludes the control region), attach a RealityKit HoverEffectComponent (.highlight or .spotlight) for visible hover, add an Entity.animate scale 'jitter' on WillBegin/hover for the playful feel, and route its tap via its own GestureComponent instead of name-parsing.
- REPLACE name-prefix gesture dispatch with GestureComponent per entity. Cards get a tap-to-edit gesture; the x-badge gets a tap-to-dismiss; ports/edges get none (or their own). This removes the 'ctl::'/'edge::' string protocol and the central tapGesture/dragGesture switch, scaling cleanly as new entity kinds and controls are added.
- REBUILD edges incrementally and as a managed sub-graph. Key each edge by a stable composite id (sourceId+targetId+kind), diff against existing edge entities (add/remove/reposition only deltas), and update endpoints from live entity transforms each frame only while an endpoint is moving — gate full rebuild behind edgesDirty. Avoid full teardown/re-add.
- KEEP the locked paradigm and the pure layout engine; this is purely a client interaction-architecture change. Continue SwiftUI-renders-content via ViewAttachmentComponent (consider the imperative component form to kill the setup-race), keep the nonisolated unit-tested SpatialLayoutEngine producing targets, and verify with xcodebuild -scheme TaskPlannerVision -destination 'generic/platform=visionOS Simulator' (trust xcodebuild over SourceKit). Add Swift unit tests for the new dirty-diff reducer (pure function: previous scene state + new model -> mutation list) so the reconciler logic is testable without the simulator.
- CONSIDER a small ECS-style System for continuous work (edge endpoint following, hover jitter, damped transform follow) instead of doing it in the update closure — Apple and the visionOS 26 AGENTS guidance both push continuous behavior into custom Systems, keeping update for discrete structural changes only.

## visionOS/RealityKit gestures and direct object manipulation (grab-and-drag, ManipulationComponent, hover/highlight, coordinate spaces)

visionOS offers two paths for moving entities: hand-rolled SwiftUI gestures targeted to RealityKit entities (visionOS 1.0+), and the new visionOS 26 ManipulationComponent / .manipulable that bundle move/rotate/scale/handoff. Both require entities to carry InputTargetComponent + CollisionComponent. The single biggest lesson for our app is that drag must be done by converting the gesture TRANSLATION DELTA (value.translation3D) added to a transform captured on the first onChanged event, never by teleporting to value.location3D (which is the pinch point on the surface, not the origin, and is the documented cause of grab-time snapping). Our drag snap-back (bug #1) is fundamentally a transform-OWNERSHIP problem: the reconcile pass re-asserts the stale stored position every RealityView update, which Apple explicitly warns against — observed entity state must not be mutated in the update closure, and an entity under active gesture or animation must be exempt from reconcile via an explicit ownership flag/component, not a view @State the update closure cannot observe in time. Our x-control hover failure (bug #2) is because a SwiftUI .hoverEffect inside an attachment does not drive RealityKit-visible highlight and the parent card's collision box encloses the child control; the fix is a real RealityKit HoverEffectComponent on the control with its own visionOS 26 GroupID (to decouple from parent hover propagation) and its own non-occluded collider. visionOS 26 also adds GestureComponent (per-entity gestures, eliminating our name-prefix string dispatch), the CoordinateSpace3D protocol for unified SwiftUI/RealityKit conversion, and Entity.animate/content.animate spring animations that let transitions and the desired playful "jitter" own the transform cleanly. Indirect gaze+pinch is the comfortable primary input model and pairs naturally with translation-delta drag.

### Core principles
- **Transform ownership must be explicit: at any instant exactly one authority (data model, live gesture, or animation) owns an entity's transform** — Our reconcile re-asserts the data-model position every RealityView update pass, fighting the live gesture (whose new position isn't written until release). Apple explicitly warns: do NOT modify observed entity state inside the update closure (it causes a feedback loop). The fix is an ownership handshake: when a drag/animation begins, mark the entity as owned (a custom component or a Set on the view-model that reconcile reads); reconcile then SKIPS owned entities entirely; on release the gesture writes the final transform to the model and relinquishes ownership. A view @State set inside a RealityKit gesture .onChanged is NOT reliably visible to the next update pass — ownership must live in observed model state or an entity component, never view @State. _(src: WWDC25 Session 274 'Better together: SwiftUI and RealityKit' (Avoiding Infinite Loops / safe places to modify observed state))_
- **Drag by converted TRANSLATION DELTA from a stored start transform, never teleport to location3D** — value.location3D is the gesture point ON the entity (where the pinch landed), not the entity origin — assigning it directly makes the entity jump/snap on grab. Robust recipe: on first onChanged capture sourceTransform = entity.transform; then entity.position = sourceTransform.translation + value.convert(value.translation3D, from: .local, to: entity.parent!); clear sourceTransform on .onEnded (and there persist to SpatialEntity). This verbatim pattern (from simonbs/ImmersiveMoveAndRotate) is exactly what our code is missing. _(src: simonbs/ImmersiveMoveAndRotate EntityMovementViewModifier.swift; Step Into Vision 'Drag Gesture' (documents the location3D snap caveat))_
- **Hover/highlight must be a RealityKit HoverEffectComponent, and hierarchy is controlled by GroupID** — A SwiftUI .hoverEffect{} buried in an attachment's rootView does not reliably produce RealityKit-visible scaling. Use a real HoverEffectComponent on the entity. visionOS 26's GroupID controls propagation: by default child entities inherit the parent's hover; assigning distinct GroupIDs to the card vs the 'x' control decouples them so the control highlights independently. Combined with separating the control's collider from the card's box (which currently encloses the +0.006z child and swallows its input), this fixes bug #2. _(src: WWDC25 Session 287 'What's new in RealityKit' (HoverEffectComponent GroupID))_
- **Animations are a transform-ownership state, not a fire-and-forget side effect** — visionOS 26 drives RealityKit transforms with SwiftUI Animation curves via entity.animate(with:) / content.animate(with:). Transitions (snap-to-layout, return-to-tray, the playful 'jitter' on highlight) should run as animations that temporarily OWN the transform, with reconcile suppressed for that entity until the animation completes — otherwise model and animation fight. Jitter = a short spring on position/scale; Transform is implicitly animatable. _(src: WWDC25 Session 274 (SwiftUI-Driven Animation: Entity.animate / content.animate spring))_
- **Gesture-targetable entities need BOTH InputTargetComponent and CollisionComponent; coordinate spaces differ across frameworks** — Every targetable entity needs InputTargetComponent + CollisionComponent (ManipulationComponent.configureEntity adds these + HoverEffect automatically). RealityKit is meters, y-up, +z toward the user; SwiftUI gesture values are points with inverted y, so convert() is mandatory. visionOS 26 unifies this via the CoordinateSpace3D protocol (Entity and Scene conform). InputTargetComponent.allowedInputTypes selects direct (reach-out) vs indirect (gaze+pinch). _(src: WWDC25 Session 274; Apple 'Transforming RealityKit entities using gestures'; WWDC24 Session 10094)_
- **Indirect (gaze+pinch) is the primary, comfortable input model — design for amplified small movements** — Indirect input (look + pinch at a distance) is the default and most comfortable visionOS modality because small hand motions are amplified into large actions; direct input (reach out and touch) suits nearby objects. Translation-delta drag naturally supports indirect input; teleport-to-location does not. Apps cannot read gaze directly (privacy), so the hover highlight is the only cue of what will receive the pinch — making a working HoverEffectComponent functionally important, not cosmetic. _(src: WWDC24 Session 10094 'Explore game input in visionOS'; Unity PolySpatial visionOS Hover Effects docs (gaze-privacy rationale))_

### Key APIs
- **`DragGesture().targetedToAnyEntity() + EntityTargetValue`** _(min visionOS 1.0)_ — Hand-rolled, fully-controllable drag. EntityTargetValue gives value.entity (hit entity), value.translation3D, value.location3D, and value.convert(_:from:to:) for coordinate conversion.
  - When: Use as our PRIMARY drag mechanism for cards. Gives total control over clamping to VolumeMetrics bounds, layout-engine snap, and persist-on-release to SpatialEntity — control ManipulationComponent's opinionated behavior does not expose.
  - ⚠️ Requires InputTargetComponent + CollisionComponent. Use value.translation3D (delta) + stored start transform, NOT value.location3D (teleport = snap). .targetedToAnyEntity() serves all entities via one gesture; disambiguate by value.entity identity (entity reference / component lookup), far more robust than name-prefix string parsing.
- **`ManipulationComponent + ManipulationComponent.configureEntity(_:collisionShapes:hoverEffect:allowedInputTypes:)`** _(min visionOS 26.0)_ — One component providing move (one hand), rotate (one/two hands), scale (two-hand pinch-drag), and hand-to-hand passing, with built-in hover and audio. configureEntity also adds InputTarget, Collision, and HoverEffect components.
  - When: Strong candidate for workflowVolume and note entities where free 6DoF move/rotate/scale is desirable. Less ideal for taskNode/stepNode cards that must clamp to bounds and snap to type-cluster slots, since its translation/clamping is internal.
  - ⚠️ Default releaseBehavior = .reset animates back to origin (a built-in snap-back). Set releaseBehavior = .stay to persist, then write the final transform to SpatialEntity in a ManipulationEvents.WillEnd handler. Set audioConfiguration = .none to suppress default sounds.
- **`ManipulationEvents (WillBegin, DidUpdateTransform, WillRelease, WillEnd, DidHandOff)`** _(min visionOS 26.0)_ — Lifecycle hooks for ManipulationComponent; subscribe via content.subscribe(to: ManipulationEvents.WillBegin.self){...}. WillBegin=picked up, WillEnd=at rest after release, DidUpdateTransform=per-frame, DidHandOff=passed between hands.
  - When: Use WillBegin to claim transform ownership (suppress reconcile for that entity) and WillEnd to release ownership and persist the final transform via tRPC — the ManipulationComponent equivalent of our onChanged/onEnded ownership handshake.
  - ⚠️ Canonical physics handoff: WillBegin sets PhysicsBodyComponent.mode = .kinematic, WillEnd sets .dynamic — only relevant if we add gravity/drop physics (likely unnecessary for clamped cards).
- **`GestureComponent`** _(min visionOS 26.0)_ — Attaches a SwiftUI gesture directly to a RealityKit entity (entity.components.set(GestureComponent(TapGesture().onEnded{...}))) instead of a top-level .gesture. Gesture values are reported in the entity's coordinate space by default.
  - When: Cleanest way to give each entity (card, x-control, volume) its OWN gestures, replacing our single top-level gesture + name-prefix dispatch. Per-entity gestures give per-entity hit-testing and remove the disambiguation hack.
  - ⚠️ Still requires InputTargetComponent + CollisionComponent (Apple's explicit pro tip). Reports in entity-local space, so convert when writing scene/parent positions.
- **`.manipulable(transform:coordinateSpace:operations:inertia:isEnabled:) SwiftUI modifier`** _(min visionOS 26.0)_ — SwiftUI-side manipulation for views presenting 3D content (notably Model3D). With a transform: binding it persists ('stays where the user leaves it'); operations restricts to [.translation,.primaryRotation,.secondaryRotation,.scale]; inertia adds weight; event closure exposes .active/.ended phases.
  - When: Limited use for us: docs note it targets Model3D and that other views get clipped by window/volume bounds. Our cards are SwiftUI rendered as RealityKit attachments, so RealityKit-side ManipulationComponent or hand-rolled gestures fit better.
  - ⚠️ Default (no transform binding) auto-resets on release — same reset-vs-persist trap. Clipping makes it unsuitable for free-roaming cards in a volume.
- **`HoverEffectComponent (with GroupID) + InputTargetComponent.allowedInputTypes`** _(min HoverEffectComponent visionOS 1.0; GroupID visionOS 26.0)_ — HoverEffectComponent gives system-rendered highlight (e.g. .spotlight) on gaze/hover. visionOS 26 GroupID controls hierarchy propagation (default: children inherit parent; distinct GroupIDs decouple). allowedInputTypes selects direct/indirect/all.
  - When: Use a real HoverEffectComponent on cards and controls instead of SwiftUI .hoverEffect inside attachments. Give the x-control a distinct GroupID from its parent card so it highlights independently — direct fix for bug #2. Pair with a jitter spring via Entity.animate.
  - ⚠️ Works only if the entity has its own non-occluded CollisionComponent. The parent card's collision box (depth max(z,0.03)) encloses the child x at +0.006z and swallows input — offset the control outward, shrink the parent box, or use input-target precedence so the child is independently hittable.

### Anti-patterns
- **Re-asserting the data-model transform every RealityView update pass (our reconcile re-sets card.position each pass)** — Apple explicitly warns against modifying observed entity state inside the update closure — it creates a feedback loop and fights any concurrent gesture/animation. While a gesture writes the transform, reconcile overwrites it with the stale stored position, producing the jerky snap-back. Reconcile must skip entities whose transform is currently owned by a gesture/animation (via an explicit ownership flag/component in observed state), not via a view @State the update closure cannot see in time.
- **Setting entity.position = value.convert(value.location3D, ...) (teleport-to-location drag)** — location3D is the pinch point on the entity surface, not its origin, so the entity jumps to the cursor on grab. Documented snap cause. Use translation3D delta + stored start transform instead.
- **Disambiguating gesture targets by parsing entity names ('ctl::role::ownerId', 'edge::')** — Stringly-typed dispatch is brittle and will not scale. Replace with per-entity GestureComponent (visionOS 26) or component-based identity (a custom EntityRole/EntityRef component read via value.entity.components[...]), giving type-safe routing and per-entity hit-testing.
- **SwiftUI .hoverEffect{ scaleEffect } inside a ViewAttachmentComponent rootView to highlight a 3D control** — The attachment's SwiftUI hover state does not drive RealityKit-visible transform changes reliably, and the parent card's collision swallows the child's hover. Use a RealityKit HoverEffectComponent on the control entity with its own GroupID and its own non-occluded collider.
- **Rebuilding ALL edges/ports and the x-control from scratch every reconcile pass** — Delete-and-re-add every frame thrashes the entity graph, invalidates collision/hover state mid-interaction, and is O(n) churn that won't scale. Edges/controls should be diffed and updated in place (move endpoints, toggle visibility) by a custom RealityKit System that updates edge endpoints from live entity positions each frame, not by recreating entities.
- **Leaving ManipulationComponent / .manipulable at default releaseBehavior (.reset)** — Default behavior animates the object back to its origin on release — a built-in snap-back. If we adopt ManipulationComponent we MUST set releaseBehavior = .stay (or bind a transform) and persist via WillEnd, or we reproduce our current bug through the framework.

### Recommendations for our app
- Adopt an explicit transform-ownership model. Add an ownership authority per entity (e.g. an InteractionState component or an observed Set<EntityID> on the view-model: .model / .gesture(id) / .animating). reconcile() reads it and SKIPS any entity not owned by .model. Drag onChanged sets ownership to .gesture; onEnded writes the final transform to SpatialEntity via tRPC and returns ownership to .model. This is the root fix for bug #1 and replaces the non-working view @State draggingID guard.
- Rewrite card drag with the canonical delta recipe: capture sourceTransform on first onChanged, set position = sourceTransform.translation + value.convert(value.translation3D, from: .local, to: entity.parent!), clamp to VolumeMetrics, persist on onEnded. Never use value.location3D. This makes grab-and-drag the near-trivial operation the developer expects.
- Fix the x-control hover (bug #2) with a real RealityKit HoverEffectComponent on the control entity, give it a distinct visionOS 26 HoverEffectComponent GroupID so it does not inherit/share the card's hover, and stop relying on SwiftUI .hoverEffect inside the attachment. Ensure the control has its own collider that is not enclosed by the parent card's collision box (offset it on +z beyond the card depth, or shrink the card collider, or set input-target precedence).
- Migrate from one top-level gesture + entity-name string parsing to per-entity GestureComponent (visionOS 26) and component-based identity (a typed EntityRole component). This removes the brittle 'ctl::'/'edge::' parsing and gives type-safe, per-entity hit testing that scales as kinds grow.
- Make all motion animation-owned: use entity.animate(with: .spring(...)) for layout snaps, return-to-tray, and the playful highlight 'jitter', and suppress reconcile for the entity for the animation's duration. This delivers the smooth-animation and jitter requirements without model/animation conflict.
- Stop delete-and-re-adding edges/ports and controls every reconcile pass. Create them once and update them in place via a small custom RealityKit System that recomputes edge endpoints from live entity positions each frame; reconcile should diff (add/remove/show-hide) rather than rebuild. This is required for the system to scale.
- Evaluate ManipulationComponent specifically for workflowVolume and note entities (free 6DoF feel, hand-to-hand passing) while keeping the hand-rolled translation-delta drag for taskNode/stepNode cards that must clamp and snap to layout slots. If ManipulationComponent is used anywhere, set releaseBehavior = .stay and persist in WillEnd. Both approaches must share the same ownership model so reconcile treats them uniformly.

## Hover / Highlight / Focus / Selection feedback in visionOS & RealityKit (with direct fixes for the "x doesn't highlight" bug and drag snap-back)

visionOS draws a hard line between TWO completely different feedback systems that our code currently conflates. (1) System gaze hover: when an entity has InputTargetComponent + CollisionComponent + HoverEffectComponent, the OS highlights it on gaze entirely OUTSIDE the app process for privacy — the app is never told what is being looked at, so app code can never read a live "is hovered" boolean and use it to drive a transform. (2) App selection state: app-driven highlight (e.g. our isPicked merge/link selection) which the app fully controls and can animate. Our "x doesn't highlight" bug has two root causes confirmed by primary sources: first, the parent card's CollisionComponent (depth max(z,0.03)) encloses the ×-badge child which sits only +0.006 m forward, and per a RealityKit engineer "HoverEffectComponent only affects its Entity and won't affect its children" — so hover does not naturally cascade and the parent's larger collision wins the gaze/hit-test, swallowing the child; second, the SwiftUI .hoverEffect{ scaleEffect } inside the ViewAttachmentComponent rootView is applied by the system to the SwiftUI layer and (in our manual-component setup) is not reliably activated because the badge entity's own hover/collision is dominated by the parent. visionOS 2 fixed the cascade ambiguity with HoverEffectComponent.GroupID (shared/independent activation regardless of hierarchy), and visionOS 26 introduced ManipulationComponent — which is the real foundation fix for BOTH the snap-back and the hover bugs: configureEntity adds collision+input-target+hover for you, releaseBehavior=.stay keeps the entity where the user dropped it (no snap-back), and the system OWNS the transform during manipulation so reconcile can never fight a live drag. Custom playful effects (glow/jitter/scale) that respond to gaze must be done via a ShaderGraphMaterial HoverState node (.shader style), because that is the only path where gaze drives a RealityKit-visible material/geometry change without the app learning hover. WWDC25's GestureComponent + ViewAttachmentComponent + observable transforms let us replace the entire name-parsing reconcile loop with per-entity gesture ownership.

### Core principles
- **Gaze hover is applied by the system OUTSIDE the app process — the app is NEVER told what is hovered** — Custom hover effects 'were designed from the ground up to preserve privacy, as they're applied by the system outside the app process without requiring extra entitlements.' Consequence for us: you cannot read a live hover boolean in Swift and use it to move a RealityKit entity. Any gaze-reactive visual must be declared up front (HoverEffectComponent style, or a SwiftUI .hoverEffect closure, or a shader HoverState node) and the SYSTEM evaluates/animates it. This is exactly why a hand-rolled 'on hover, set entity.scale' design is impossible and why our intuition keeps failing. _(src: WWDC24 'Create custom hover effects in visionOS' (Session 10152); WWDC24 'Discover RealityKit APIs' (Session 10103))_
- **HoverEffectComponent requires the trio AND only affects its own entity, not children** — An Apple Graphics/Games engineer: 'HoverEffectComponent ... only affect[s] its Entity and won't affect its children.' Hover detection also requires InputTargetComponent + CollisionComponent on the SAME entity (confirmed twice on the dev forums). So a child × badge needs its OWN hover+input+collision, AND must not be eclipsed by the parent's collision volume. _(src: Apple Developer Forums thread 732863 and thread 761151)_
- **HoverEffectComponent.GroupID (visionOS 2) controls cascade — without it hover propagates ambiguously** — 'GroupIDs are a way to create associations between hover effects. Any hover effects that share a GroupID will also share activations.' And: 'Normally hover effects are applied hierarchically where child entities inherit the effects of their parent; if an entity has a GroupID, then they will not propagate their effects to their children.' This is the precedence/isolation knob to make the parent card and the × badge each light up independently. _(src: WWDC24 'Discover RealityKit APIs' (Session 10103); Unity PolySpatial GroupID discussion)_
- **Three built-in HoverEffectComponent styles: spotlight, highlight, shader — only shader can do custom glow/jitter** — spotlight (default): subtle gaze-following spotlight, tint+strength configurable. highlight: uniform tint over the whole mesh (HighlightHoverEffectStyle(color:strength:)). shader: drives a ShaderGraphMaterial HoverState node whose 'intensity value is animated from 0 to 1 when the player looks at the entity' — the ONLY built-in path to a gaze-reactive glow/scale/jitter because a GeometryModifier or surface shader reads that intensity. Spotlight/highlight cannot scale or jitter geometry. _(src: WWDC24 Session 10103; HoverState in RealityView forum 761151; Reality Composer Pro materials WWDC23 Session 10202)_
- **ManipulationComponent (visionOS 26) is the transform-ownership foundation: the SYSTEM owns the transform during a drag** — ManipulationComponent.configureEntity() 'will set up the subject with collision, assign the Input Target Component and the default Hover Effect.' During manipulation the system drives the entity transform; releaseBehavior=.stay leaves it at the dropped transform (default .reset snaps it back); persist the final transform in the ManipulationEvents.WillEnd handler. This structurally eliminates the 'reconcile re-asserts stored position mid-drag' snap-back because the app does not own the transform while the user is holding the entity, and the persisted value is written exactly when the gesture ends. _(src: WWDC25 'What's new in RealityKit' (Session 287); 'Deep Dive into Manipulation on visionOS' (Step Into Vision))_
- **SwiftUI .hoverEffect inside an attachment affects the SwiftUI layer, not a RealityKit entity transform** — The closure form `.hoverEffect { effect, isActive, proxy in effect.scaleEffect(isActive ? 1.3 : 1.0) }` is a SwiftUI-layer effect the system applies to the rendered view; it CAN scale the attachment's rendered content, but it is gated by that view actually receiving system hover. In our setup the badge entity's gaze/hit is swallowed by the parent card's oversized collision, so isActive effectively never flips. The effect itself is correct SwiftUI; the problem is upstream input/collision precedence, not the modifier. _(src: WWDC24 Session 10152; createwithswift 'Creating advanced hover effects in visionOS')_
- **App selection state is separate from system hover and is fully app-owned** — Our merge/link 'isPicked' highlight (NodeCardView isHighlighted) is correct as an APP-state visual — it is not hover, it is selection, and the app legitimately controls it and can animate it with SwiftUI .animation. Keep these two concepts in separate code paths: HoverEffectComponent/.hoverEffect for gaze feedback; explicit @Observable selection flags for selection feedback. Never try to read system hover to compute selection. _(src: WWDC24 Session 10152 (hover is system/privacy-applied); general SwiftUI state model)_

### Key APIs
- **`HoverEffectComponent(.spotlight/.highlight/.shader)`** _(min visionOS 1.0 (component + spotlight/highlight); .shader style and GroupID added visionOS 2.0)_ — Declarative gaze-hover highlight on an entity; system-applied, privacy-preserving
  - When: On every interactive entity (card, badge, port) that should light up on gaze. Use .highlight(HighlightHoverEffectStyle(color:strength:)) for a tinted card glow; .shader(.default) when a ShaderGraphMaterial HoverState node drives a custom glow/jitter.
  - ⚠️ Requires InputTargetComponent + CollisionComponent on the same entity. Only affects its own entity, not children. Default plain HoverEffectComponent() = spotlight. Cannot scale/move geometry except via the shader style.
- **`HoverEffectComponent.GroupID`** _(min visionOS 2.0)_ — Associate or isolate hover activations across entities regardless of parent/child hierarchy
  - When: Give the parent card and the × badge DIFFERENT GroupIDs so each activates independently (fixes the × not highlighting because the parent swallows it); or the SAME GroupID to make a card + its ports glow together.
  - ⚠️ Must be set on all participating entities. Without it, hierarchical propagation is ambiguous and effects can leak globally.
- **`ManipulationComponent + ManipulationComponent.configureEntity(_:)`** _(min visionOS 26)_ — Built-in pick-up/move/rotate/scale of an entity; the system owns the transform during the gesture
  - When: Replace our hand-rolled translation DragGesture for moving cards. configureEntity auto-adds CollisionComponent + InputTargetComponent + default HoverEffectComponent. This is THE fix for drag snap-back.
  - ⚠️ Set releaseBehavior=.stay (default .reset snaps the entity back to its origin on release — likely a second contributor to our snap-back). Tune dynamics (translation/rotation/scaling/inertia). Persist the final transform in the WillEnd event, not per-frame.
- **`ManipulationEvents (WillBegin / DidUpdateTransform / WillRelease / WillEnd / DidHandOff)`** _(min visionOS 26)_ — Lifecycle hooks around a manipulation to constrain transforms and persist on release
  - When: DidUpdateTransform: clamp to VolumeMetrics bounds per frame (read/modify event.entity.transform). WillEnd: write the final x/y/z + quaternion to the view model / tRPC (replaces our endDrag-on-DragGesture.onEnded).
  - ⚠️ Constrain in DidUpdateTransform; do NOT persist every frame. WillBegin/WillRelease are the right place to mute/restore a hover effect if needed.
- **`GestureComponent`** _(min visionOS 26)_ — Attach a specific SwiftUI gesture (TapGesture/DragGesture) directly to a single entity
  - When: Replace the two top-level .targetedToAnyEntity() gestures + the 'ctl::/edge::' name-parsing disambiguation. Put a TapGesture on each card (open edit), a separate gesture on the × badge (dismiss), etc. — ownership becomes explicit per entity.
  - ⚠️ visionOS 26 only. Removes the fragile string-prefix routing; each entity declares its own behavior. Combine with ManipulationComponent for move + tap.
- **`SwiftUI .hoverEffect(_:in:isEnabled:) closure form`** _(min visionOS 2.0 (custom closure form); basic .hoverEffect() visionOS 1.0)_ — Custom system-applied hover effect on a SwiftUI view (incl. RealityView attachments & ornaments)
  - When: For attachment-rendered controls (the × badge's SwiftUI content) to scale/clip/opacity on gaze: `.hoverEffect { effect, isActive, proxy in effect.scaleEffect(isActive ? 1.2 : 1.0) }`. Use .hoverEffectGroup to group/isolate within the view tree.
  - ⚠️ isActive is evaluated by the SYSTEM, not delivered to your logic; you only declare the mapping. The effect only fires if that view actually receives system hover — which fails when a parent RealityKit collision eclipses the attachment entity. Affects the SwiftUI layer, not the RealityKit entity transform.
- **`ShaderGraphMaterial + HoverState node (Reality Composer Pro)`** _(min visionOS 1.0 (ShaderGraphMaterial); HoverState node usable with .shader hover style visionOS 2.0)_ — Drive a gaze-reactive glow/scale/jitter on geometry via the shader, paired with HoverEffectComponent(.shader)
  - When: For the developer's wanted 'playful jitter / glow on highlight' on the 3D card mesh itself (not the SwiftUI content). HoverState.intensity (0→1 on gaze) feeds a Mix into emissive (glow) or a GeometryModifier (scale/jitter).
  - ⚠️ Requires building the material in Reality Composer Pro; surface shader = PBR/emissive changes, GeometryModifier = vertex/scale changes. Still needs InputTargetComponent + CollisionComponent. Our cards are SwiftUI attachments (flat quads), so shader-jitter applies to a mesh card, not the attachment's SwiftUI text.
- **`InputTargetComponent (.isEnabled) + CollisionComponent`** _(min visionOS 1.0)_ — Make an entity hit-testable for gaze/pinch and define the precise input region
  - When: Required prerequisites for ANY hover/tap/drag. Set isEnabled=false to temporarily make an entity transparent to input. Size the collision to the actual quad — and INSET the parent card's collision (or reduce its z-depth) so it does not enclose the forward-offset × badge.
  - ⚠️ Our parent collision uses depth max(z,0.03) and the badge sits only +0.006 m forward — the parent box swallows the badge. Shrink parent z-depth, push the badge further forward, or give them distinct GroupIDs / separate input ownership.

### Anti-patterns
- **Reading or inferring a live 'is hovered' boolean in app code to drive an entity transform** — Gaze hover is applied by the system outside the app process for privacy; the app is never told what is hovered. Any design that branches on hover in Swift to move/scale an entity cannot work. Declare the effect (HoverEffectComponent style / .hoverEffect closure / shader HoverState) and let the system animate it.
- **A single reconcile(content:) re-asserting card.position = storedPosition on EVERY update pass, including mid-drag** — This is the literal root cause of snap-back: the stored VM position is stale until release, so every observed change yanks the card back. The @State draggingID skip is unreliable because @State mutated in a RealityKit gesture .onChanged isn't visible to the update closure in time. Fix by giving the transform a single owner: ManipulationComponent (system owns it during drag; .stay keeps it; persist in WillEnd).
- **Wrapping a child × badge in the parent card's collision volume (parent depth max(z,0.03), child at +0.006 z)** — Hover/hit-testing of the badge is eclipsed by the larger parent collision, and HoverEffectComponent does not cascade to children — so the badge never gets its own gaze highlight. Inset/shrink the parent collision, push the badge forward, and/or assign distinct HoverEffectComponent.GroupIDs.
- **Routing all gestures through two top-level .targetedToAnyEntity() handlers + parsing entity names ('ctl::role::owner', 'edge::')** — Fragile, unscalable disambiguation that hard-codes roles into strings and forces every new interactive element to extend a central parser. visionOS 26 GestureComponent attaches the right gesture to the right entity directly, making ownership explicit and grab-and-drag near-trivial.
- **Deleting and re-adding ALL edges + ports from scratch on every reconcile pass** — Wasteful churn that also re-triggers @Observable invalidation and contributes to the feedback loops; edges should be persistent entities whose endpoints are updated (or driven by an ECS System) rather than destroyed/rebuilt each frame.
- **Using .spotlight or .highlight hover style to try to achieve scale/jitter/glow** — Those styles only tint/spotlight the existing mesh; they cannot scale or deform geometry. Playful jitter/glow on gaze requires the .shader style with a ShaderGraphMaterial HoverState node (GeometryModifier for scale, emissive Mix for glow).

### Recommendations for our app
- Adopt ManipulationComponent (visionOS 26) for moving cards instead of the hand-rolled translation DragGesture. Call ManipulationComponent.configureEntity(card) once when the card is created (it adds collision + input-target + default hover), set releaseBehavior=.stay, and persist the final transform in the ManipulationEvents.WillEnd subscription (calling viewModel.endDrag). This makes the system own the transform during a drag, so the reconcile loop can no longer fight it — eliminating snap-back at the root rather than via the unreliable @State draggingID skip.
- Stop re-asserting card.position every reconcile pass. Establish single transform ownership: layout engine sets position only when an entity is first placed or when auto-flow explicitly re-lays-out (hand-moved entities already excluded); after that, the entity transform is owned by ManipulationComponent during drag and by the persisted value otherwise. Reconcile should add/remove entities and update content, not re-stamp positions of already-placed, user-movable entities every frame.
- Fix the × badge hover with three combined changes: (a) shrink the parent card's CollisionComponent z-depth and inset it so it does not enclose the badge; (b) push the badge further forward than +0.006 m; (c) give the parent card and the badge DIFFERENT HoverEffectComponent.GroupIDs so the badge activates independently (hover does not cascade to children, so the badge must have its own InputTargetComponent + CollisionComponent + HoverEffectComponent — which it already does — plus an isolating GroupID).
- Replace the name-parsing gesture router with per-entity GestureComponent (visionOS 26): a TapGesture on the card (open edit / toggle volume), a separate TapGesture on the × badge (dismiss), ManipulationComponent for move. This removes the 'ctl::'/'edge::' string parsing and makes each new interactive element self-contained — directly serving the developer's 'should scale as the UI grows' goal.
- For gaze feedback on cards, prefer the built-in HoverEffectComponent(.highlight(HighlightHoverEffectStyle(color: typeTint, strength:))) for an elegant tinted glow that matches the type color, since it is battle-tested and free. Reserve the .shader style + ShaderGraphMaterial HoverState node for the playful jitter/glow the developer wants — but note our cards are SwiftUI attachments (flat quads), so shader jitter must target a backing mesh, while SwiftUI-layer scale can be done with the .hoverEffect closure inside the attachment rootView (once the badge/card actually receives system hover).
- Keep system hover and app selection strictly separate. The merge/link 'isPicked' highlight (NodeCardView.isHighlighted) is app selection state — correct as-is, and it can be animated with SwiftUI .animation. Do not attempt to derive selection from hover. Document this two-system split in .claude/rules/ so future sessions don't re-conflate them.
- Validate the .shader/HoverState path early: per forum 761151, HoverEffectComponent(.shader(.default)) shows nothing unless InputTargetComponent + CollisionComponent are also present. Add a small spike before committing to shader-based effects, since it requires authoring a ShaderGraphMaterial in Reality Composer Pro (heavier than the declarative .highlight style).

## Collision and hit-testing hierarchy in RealityKit on visionOS (CollisionComponent, InputTargetComponent, CollisionGroup/Filter, raycast, hover) — concrete recipe for an independently hit-testable card + child "x"

In RealityKit on visionOS, an entity is hit-testable for gestures only if it has BOTH a CollisionComponent (defines the hit region) and an InputTargetComponent (marks it input-capable); HoverEffectComponent adds gaze feedback. Hit-test resolution among nested/overlapping entities is decided by collision geometry and depth: when a parent's collision shape fully encloses a child's shape and the parent also has an InputTargetComponent, the parent is hit FIRST and the child becomes unreachable. This is the exact, Apple-DTS-confirmed root cause of bug #2: the card's collision box (depth max(z,0.03), ~3cm) swallows the "x" child placed only +0.006m (6mm) in front of it, so the child never receives hover or tap — and HoverEffectComponent does not propagate to children, while the SwiftUI .hoverEffect inside the attachment is correct but never fires because the system never hit-tests the child. The fix is hierarchical: make the parent NOT input-targetable in the region where the child lives (use a precisely-sized/inset collision shape, push the child out in front by a real gap of >=2-3cm so it sits clear of the parent box, or disable the parent's InputTargetComponent and decompose the card's own hit area into child sub-targets), plus use CollisionGroup/CollisionFilter masks to route gestures only to intended categories. Bug #1 (drag snap-back) is a transform-ownership problem rooted in the reconcile-in-update pattern: writing entity.position every update pass while only persisting on release means the stored value is stale during the drag, and RealityView's update closure re-asserts it. visionOS 26 provides first-class solutions: ManipulationComponent (handles grab/drag/persist natively with releaseBehavior=.stay), GestureComponent (per-entity gestures with correct local targeting), Observable entities + content.animate()/Entity.animate() for smooth SwiftUI-driven transforms, and ViewAttachmentComponent for component-based attachments — all of which let the developer stop hand-rolling the reconcile/transform-ownership engine. scene.raycast(origin:direction:length:query:mask:) with CollisionGroup masks gives precise, filtered hit queries when manual hit-testing is needed; many colliders are fine if shapes are simple boxes/spheres and you avoid per-frame shape regeneration (the recursive visualBounds feedback-loop trap).

### Core principles
- **Two components are mandatory and complementary: CollisionComponent + InputTargetComponent** — An entity receives gestures ONLY if it has a CollisionComponent (its shapes define the hit region) AND an InputTargetComponent (marks it as input-capable). Missing either means no hits. HoverEffectComponent is separate and adds gaze-driven visual feedback. This is confirmed across Apple docs, the Lickability collision guide, and the stepinto SwiftUI-gestures guide. _(src: Apple RealityKit docs; Better together SwiftUI and RealityKit (WWDC25-274); stepinto.vision SwiftUI gestures guide. visionOS 1.0+.)_
- **Hit-test precedence: an enclosing parent collision shape with InputTargetComponent ALWAYS wins over its children** — Per Apple DTS: 'If your surface has a collision shape that fully encompasses the children's collision shapes, and it also has an InputTargetComponent, it is expected that you receive hits on the surface and not its children, because the children cannot be hit, the parent would always be hit first.' Resolution is closest/frontmost by collision geometry, NOT by entity z-position alone — a parent box that spatially overlaps the child region intercepts the hit even if the child is nominally in front by a few mm. _(src: Apple Developer Forums thread 747256 (DTS Engineer); echoed in forum 732863. visionOS 1.0+.)_
- **HoverEffectComponent affects only its own entity, not children** — Adding HoverEffectComponent to a parent does NOT highlight children. For a child control you must add HoverEffectComponent to the child entity itself AND ensure the system actually hit-tests that child (i.e. the parent must not swallow the hover). For SwiftUI content rendered via ViewAttachmentComponent, the in-view .hoverEffect{ effect, isActive, _ in effect.scaleEffect(...) } is the correct mechanism, but it only fires when the child entity receives the hover. _(src: Apple Developer Forums thread 732863; Create custom hover effects in visionOS (WWDC24-10152). visionOS 1.0 (component), 2.0 (highlight/shader styles).)_
- **Do not mutate observed entity transforms inside RealityView's update closure — this is the root of drag snap-back** — Writing entity.position/transform in update creates a feedback/ownership conflict: any observed change re-fires update, which re-asserts a (stale-during-drag) stored position. Apple explicitly warns this risks infinite loops and says the safe places to modify observed entities are gesture closures, custom Systems, or the make closure — never blindly in update. Guard writes (only write if value actually changed) if you must. _(src: Better together: SwiftUI and RealityKit (WWDC25-274). visionOS 26 for Observable entities; the anti-pattern applies on all versions.)_
- **CollisionGroup + CollisionFilter route which entities a query/gesture can hit** — CollisionGroup is an OptionSet (bitwise, e.g. 1<<0). CollisionFilter(group:mask:) declares an entity's category and which categories it interacts with. Used with scene.raycast(..., mask:) or hitTest(..., mask:) to restrict hits to a category (e.g. only 'control' entities, or exclude edges). This is the clean way to disambiguate instead of parsing entity names. _(src: codingxr.markhorgan.com Interactivity in RealityKit; Apple CollisionFilter docs; Discover RealityKit APIs (WWDC24-10103). visionOS 1.0+.)_
- **Collision shape sizing/offset is the lever for nested hit-testability** — ShapeResource.generateBox(size:).offsetBy(translation:) lets you size and position a hit region independent of the visual mesh. To keep a child reachable: size the parent's collision box to NOT cover the child's screen region (inset it, or cut the depth so it does not extend toward the viewer past the child), and/or move the child entity a real gap (>= 2-3cm, not 6mm) in front so its own box clears the parent box. visualBounds(recursive:false) measures only the entity's own quad — correct — but regenerating shapes every frame from visualBounds risks drift/feedback; cache the shape and only rebuild on size change. _(src: Apple ShapeResource docs; Lickability collision guide; current SpatialSceneView.swift analysis (depth max(z,0.03) vs child +0.006). visionOS 1.0+.)_
- **visionOS 26 makes grab-and-drag near-trivial via ManipulationComponent and per-entity GestureComponent** — ManipulationComponent.configureEntity(entity, hoverEffect:, allowedInputTypes:, collisionShapes:) wires InputTargetComponent + CollisionComponent + HoverEffectComponent and handles translate/rotate/scale natively, with releaseBehavior=.stay to persist position (vs .reset). GestureComponent attaches a SwiftUI gesture to ONE entity so values arrive in that entity's local space — eliminating .targetedToAnyEntity name-parsing disambiguation. Both remove the need to hand-own transforms in reconcile. _(src: Better together: SwiftUI and RealityKit (WWDC25-274); What's new in RealityKit (WWDC25-287); stepinto.vision deep dive into manipulation. visionOS 26+.)_

### Key APIs
- **`CollisionComponent(shapes:mode:filter:)`** _(min visionOS 1.0 (RealityKit, also iOS 13/macOS 10.15))_ — Defines the hit-test/physics region of an entity via ShapeResource shapes. mode .default (collides) vs .trigger (events only). filter (CollisionFilter) controls group/mask routing.
  - When: On every entity that must be hit by a gesture or raycast. Size the shapes to the intended hit region, not necessarily the visual mesh.
  - ⚠️ A parent shape that encloses a child shape intercepts the child's hits. Regenerating shapes from visualBounds every frame causes drift/feedback — cache and rebuild only on actual size change.
- **`InputTargetComponent (isEnabled, allowedInputTypes / InputType: .direct, .indirect, .all)`** _(min visionOS 1.0)_ — Marks an entity as eligible to receive input gestures. isEnabled=false makes it pass-through (does NOT consume hits). allowedInputTypes restricts to direct (hand-touch) vs indirect (gaze+pinch) input.
  - When: Pair with CollisionComponent on anything interactive. Set isEnabled=false on a container parent so its children become the hit targets (key fix for the card/x problem).
  - ⚠️ Without it, an entity with a collision shape still gets no gestures. Disabling it on the parent is the cleanest way to let a child control win the hit test.
- **`HoverEffectComponent(.spotlight / .highlight / .shader)`** _(min visionOS 1.0 (spotlight); highlight + shader styles visionOS 2.0)_ — System-rendered gaze feedback. spotlight (default subtle), highlight (uniform tint over whole mesh, tint+strength configurable), shader (drives a ShaderGraphMaterial HoverState node).
  - When: On RealityKit mesh entities for built-in highlight. For SwiftUI-attachment content (your cards/x), prefer the in-view SwiftUI .hoverEffect modifier instead, which can scale/clip/fade the rendered view.
  - ⚠️ Affects only its own entity, never children. Hover is resolved by the system out-of-process (privacy) and only fires when that entity is actually hit-tested — if the parent box swallows the hover, neither the component nor the attachment's .hoverEffect will fire.
- **`SwiftUI .hoverEffect { effect, isActive, proxy in ... } (CustomHoverEffect, HoverEffectGroup)`** _(min visionOS 1.0 (CustomHoverEffect/groups visionOS 2.0))_ — Drives view-level visual changes (scaleEffect/opacity/clipShape) on gaze. Supports grouping (HoverEffectGroup) so a hover on one element animates several, and delayed/animated reveals.
  - When: Inside the SwiftUI rootView of a ViewAttachmentComponent to make cards/controls visibly react — this is the correct, locked SwiftUI-renders-content pattern. Use a HoverEffectGroup to do the playful 'jitter on highlight' across the card and its controls together.
  - ⚠️ It is purely visual and only activates when the underlying entity receives the hover hit. The current DismissBadge .hoverEffect is correct code — it fails only because the parent collision box prevents the badge entity from being hovered.
- **`ShapeResource.generateBox(size:)/generateSphere(radius:)/generateConvex(from:), .offsetBy(translation:rotation:)`** _(min visionOS 1.0)_ — Builds collision/hit shapes decoupled from the visual mesh; offsetBy repositions/reorients the shape within the entity's local space.
  - When: To inset or shrink a parent's hit region so it doesn't cover child controls, and to place a child control's hit box precisely. Box/sphere are cheapest; convex is heavier.
  - ⚠️ Async variants exist (generateConvex async). Offsetting must account for the entity's own pivot; verify with Reality Composer Pro 'Check Collision Shapes' or a debug wireframe.
- **`scene.raycast(origin:direction:length:query:mask:relativeTo:) and raycast(from:to:query:mask:)`** _(min visionOS 1.0)_ — Manual ray hit-test against collision geometry. query .nearest (frontmost) or .all; mask (CollisionGroup) filters which categories are eligible. Returns CollisionCastHit with entity, position, distance, normal.
  - When: When you need explicit, filtered hit resolution (e.g. find the frontmost 'node' under a point ignoring edges/ports) instead of relying on gesture targeting.
  - ⚠️ Only hits entities with collision shapes. Use a mask to avoid hitting decorative entities (edges/ports) — cleaner than name-prefix parsing.
- **`CollisionGroup (OptionSet) + CollisionFilter(group:mask:)`** _(min visionOS 1.0)_ — Categorize entities and declare which categories interact. Bidirectional: both sides' masks must agree for a collision; for hit-testing a mask selects eligible groups.
  - When: Define groups like .node, .control, .edge. Put cards in .node, the x in .control, edges in .edge with empty input mask. Route gestures/raycasts by mask to disambiguate cleanly.
  - ⚠️ Easy to get the bitwise math wrong; wrap in named constants. Filter affects physics collisions too if mode=.default — use .trigger if you only want events.
- **`GestureComponent(_:) (per-entity SwiftUI gesture)`** _(min visionOS 26)_ — Attaches a SwiftUI gesture directly to a single entity; values are reported in that entity's local coordinate space, removing the need for .targetedToAnyEntity + manual entity disambiguation.
  - When: Give each card its own drag/tap, and give the x its own tap. Eliminates the ctl::/edge:: name-parsing dispatch in the current code.
  - ⚠️ Still requires InputTargetComponent + CollisionComponent on the entity. New API — battle-testing limited; keep RealityView-level gestures as a fallback path.
- **`ManipulationComponent + ManipulationComponent.configureEntity(_:hoverEffect:allowedInputTypes:collisionShapes:) and .manipulable() / ManipulationEvents`** _(min visionOS 26)_ — Native object grab/move/rotate/scale with audio + hover, releaseBehavior=.stay to persist position. configureEntity wires Input/Collision/Hover in one call. Events: WillBegin/DidUpdateTransform/WillRelease/WillEnd/DidHandOff.
  - When: To replace the hand-rolled DragGesture + reconcile transform ownership for moving cards. DidUpdateTransform lets you clamp to VolumeMetrics bounds; WillEnd is where you persist to the backend.
  - ⚠️ Default releaseBehavior is .reset (snaps back!) — must set .stay. A known beta bug can stop translation app-wide until device restart. Clamp inside DidUpdateTransform, not by re-asserting position in update.
- **`ViewAttachmentComponent(rootView:) and PresentationComponent`** _(min visionOS 26 (component-based attachments); visionOS 1.0 had builder-based attachments)_ — ViewAttachmentComponent puts a SwiftUI view on an entity from anywhere (replaces the make-closure attachments builder, fixing the setup race). PresentationComponent shows SwiftUI popovers/sheets/menus anchored to an entity.
  - When: Render every card/note/control as a ViewAttachmentComponent entity. Use PresentationComponent for the edit form/menus instead of the top-level .sheet, keeping UI spatially anchored.
  - ⚠️ Component-based attachments still need Input/Collision to be interactive. The current code adds attachments in update due to a setup race — the component form removes that race.
- **`content.animate{ } / Entity.animate(animation:) (SwiftUI-driven implicit animation of components)`** _(min visionOS 26)_ — Animate Transform/Audio/Light/Model component changes using SwiftUI animation curves (.bouncy etc.) without manual timers — the proper home for smooth moves and the 'jitter' highlight.
  - When: For return-to-tray, layout re-flow, and playful highlight jitter. Set the target transform inside the animate block; the framework interpolates.
  - ⚠️ Do not combine with re-asserting transforms in update (conflicts). On <26, fall back to FromToByAnimation / Transform-based animation controllers.

### Anti-patterns
- **Re-asserting card.position from the stored (data-model) value on EVERY RealityView update pass** — This is the direct cause of drag snap-back (bug #1). During a drag the stored value is stale (written only on release), and update re-fires on any observed change, fighting the live gesture. Transform ownership must shift to the gesture/manipulation while active. Apple explicitly warns against mutating observed entities in update (infinite-loop / ownership conflict).
- **Trying to skip the dragged card via a @State draggingID checked inside reconcile** — @State mutated inside a RealityKit gesture .onChanged is not reliably visible to the update closure before the next reconcile pass, so the skip misses and the card still snaps. The fix is architectural (own the transform in the gesture/ManipulationComponent), not a flag check.
- **Giving a parent card a collision box that fully encloses a child control placed only 6mm in front** — Apple DTS-confirmed: an enclosing parent with InputTargetComponent is hit first, so the child x never receives hover or tap (bug #2). max(z,0.03) depth (3cm) over a +0.006 (6mm) child guarantees the parent swallows it. Inset/shrink the parent shape, push the child out by a real gap, or disable the parent's InputTargetComponent.
- **Adding HoverEffectComponent to the parent and expecting the child x to highlight** — HoverEffectComponent only affects its own entity, never children. The child needs its own hover mechanism (its attachment's SwiftUI .hoverEffect) AND must actually be hit-tested — which it currently never is because the parent box intercepts the hover.
- **Disambiguating gestures by parsing entity names ('ctl::', 'edge::') with .targetedToAnyEntity()** — Fragile and unscalable. Routing should use CollisionGroup/CollisionFilter masks (and per-entity GestureComponent on visionOS 26), so edges/ports simply aren't input targets and controls are a distinct group — no string parsing, no accidental matches.
- **Deleting and re-adding ALL edges + ports and rebuilding the x control from scratch every update pass** — Per-frame teardown/rebuild thrashes the scene, regenerates shapes from visualBounds (feedback/drift trap), and breaks any in-flight hover/animation state. Reconcile should diff and update in place; transforms should be SwiftUI-animated, not recreated.
- **Regenerating CollisionComponent shapes from visualBounds(recursive:true) each frame** — Recursive bounds include children (controls/ports), inflating the box every pass → outward drift and a self-reinforcing feedback loop. Use recursive:false, cache the shape, and rebuild only when the card's intrinsic size actually changes.

### Recommendations for our app
- Fix bug #2 with hierarchical collision, not patches: either (a) set the card parent's InputTargetComponent.isEnabled=false and add a dedicated child 'card body' entity with its own correctly-sized collision box as the drag target, so the x child (its own Input+Collision+HoverEffect) is independently hit-tested; or (b) inset/shrink the parent's collision box (offsetBy + smaller extents) so it does not cover the x's screen region, and push the x out by a real >=0.02-0.03m gap, not 0.006m.
- Route gestures by CollisionGroup/CollisionFilter instead of name parsing: define groups .node, .control, .edge. Cards in .node, x/ports controls in .control, edge lines in .edge with no InputTargetComponent. Then a TapGesture masked to .control hits only controls; drag masked to .node hits only cards. Delete the ctl::/edge:: string-parsing dispatch.
- Fix bug #1 by moving transform ownership off the data model during interaction. Two viable paths: (Preferred, visionOS 26) replace the hand-rolled DragGesture with ManipulationComponent.configureEntity(card, collisionShapes:[...]) and releaseBehavior=.stay; clamp to VolumeMetrics in a DidUpdateTransform subscription; persist in WillEnd. (Or) keep a custom gesture but STOP writing position in reconcile — let the gesture own the live transform and only reconcile entities that are not the active drag target, tracked via an @Observable viewModel.activeDragID (observable, not @State) so update sees it.
- Make reconcile diff-and-update-in-place, not teardown-and-rebuild. Keep a stable id->entity map; on update only add new entities, remove gone ones, and write transforms only for entities whose model position changed AND that are not being dragged/animated. Never rebuild edges/ports/x wholesale each pass — update their endpoints in place.
- Stop regenerating collision shapes every frame. Compute the card's collision box once from visualBounds(recursive:false) when its size is first known (or changes), cache it on the entity, and reuse — eliminating the visualBounds feedback/drift loop the memory notes flag as an open bug.
- Use the SwiftUI .hoverEffect inside attachment rootViews as the single highlight mechanism (the DismissBadge code is already correct). For the playful 'jitter on highlight', wrap card + its controls in a HoverEffectGroup so they animate together, and use Entity.animate(animation: .bouncy) for spatial jitter on the entity transform. This honors the locked 'SwiftUI renders content' rule.
- Adopt the visionOS 26 component-based stack to delete the foundation cruft: ViewAttachmentComponent (removes the make/update setup-race for attachments), GestureComponent (per-entity gestures in local space, removes .targetedToAnyEntity disambiguation), and content.animate/Entity.animate (removes manual transform stepping). Target is visionOS 26.2 so all are in scope.
- Make activeDragID / animation state @Observable on the view model (not SwiftUI @State) so the RealityView update closure reliably observes it — this is the specific reason the prior draggingID skip failed.
- Establish a small component vocabulary as the design system: an InteractiveCard assembler that, given a SwiftUI view, produces an entity with ViewAttachmentComponent + a cached right-sized CollisionComponent (group .node) + InputTargetComponent + HoverEffectComponent; and a Control assembler (group .control, parent isEnabled handling) for x/ports. Build all entities through these two factories so new entity kinds scale without re-deriving collision/hover wiring.
- Keep edges/ports purely decorative: no InputTargetComponent, no collision (or .trigger-only), and in group .edge excluded from all input masks — they should never participate in hit-testing.

## RealityKit/visionOS animation for UI feedback (transitions + jitter)

visionOS 26 adds a SwiftUI-to-RealityKit animation bridge; one transform authority at a time avoids our snap-back and dead-hover bugs. Entity.animate/content.animate drive transitions, a deltaTime System drives jitter, ManipulationComponent(.stay) drives drag.

### Core principles
- **Exclusive transform ownership per instant** — Reconcile re-asserting stored positions while a gesture writes = snap-back. Gesture owns during drag, data model after release. Never write observed entity state in RealityView.update (infinite loop, WWDC25 274); write in gesture/event closures, Systems, or make.
- **Bridge for transitions; deltaTime System for jitter** — Entity.animate(.spring/.easeInOut){...} (gesture/event closures) and content.animate (inherits the SwiftUI transaction) drive layout moves and hover pop. Idle/selected jitter+breathing lives in a custom System.update integrating context.deltaTime, offsetting a clean base transform; gate behind accessibilityReduceMotion (ADHD users).
- **Prefer ManipulationComponent over hand-rolled DragGesture** — visionOS 2.6+; releaseBehavior=.stay persists (no snap-back), ManipulationEvents handle handoff, DidUpdateTransform clamps to bounds. RealityKit owns the transform during manipulation, fixing the @State-not-visible-in-update class of bug.

### Key APIs
- **`Entity.animate / RealityViewContent.animate`** _(min visionOS 26)_ — Apply a SwiftUI Animation implicitly to entity Transform/color/audio.
  - ⚠️ Entity.animate in gesture/event closures; content.animate inherits the transaction; read+write observed in update = loop. Pre-26 fallback: entity.move(to:relativeTo:duration:timingFunction:) (AnimationTimingFunction, no spring).
- **`System + EntityQuery + SceneUpdateContext.deltaTime`** _(min visionOS 1.0)_ — Per-frame procedural jitter/breathing/wiggle over matching entities.
  - ⚠️ deltaTime is seconds (never assume 90fps); offset a clean base; no allocations; registerSystem(). Authored-loop alt: AnimationResource/FromToByAnimation/OrbitAnimation + BindTarget(.transform/.parameter).
- **`ManipulationComponent / HoverEffectComponent`** _(min Manipulation 2.6; HoverEffect 1.0)_ — Built-in drag/rotate/scale; gaze/hover feedback.
  - ⚠️ Default releaseBehavior .reset snaps back—set .stay; clamp in DidUpdateTransform. SwiftUI hoverEffect inside an attachment does NOT scale the entity (bug #2); a parent collision enclosing the child swallows hover/taps.

### Anti-patterns
- **Re-asserting stored transforms / writing observed props in update** — Snap-back (update overwrites the gesture's live position with stale stored) and infinite loop (read+write @Observable in update, per Apple). The @State draggingID skip is unreliable (not visible to update in time). Diff and only move changed dataOwned entities; write in gestures/Systems.
- **SwiftUI hoverEffect/scaleEffect in the attachment + child inside the parent collision** — The effect scales flat 2D content not the entity (bug #2 no-highlight); the parent collision (depth max(z,0.03)) encloses the child x (+0.006 z) and swallows hover/taps. Pop the entity via Entity.animate; give the child a tight non-overlapping collision or HitTarget.
- **Rebuilding entities/edges each frame; baking jitter into the persisted position** — Recreation kills in-flight animations/hover/System motion and churns the graph; jitter in the stored transform drifts saved layout. Diff in place (move edge endpoints); keep a clean base + ephemeral deltaTime offset.

### Recommendations for our app
- Per-entity ownership state machine (dataOwned/dragging/animating); reconcile becomes a diff that writes transforms only for dataOwned entities and only when the data-model position changed. Root-fixes snap-back instead of the @State skip.
- Replace DragGesture with ManipulationComponent(releaseBehavior=.stay); clamp in DidUpdateTransform, persist via tRPC on WillEnd/WillRelease. Animate moves Entity.animate(.easeOut(0.35)){move(to:target)} and hover pop Entity.animate(.spring(0.25,0.6)){scale=1.08}; keep HoverEffectComponent; gate decorative motion behind reduceMotion.
- JitterSystem+JitterComponent(base,amplitude,frequency,seed): phase+=context.deltaTime, transform=base+sin offset+tiny rotation (clean base=no drift). Fix the x control with its own tight non-overlapping collision (further +z / shrink card depth) or HitTarget; highlight via Entity.animate. Add a FeedbackStyle layer; glass glow via ShaderGraphMaterial BindTarget(.parameter); allocation-free System, profile RealityKit Trace at 90fps.

## Materials, Glass, and Semi-Transparency for elegant visionOS UI (visionOS 26.2, SwiftUI-renders-content + RealityKit-places-it)

For our locked "SwiftUI renders content, RealityKit places it" paradigm, the correct material engine is SwiftUI itself, not RealityKit materials. Every card/panel/note is a SwiftUI view inside a ViewAttachmentComponent, so it should style with SwiftUI glass/materials (.glassBackgroundEffect on visionOS, the new .glassEffect/Glass/GlassEffectContainer from the unified visionOS 26 Liquid Glass design, or system materials .ultraThin/.thin/.regular/.thick with vibrancy). RealityKit materials (PhysicallyBasedMaterial, UnlitMaterial, ShaderGraphMaterial, OcclusionMaterial) are only relevant for non-attachment geometry we actually draw with meshes: our edge lines, ports, grounding/occlusion planes, and any volume "shell." Mixing the two is where translucency bugs come from. There is a documented, Apple-engineer-confirmed limitation that .glassBackgroundEffect can render opaque over a skybox/custom immersive background — but in a standard volumetric WindowGroup over passthrough (our case) glass works as intended, so we should prefer it. Overlapping translucent RealityKit meshes (our edges/ports crossing cards) suffer order-dependent sorting artifacts that are controlled with ModelSortGroup/ModelSortGroupComponent and opacityThreshold, not by hoping the renderer sorts correctly. The biggest architectural lever is that visionOS 26 makes Entities Observable and lets SwiftUI animations drive RealityKit Transforms (entity.animate), and provides ManipulationComponent for drag — which is the real fix for our drag snap-back and the foundation for a token-driven, cohesive glass design system. Text legibility on glass comes for free from system text styles + vibrancy (primary/secondary/tertiary); colored text on glass is an explicit anti-pattern.

### Core principles
- **For attachment content, style with SwiftUI materials/glass — never substitute RealityKit materials** — Our cards/panels/notes are SwiftUI views surfaced via ViewAttachmentComponent. They must be styled with SwiftUI: .glassBackgroundEffect() (visionOS-native frosted glass), the visionOS 26 .glassEffect(_:in:)/Glass/GlassEffectContainer Liquid Glass API, or .background(.thinMaterial/.regularMaterial, in: shape) with system vibrancy. RealityKit PhysicallyBasedMaterial/ShaderGraphMaterial are for meshes we draw ourselves (edges, ports, planes), not for the look of SwiftUI cards. Our NodeCardView already does this correctly (.background(.regularMaterial)); the architecture should make that the only sanctioned path. _(src: WWDC25 Session 274 'Better together: SwiftUI and RealityKit'; createwithswift 'Using Attachments in a RealityView'; HIG Materials)_
- **Glass over passthrough works; glass over a custom skybox/immersive background can render opaque (known limitation)** — An Apple Vision Pro engineer confirmed on the dev forums that .glassBackgroundEffect does not let a skybox texture show through — it renders opaque/frosted with no environment behind it. This is a limitation of immersive-skybox setups, NOT of our volumetric WindowGroup over real passthrough. Since our volume is gravity-aligned over passthrough, glass behaves correctly, so we should adopt it confidently. If we ever add a skybox/full-immersion mode, fall back to .background(Color.white.opacity(0.5)) or a mesh plane. _(src: Apple Developer Forums thread 755910 (glassMaterialEffect not working with skybox))_
- **Material level encodes elevation/hierarchy; this is the basis for design tokens** — Apple's HIG defines a layering rule: lighter/thicker glass = foreground/closer, darker/thinner = background; .thin highlights interactive elements (buttons, selected items), .regular separates sections (panels/sidebars), .thick makes things recede (text fields). 'Use darker material over lighter to create depth; lighter over darker to highlight interactive areas.' This maps cleanly to a token system: each entity kind gets a material level + tint + corner radius + elevation, instead of ad-hoc per-card .background calls. _(src: Apple HIG — Materials foundation; createwithswift 'Ensuring interface legibility and contrast in visionOS')_
- **Text legibility on glass is handled by vibrancy + system text styles, not custom colors** — visionOS provides primary/secondary/tertiary vibrancy. System text styles include appropriate vibrancy automatically and remain legible over arbitrary passthrough. Use .foregroundStyle(.primary/.secondary) for hierarchy; reserve tertiary for low-importance text. Explicit DON'Ts: colored text on glass, stacking light materials without a darker separator, fully opaque windows. Our cards use .foregroundStyle(.secondary) correctly; the system should forbid hardcoded text colors in the token layer. _(src: Apple HIG — Materials; createwithswift legibility article)_
- **Liquid Glass (visionOS 26) is the unified forward direction; GlassEffectContainer enables cohesive multi-element glass** — .glassEffect(_:in:isEnabled:) with Glass (.regular/.clear/.identity, .tint(_:), .interactive()) is the cross-platform Liquid Glass API (visionOS 26.0+). Critical rule: 'glass cannot sample other glass' — wrap related glass elements in a GlassEffectContainer so they share a sampling region and can morph (glassEffectID + Namespace, glassEffectUnion, glassEffectTransition). .regular is universal; .clear needs bold content over media. Accessibility (Reduce Transparency, Increase Contrast, Reduce Motion) is handled automatically — do not override. For visionOS specifically, .glassBackgroundEffect remains the established window/attachment glass; treat .glassEffect as the newer in-content variant and validate per-surface with xcodebuild. _(src: WWDC25 'Liquid Glass' direction; LiquidGlassReference (conorluddy); Apple newsroom 2025-06)_
- **Overlapping translucent RealityKit meshes are order-dependent — control draw order explicitly** — RealityKit uses rasterization with depth-based sorting that fails for intersecting/overlapping transparent surfaces (flicker, vanishing outlines). For our derived edges/ports that cross translucent cards, control order with ModelSortGroup/ModelSortGroupComponent (render innermost/smallest first, outermost/largest last) and choose a depth-pass strategy. Also set opacityThreshold on transparent PhysicallyBasedMaterial (even 0.0) to avoid the FB12605220 erroneous-culling bug when blending with faceCulling=.none. _(src: xreality.zone 'Solving Nested Transparent Objects in RealityKit with Rendering Ordering'; varrall 'Skyboxes and transparency in visionOS' (FB12605220))_
- **Depth/elevation for floating cards comes from grounding shadows + subtle specular, not fake drop shadows** — Without shadows it is hard to read relative depth of floating entities. RealityKit grounding shadows (GroundingShadowComponent(castsShadow: true)) give real spatial shadows; an OcclusionMaterial plane beneath content plus a DirectionalLight shadow is the manual route. HIG: interactive elements get subtle specular highlights that reinforce dimensionality. For UI color fidelity on any mesh we draw, use UnlitMaterial with applyPostProcessToneMap:false so colors match SwiftUI exactly (tone mapping is on by default and shifts UI colors). _(src: WWDC23 Session 10095 'Explore rendering for spatial computing'; OcclusionMaterial docs; HIG Materials)_

### Key APIs
- **`.glassBackgroundEffect(displayMode:)`** _(min visionOS 1.0+)_ — visionOS-native frosted/glass background for SwiftUI views, including RealityView attachment root views
  - When: Default glass for our card/panel/note attachment root views over passthrough. The established, battle-tested visionOS window/attachment glass.
  - ⚠️ Renders opaque over a custom skybox/immersive background (confirmed Apple limitation, FB-worthy). Fine over real passthrough in a volumetric WindowGroup (our case).

- **`System materials: .ultraThinMaterial/.thinMaterial/.regularMaterial/.thickMaterial + vibrancy`** _(min visionOS 1.0+ (materials); vibrancy via system text styles)_ — Layered translucent backgrounds that encode elevation; primary/secondary/tertiary vibrancy for legible text
  - When: The token vocabulary for our design system: map entity kind -> material level. .thin for interactive, .regular for section/panel, .thick for recessed.
  - ⚠️ Don't stack two light materials without a darker separator; don't use colored text on glass. Let system vibrancy handle contrast.
- **`ViewAttachmentComponent(rootView:)`** _(min visionOS 26.0+ (old Attachment(id:) builder is visionOS 1.0+))_ — Attach any SwiftUI view to a RealityKit entity from anywhere in code (visionOS 26 attachment model)
  - When: Replace the old builder-based RealityView attachments closure. Each SpatialEntity card becomes Entity(components: ViewAttachmentComponent(rootView: ...)). This is the seam where SwiftUI material styling lives.
  - ⚠️ Pair with InputTargetComponent + CollisionComponent on the gesture target. Coordinate space: 1 point = 1 meter, y-up.
- **`ManipulationComponent + ManipulationComponent.configureEntity(_:hoverEffect:allowedInputTypes:collisionShapes:)`** _(min visionOS 26.0+)_ — Built-in grab/move/rotate/scale with automatic hover effect, collision, and input-target wiring; release behavior hooks
  - When: The ROOT FIX for our drag system. Replaces hand-rolled DragGesture + per-pass position re-assertion. configureEntity auto-adds Manipulation/Collision/InputTarget/HoverEffect components and owns the transform during drag.
  - ⚠️ Use releaseBehavior:.stay to keep dragged position; subscribe to ManipulationEvents.WillRelease to persist/animate. hoverEffect:.spotlight/.highlight gives the highlight our 'x' control lacks.
- **`Entity Observable + entity.animate(using:) / content.animate { }`** _(min visionOS 26.0+)_ — Entities are Observable; SwiftUI animations can implicitly drive Transform/color/light/audio component changes
  - When: Smooth animations (snap-back-on-release, jitter on highlight, layout transitions) and to drive SwiftUI state from entity transforms. Core of the animation architecture.
  - ⚠️ DO NOT write observed entity state inside RealityView update closure (infinite loop — exactly our reconcile-per-pass bug pattern). Guard writes: if entity.position != new { ... }. Mutate in systems, gesture closures, or make closure.
- **`HoverEffectComponent / .hoverEffect(_:)`** _(min visionOS 1.0+ (component); .hoverEffect SwiftUI variants 1.0+, richer styles in 26)_ — Eye-tracking highlight feedback on entities (RealityKit) or attachment views (SwiftUI)
  - When: Every interactive entity AND the 'x' dismiss control. SwiftUI .hoverEffect(.highlight) inside the attachment, or HoverEffectComponent on the entity. Custom 'jitter' can be driven by reacting to hover via a custom component/system + entity.animate.
  - ⚠️ Our 'x' fails to highlight because the parent card CollisionComponent (depth max(z,0.03)) swallows it. Fix with proper child collision shape + InputTargetComponent precedence, or make 'x' a separate sibling entity, not nested inside the card's collision volume.
- **`CoordinateSpace3D / GeometryProxy3D.coordinateSpace3D() / convert(_:from:to:)`** _(min visionOS 26.0+)_ — Unified points<->meters and axis conversion between SwiftUI and RealityKit
  - When: Converting drag/gesture values and layout-engine meter positions into entity space cleanly instead of value.convert(... .local,.scene) ad hoc math.
  - ⚠️ Entity and Scene conform to CoordinateSpace3D; use it instead of manual unit assumptions.
- **`PhysicallyBasedMaterial.blending = .transparent(opacity:) + opacityThreshold + faceCulling`** _(min visionOS 1.0+)_ — Transparency on RealityKit meshes we draw (edges, ports, volume shells)
  - When: Only for actual mesh geometry, not SwiftUI cards. Set blending for translucency; set opacityThreshold (even 0.0) to dodge FB12605220 culling; faceCulling .none for double-sided.
  - ⚠️ No 'opacity' property in code (only in Reality Composer Pro) — set .blending. Overlapping transparent meshes sort incorrectly without ModelSortGroup.
- **`ModelSortGroup / ModelSortGroupComponent`** _(min visionOS 1.0+)_ — Explicit draw-order control for overlapping translucent meshes (depth-pass: pre/post/none)
  - When: Our derived edges/ports crossing translucent cards. Render smallest/innermost first, largest/outermost last to stop flicker and vanishing lines.
  - ⚠️ Each depth-pass strategy trades color correctness vs outline visibility; no perfect option for dynamic intersecting transparency — keep edges thin and minimize overlap.
- **`OcclusionMaterial / GroundingShadowComponent + UnlitMaterial(applyPostProcessToneMap:false)`** _(min visionOS 1.0+)_ — Depth cues: real grounding shadows and invisible occluders; tone-map-free color for UI-accurate mesh color
  - When: GroundingShadowComponent on cards/volumes for floating-depth readability. OcclusionMaterial plane for shadow catch or hiding. UnlitMaterial(...,applyPostProcessToneMap:false) so any colored mesh (ports/edges) matches SwiftUI palette.
  - ⚠️ Tone mapping is ON by default and shifts UI colors — disable for menus/HUD/UI-colored meshes per WWDC23 guidance.

### Anti-patterns
- **Re-asserting card.position to the stored model value on EVERY RealityView update pass** — This is the root of the drag snap-back. WWDC25 explicitly warns: do not modify observed entity state in the update closure (creates a fight/loop). The renderer's update pass must not own transforms during a gesture. Use ManipulationComponent (owns transform during drag) + persist on ManipulationEvents.WillRelease; if you must write in update, guard with if entity.position != new.
- **Building UI look from RealityKit primitive meshes / 3D text / SimpleMaterial planes behind attachments** — Already rejected in our project and contradicts the locked paradigm. SwiftUI must render content; RealityKit only places it. Mesh-backed 'glass' (SimpleMaterial(color:.blue.withAlphaComponent(0.5))) is the forum workaround for the skybox bug — not needed over passthrough and visually inferior to real glass + vibrancy.
- **Nesting the 'x' dismiss control inside the parent card's collision volume** — The parent CollisionComponent (depth max(z,0.03)) encloses the child 'x' (+0.006 z) and swallows hover/taps, so the SwiftUI .hoverEffect scale never fires. There is no hierarchical collision precedence by default. Fix: make controls siblings with their own tight collision + InputTargetComponent, or use one ViewAttachmentComponent whose SwiftUI root contains the button (SwiftUI handles intra-view hit-testing), not a separate RealityKit child poking through the card's collider.
- **Relying on a SwiftUI .hoverEffect{ scaleEffect } inside an attachment to scale a RealityKit child entity** — SwiftUI hover state inside an attachment drives the SwiftUI view's own rendering, not a sibling/child RealityKit entity's transform. For entity-level highlight/jitter, drive a RealityKit HoverEffectComponent or react to hover in a system and animate via entity.animate.
- **Ad-hoc per-card .background(.regularMaterial) / .opacity / stroke calls scattered across views** — Produces an inconsistent visual language and makes global changes (accessibility, theming, elevation) impossible. Replace with design tokens: a Material/Tint/CornerRadius/Elevation enum keyed by SpatialEntityKind, applied via one shared ViewModifier. HIG's material-level=elevation mapping makes this principled.
- **Colored text on glass and stacking two light materials without a darker separator** — HIG explicitly calls these out as legibility failures. Use system text styles + vibrancy (primary/secondary/tertiary) and alternate material darkness to preserve contrast over arbitrary passthrough.
- **Drawing many overlapping translucent edge/port meshes and trusting default depth sorting** — RealityKit transparency is order-dependent; overlapping translucent surfaces flicker and lose outlines. Requires ModelSortGroup + opacityThreshold. Keep edges thin/opaque-ish, minimize overlap, or render edges with UnlitMaterial and a sort group.
- **Leaving RealityKit tone mapping enabled on UI-colored meshes** — Tone mapping (on by default) shifts colors so port/edge colors won't match the SwiftUI type palette. Use UnlitMaterial(applyPostProcessToneMap:false) for color-accurate UI geometry.

### Recommendations for our app
- Adopt a token-driven material system: define a SurfaceStyle token (material level, tint color, corner radius, elevation, stroke) keyed by SpatialEntityKind (taskNode/stepNode/typePanel/workflowVolume/note). Apply via ONE shared SwiftUI ViewModifier so all cards share a cohesive translucent language instead of per-view .background calls. This directly answers the 'scalable component/design system' goal.
- Use .glassBackgroundEffect() as the default card surface over passthrough (it works in our volumetric WindowGroup; the opaque-over-skybox bug does not apply). Layer with HIG semantics: typePanel = .regular (section), nodes = .thin (interactive), recessed/secondary content = .thick. Reserve the visionOS 26 .glassEffect/GlassEffectContainer for grouped controls (toolbar, the 'x' + future per-card buttons) so they morph cohesively.
- Replace the hand-rolled DragGesture + per-pass position re-assertion with ManipulationComponent via ManipulationComponent.configureEntity(card, hoverEffect: .spotlight/.highlight, allowedInputTypes:.all, collisionShapes:[...]). Set releaseBehavior:.stay and subscribe to ManipulationEvents.WillRelease to persist position to the SpatialEntity. This makes grab-and-drag near-trivial and eliminates snap-back at the root because RealityKit owns the transform during the gesture — not reconcile.
- Establish explicit transform ownership: the layout engine/model owns position ONLY when an entity is in auto-flow and not being manipulated; the gesture/ManipulationComponent owns it during drag; entity.animate owns it during transitions. Stop writing transforms in the RealityView update closure; move reconcile logic to a custom RealityKit System and guard all writes with if entity.position != target. This is the architectural fix WWDC25 prescribes for the infinite-loop/fight pattern.
- Fix the 'x' control by putting it INSIDE the card's SwiftUI attachment root (so SwiftUI does the hit-testing and .hoverEffect works) rather than as a separate RealityKit child entity poking through the parent's collider. If a control must be a separate entity, give it its own tight CollisionComponent + InputTargetComponent positioned clearly outside the parent's collision box, and drive highlight via HoverEffectComponent.
- Add depth cues for floating cards: GroundingShadowComponent(castsShadow:true) on nodes/volumes and subtle specular via the glass material. This restores the spatial-depth readability that flat glass cards lack.
- Implement playful 'jitter on highlight' via entity.animate(using:) reacting to hover (HoverEffectComponent state read in a System, or onHover in the SwiftUI attachment toggling an entity transform with a spring/easeInOut), NOT via SwiftUI scaleEffect inside the attachment which cannot move the entity.
- For derived edges/ports (real RealityKit meshes): keep them thin and use UnlitMaterial(...,applyPostProcessToneMap:false) for color-accurate type tints; if they visibly cross translucent cards, assign a ModelSortGroupComponent and set opacityThreshold to avoid sorting flicker and the FB12605220 culling bug. Rebuild edges incrementally (diff) inside a System rather than delete-and-re-add-all every pass.
- Migrate attachments to ViewAttachmentComponent and gestures toward GestureComponent/targetedToEntity with CoordinateSpace3D conversion — this is the visionOS 26 idiom, removes our name-parsing gesture disambiguation, and gives clean points<->meters conversion for the layout engine.
- Lean on automatic accessibility: do NOT override Reduce Transparency / Increase Contrast / Reduce Motion; let system materials and Liquid Glass adapt. Keep text on system styles + vibrancy (primary/secondary/tertiary); ban hardcoded text colors in the token layer.
- Validate every material/glass choice with xcodebuild -scheme TaskPlannerVision -destination 'generic/platform=visionOS Simulator' (SourceKit reports false errors) and visually in Simulator, since glass behavior in RealityView attachments is the kind of thing the compiler accepts but renders wrong.

## Apple Human Interface Guidelines & ergonomics for visionOS spatial design — translated to rules for a draggable-card task-planner volume

For visionOS 26.2, the right container for a workspace of movable cards/panels/notes is a Volume launched in the Shared Space (not an immersive space); volumes are for self-contained 3D experiences where movable 3D content is the primary point. The single biggest finding is that both known bugs are symptoms of fighting the platform instead of using its modern interaction stack. Bug #1 (drag snap-back) is a transform-ownership problem that visionOS 26's ManipulationComponent solves outright: it owns the entity transform during a pinch-drag (with z-telescoping, rotate, two-handed scale, audio), and releaseBehavior=.stay persists the moved position — making grab-and-drag genuinely trivial. The deeper root cause is mutating @Observable entity transforms inside RealityView's update closure, which WWDC25 #274 explicitly warns against; combined with GestureComponent (per-entity gestures in entity coordinate space) and the Unified Coordinate Conversion API, this lets the architecture drop the reconcile-reasserts-position pattern, the name-prefix gesture parsing, and the manual point-to-meter math. Bug #2 (the × not highlighting) is a target-size + hierarchical-collision failure: the × is far under the 60 pt (≈2.5°, ≈4.4 cm @ 1 m) minimum and its collision is enclosed by the parent card's 30 mm-deep collision box that swallows hover/taps; a SwiftUI .hoverEffect inside a ViewAttachmentComponent also won't visibly scale a placed entity. Ergonomically, primary content belongs centered and slightly below eye level beyond arm's length at consistent depth (the current upper-back panel row strains neck/eyes), legibility comes from glass + vibrancy rather than solid/colored fills, and any playful "jitter" must be low-amplitude, gated behind Reduce Motion with a crossfade fallback, plus dynamic world scaling and full accessibility (AccessibilityComponent, VoiceOver, Dwell). These are all battle-tested, shipping visionOS 26 APIs.

### Core principles
- **Prefer the platform's combined manipulation gesture over a hand-rolled DragGesture** — ManipulationComponent (visionOS 26) is the answer to 'grab-and-drag should be near-trivial.' It owns the transform during interaction, telescopes z so far cards can be drawn close, supports rotate and two-handed scale, plays spatial audio, and offers releaseBehavior=.stay to keep the moved position. A plain DragGesture has a known confusion (movement stays relative to original facing direction if the user turns) that Manipulation solves for free. Replaces the custom translation-grab dragGesture and removes any need to suppress reconcile during drag. _(src: WWDC25 #274 'Better together: SwiftUI and RealityKit'; 'Deep Dive into Manipulation on visionOS' (stepinto.vision))_
- **Never mutate observed entity transforms inside RealityView's update closure** — This is the literal root cause of bug #1. WWDC25 #274 states entities are Observable in visionOS 26 and warns that modifying observed entities in update creates infinite loops / fights live interaction. Safe mutation zones are gesture closures, custom RealityKit systems, and the make closure. The current reconcile() re-asserts every card's stored position every update pass, and the stored position is stale mid-drag — so the card snaps to the data-model value. Stop driving transforms from update; drive them from the gesture/manipulation and persist on release. _(src: WWDC25 #274 'Better together: SwiftUI and RealityKit')_
- **60 pt (≈2.5°, ≈4.4 cm @ 1 m) minimum target, ≥4 pt spacing, and each target needs its own non-enclosed collision** — Gaze targeting requires every interactive element occupy ≥60 pt of space (visually smaller is OK if total allocated space is 60 pt) with ≥4 pt spacing to survive eye-tracking drift. Bug #2's × fails twice: it is tiny AND its collision is enclosed by the parent card's collision box (parent depth max(z,0.03)=30 mm vs child only +6 mm in front), so the parent swallows hover/tap. Targets need their own collision not contained by a parent/sibling volume; rounded shapes (circle/pill) are easier to look at than sharp corners (the eye fixates on sharp edges). _(src: WWDC23 #10073 'Design for spatial input'; WWDC25 #303 'Design hover interactions for visionOS'; HIG Eyes)_
- **Highlight every custom interactive element on gaze; never reveal a control that pulls the eye away** — System controls highlight automatically; custom ones (cards, ×, ports) must add a highlight whose shape matches the content, and it must be RealityKit-visible (a SwiftUI .hoverEffect{scaleEffect} inside a ViewAttachmentComponent rootView does not reliably scale the placed entity — use HoverEffectComponent with .highlight/.spotlight, or a custom hover effect). Hover runs outside the app's process for privacy, so highlight is the only feedback channel pre-gesture. WWDC25 #303 warns specifically against hidden close buttons that appear on hover: the eye jumps to the new control and triggers accidental dismissal — exactly the × redesign risk. Use the instant/delayed/ramp animation categories deliberately, keep an anchoring element static, keep effects small. _(src: WWDC25 #303 'Design hover interactions for visionOS'; WWDC23 #10073)_
- **Place primary content centered and slightly below eye level, beyond arm's length, at consistent interaction depth** — Eyes turn down and laterally comfortably; upward/diagonal rotation fatigues. Put frequently-used content centered and slightly below the sight line; reserve periphery and high positions for secondary/brief content. Extended-reading content goes beyond arm's length; the closest zone is for momentary effort or direct touch only. Keep interactive elements at consistent depth to avoid constant refocusing. The current type-cluster layout (panels in a row across the upper-back, tasks stacked beneath) places primary content high and far — a neck/eye-strain pattern the layout engine should correct. _(src: WWDC23 #10078 'Design considerations for vision and motion'; HIG Designing for visionOS)_
- **Adopt glass + vibrancy for legibility; never opaque windows or colored text** — Glass is an adaptive material that modulates background colors for optimal contrast and grounds content in the room. Use vibrancy tiers (.primary for standard text, .secondary for footnotes/subtitles, .tertiary for inactive/low-legibility) rather than solid colors, which have fixed contrast and can't adapt to passthrough. Prefer white/light text, avoid colored text, and stack a darker material beneath lighter ones behind text. Avoid opaque windows — they kill spatial awareness. This is the platform-correct way to get the desired elegant semi-transparent glass. _(src: 'Ensuring interface legibility and contrast in visionOS' (createwithswift); WWDC23 #10110 'Elevate your windowed app for spatial computing')_
- **Avoid head-anchored content and vection-inducing motion; provide Reduce Motion crossfade fallbacks** — Prefer world-locked placement (this app's persisted positions are already correct). Make any moving virtual object semitransparent so passthrough stays visible and the brain doesn't read object motion as self-motion. Avoid abrupt movement, oscillation near 0.2 Hz (one cycle / 5 s), and fast/large objects passing close. For the desired playful 'jitter on highlight': keep amplitude low, gate it behind accessibilityReduceMotion, substitute a crossfade/static state when Reduce Motion is on. Head-anchored content also breaks Zoom for low-vision users — honor accessibilityPrefersHeadAnchorAlternative if any head-locked UI is ever added. _(src: WWDC23 #10078 'Design considerations for vision and motion'; WWDC23 #10034 'Create accessible spatial experiences')_
- **Use a Volume in the Shared Space; enable dynamic world scaling and handle Window Zoom/resize** — Volumes are for self-contained 3D experiences where movable 3D content is the primary point — exactly this app. Apps should launch in the Shared Space (window/volume), not jump straight to an immersive space. Use .defaultWorldScalingBehavior(.dynamic) so UI grows as it recedes, preserving angular target area (the 60 pt rule) and FOV coverage; fixed scale shrinks targets to unusable sizes. Also scale the RealityView root by rendered-volume-size / defaultSize (via GeometryReader3D / coordinate conversion), or content clips at non-default Window Zoom and user resize. _(src: HIG Designing for visionOS; WWDC23 #10073; 'If you've created a visionOS app with a volume, you probably did it wrong' (lunarskydiving))_
- **Disambiguate interaction by typed component / per-entity identity, not by parsing entity.name strings** — visionOS 26's GestureComponent attaches gestures to specific entities and reports values in that entity's coordinate space, and the Unified Coordinate Conversion API (CoordinateSpace3D on Entity/Scene; GeometryProxy3D.coordinateSpace3D()) replaces manual point-to-meter math. This obsoletes the current 'ctl::/edge:: name-prefix parsing' (which also overloads entity.name as the id). Express role/identity via real components/custom component types per entity. _(src: WWDC25 #274 'Better together: SwiftUI and RealityKit')_
- **Make the spatial scene accessible from day one** — Add AccessibilityComponent to each interactive entity (isAccessibilityElement, label, value, traits like .button, systemActions=[.activate]) so VoiceOver users can navigate and activate cards. Support VoiceOver Direct Gesture mode (post AccessibilityNotification.Announcement for state changes), Dwell Control (all actions reachable via tap/scroll/long-press/drag), Reduce Motion (crossfade), and Reduce Transparency. Spatial experiences can be fully usable without precise vision or movement — design for that, not as an afterthought. _(src: WWDC23 #10034 'Create accessible spatial experiences'; 'Improving accessibility support in your visionOS app' (Apple docs))_

### Key APIs
- **`ManipulationComponent`** _(min visionOS 26)_ — Combined pinch-and-hold gesture to move/rotate/scale any entity with z-telescoping, two-handed scale, hand-off, and spatial audio — the production fix for drag snap-back.
  - When: On every movable card/note/panel/workflow-volume entity, replacing the hand-rolled DragGesture. Set releaseBehavior=.stay to persist; subscribe to ManipulationEvents.WillEnd to save the transform via tRPC. Configure via ManipulationComponent.configureEntity(entity, hoverEffect:, allowedInputTypes:, collisionShapes:); tune .dynamics (translationBehavior, primary/secondaryRotationBehavior, scalingBehavior, inertia). Requires CollisionComponent + InputTargetComponent (+ HoverEffectComponent).
  - ⚠️ Default releaseBehavior=.reset returns the entity home on release (looks like snap-back) — must set .stay or persist in WillEnd. ManipulationEvents are WillBegin / DidUpdateTransform (per-frame, for clamping to VolumeMetrics bounds) / WillRelease / WillEnd / DidHandOff. Set audioConfiguration=.none for custom audio. Known system-wide bug where translation can stop until device restart.
- **`GestureComponent`** _(min visionOS 26)_ — Attach SwiftUI gestures (Tap, etc.) directly to a specific entity; values reported in that entity's coordinate space.
  - When: For tap-to-edit, connect/link mode taps, and the × dismiss action — per entity, instead of two top-level .targetedToAnyEntity() gestures disambiguated by name parsing.
  - ⚠️ Still requires InputTargetComponent + CollisionComponent on the targeted entity. Lets you retire the ctl::/edge:: name-prefix scheme.
- **`HoverEffectComponent`** _(min visionOS 1 (custom hover effects expanded in visionOS 2/26))_ — RealityKit-visible gaze highlight on a placed entity (system .highlight, .spotlight, or a custom hover effect).
  - When: On every interactive entity (cards, ×, ports). This is what bug #2 actually needs — a SwiftUI .hoverEffect inside the attachment's rootView does not reliably scale the entity.
  - ⚠️ The effect's shape should match content; for the × use a rounded shape ≥60 pt. Custom effects animate via instant/delayed/ramp categories (WWDC25 #303).
- **`Entity.animate(with:) / RealityViewContent.animate(with:)`** _(min visionOS 26)_ — SwiftUI-style implicit animation of RealityKit Transform/Model/Light/Audio components (e.g., .bouncy, .spring).
  - When: Smooth card movement on auto-layout reflow, settle-on-release, and the low-amplitude 'jitter' on highlight (gated behind Reduce Motion).
  - ⚠️ Do not drive these from the update closure on observed entities; run from gesture closures or systems. Honor accessibilityReduceMotion.
- **`defaultWorldScalingBehavior(.dynamic)`** _(min visionOS 2 (Apple's recommended default behavior for spatial input))_ — Scene modifier making window/volume UI scale up as it moves away so angular target size (the 60 pt rule) and FOV coverage stay constant.
  - When: On the volumetric WindowGroup so cards remain readable/targetable at distance.
  - ⚠️ Distinct from RealityKit content scaling for Window Zoom/user-resize — still scale the RealityView root by rendered-size/defaultSize via GeometryReader3D / coordinate conversion or content clips.
- **`Unified Coordinate Conversion API (CoordinateSpace3D)`** _(min visionOS 26)_ — Convert points between SwiftUI and RealityKit scene space; Entity and Scene conform to CoordinateSpace3D, plus GeometryProxy3D.coordinateSpace3D().
  - When: Anywhere the code currently does value.convert(translation3D, from:.local, to:.scene) or hand-rolled point-to-meter math (edges, spawn points, drag deltas).
  - ⚠️ Use gesture.values(in: entity.coordinateSpace3D) to get values already in the right space.
- **`ViewAttachmentComponent`** _(min visionOS 1 (inline creation visionOS 26))_ — Declare a SwiftUI view inline on a RealityKit entity (the locked 'SwiftUI renders content, RealityKit places it' pattern).
  - When: Keep using it for all card/panel/note visuals; in visionOS 26 it can be created inline rather than via the RealityView attachments builder, which also resolves the make-time attachment race the current code works around.
  - ⚠️ A control rendered as a child attachment still needs its own InputTargetComponent + non-enclosed CollisionComponent + HoverEffectComponent to receive gaze/taps.
- **`AccessibilityComponent + AccessibilityNotification.Announcement`** _(min visionOS 1)_ — Make RealityKit entities navigable by VoiceOver (isAccessibilityElement, label, value, traits, systemActions) and announce state changes.
  - When: On every interactive entity; announce merges/links/dismissals and Direct Gesture mode events.
  - ⚠️ In VoiceOver Direct Gesture mode, activate actions are unavailable — must post announcements for meaningful events.
- **`accessibilityReduceMotion / accessibilityReduceTransparency / accessibilityPrefersHeadAnchorAlternative`** _(min visionOS 1 (head-anchor-alternative visionOS 2))_ — SwiftUI environment values to honor accessibility settings.
  - When: Gate jitter/animation behind Reduce Motion (crossfade fallback); reduce glass translucency for Reduce Transparency; avoid/relocate head-anchored UI when the head-anchor-alternative is preferred.
  - ⚠️ Reduce Motion fallback should be a crossfade, not simply removing feedback.
- **`PresentationComponent / SpatialContainer + depthAlignment`** _(min visionOS 26)_ — PresentationComponent presents SwiftUI popovers/sheets from an entity; SpatialContainer + depthAlignment lay out views in 3D (overlapping bounding box, depth alignment).
  - When: PresentationComponent for the edit form anchored to a tapped card (instead of a flat .sheet); SpatialContainer/depthAlignment/spatialOverlay for meaningful subtle depth in cards and clusters.
  - ⚠️ Keep interaction depth consistent; use depth for hierarchy, not gratuitous parallax that forces refocusing.

### Anti-patterns
- **Re-asserting each entity's stored transform from RealityView's update/reconcile closure every pass** — Direct cause of bug #1 drag snap-back: the data-model position is stale mid-drag, and mutating observed entities in update fights live interaction and risks infinite loops (WWDC25 #274). Transforms must be owned by the gesture/ManipulationComponent during interaction and persisted on release.
- **A @State draggingID flag set inside a RealityKit gesture .onChanged to skip the dragged card in update** — The developer already found this doesn't work: @State mutated in a gesture closure isn't reliably visible to the update closure before the next pass. It's patching the wrong layer — the fix is to not drive transforms from update at all.
- **Hover-revealed or undersized child controls (the ×) nested inside a parent's collision volume** — Bug #2. A child 6 mm in front of a 30 mm-deep parent collision box is swallowed by the parent; a SwiftUI hoverEffect inside a ViewAttachmentComponent doesn't visibly scale the placed entity. Apple also warns hover-revealed controls yank the eye and cause accidental dismissal. The × must be ≥60 pt, rounded, persistent (or deliberately ramp-revealed), with its own non-overlapping collision and a HoverEffectComponent highlight.
- **Placing primary content high and far in the upper-back of the volume** — Upward/diagonal eye rotation is the most fatiguing; primary content belongs centered and slightly below eye level beyond arm's length. The type-cluster panel row forces neck/eye strain for the most-used elements.
- **Solid/colored fills and text on cards, or opaque card backgrounds** — Solid colors have fixed contrast and can't adapt to arbitrary passthrough, so legibility breaks against bright/busy rooms; opaque windows kill spatial awareness. Use glass + vibrancy tiers and white/light text instead.
- **Custom gestures and unconstrained playful motion without accessibility fallbacks** — Custom gestures should be rare with UI fallbacks; uncontrolled jitter/oscillation (esp. ~0.2 Hz) induces discomfort/vection and excludes Reduce Motion users. Motion must be low-amplitude, semitransparent if moving, and crossfade when Reduce Motion is on.
- **Encoding gesture roles by parsing entity.name string prefixes (ctl::, edge::)** — Brittle and overloads name (also used as the entity id). visionOS 26 GestureComponent + per-entity custom components express role/identity natively; the Unified Coordinate Conversion API replaces manual point-to-meter math.
- **Deleting and re-adding ALL edges and ports from scratch on every reconcile pass** — Wasteful churn that also recreates ModelEntity meshes/materials each frame; combined with re-asserting transforms it amplifies the update-loop instability. Edges should be persistent entities whose endpoints update (ideally driven by a RealityKit system observing entity positions), not rebuilt each pass.

### Recommendations for our app
- Replace the hand-rolled DragGesture with ManipulationComponent on every movable entity: configure via ManipulationComponent.configureEntity(card, hoverEffect: .highlight, allowedInputTypes: .indirect, collisionShapes:[box]), set releaseBehavior=.stay, and subscribe to ManipulationEvents.WillEnd to call viewModel.endDrag (persist via tRPC). Use ManipulationEvents.DidUpdateTransform to clamp into VolumeMetrics bounds. This makes grab-and-drag trivial and deletes the draggingID workaround entirely.
- Stop driving transforms from reconcile/update. In update only add/remove entities and wire components; let the gesture/ManipulationComponent own the live transform and write position to the entity only when the data model genuinely changes (e.g., auto-layout reflow of non-hand-moved entities), animated with entity.animate(with:.spring). This fixes bug #1 at the root and aligns with WWDC25 #274's 'don't mutate observed entities in update' rule.
- Rebuild the × as a real ≥60 pt rounded target with its own collision NOT enclosed by the card: either move it far enough off the card face (or out to the side) that the parent's 30 mm-deep box doesn't contain it, or give the card a precise (non-inflated-depth) collision and the × a sibling collision. Add HoverEffectComponent (.highlight) so it visibly responds to gaze, and avoid hover-to-reveal (persist it, or ramp-reveal slowly per WWDC25 #303) to prevent accidental dismissal. This fixes bug #2.
- Adopt GestureComponent + the Unified Coordinate Conversion API to attach tap/dismiss/connect-link handlers per entity, and retire the ctl::/edge:: name-prefix parsing and the value.convert point-to-meter math. Stop overloading entity.name as both id and role carrier — store role in a typed custom component.
- Revise the SpatialLayoutEngine so the most-used clusters sit centered and slightly below eye level within a comfortable FOV cone, not across the upper-back; keep interactive content at consistent depth and reserve high/peripheral positions for secondary content. Add .defaultWorldScalingBehavior(.dynamic) to the volumetric WindowGroup and scale the RealityView root by rendered-volume-size/defaultSize so cards stay readable at distance and don't clip at non-default Window Zoom.
- Build the card/panel/note design system on glass + vibrancy: glassBackgroundEffect, foregroundStyle(.primary/.secondary/.tertiary) for text hierarchy, white/light text, no colored text, darker material stacked behind text; honor accessibilityReduceTransparency. This delivers the elegant semi-transparent glass aesthetic the platform way.

## Cross-platform XR/VR UI architecture patterns for complex, scalable spatial UIs (mapped to a SwiftUI+RealityKit visionOS task planner)

Every mature XR UI stack converges on the same shape: a layered pipeline (data/state -> interaction -> layout -> scene-graph placement -> visual render) with ONE explicit transform owner at each instant, and strict separation of interaction logic from visual presentation from data. MRTK3 is the clearest statement: interactables are "pure state and interaction" that render nothing; a separate "visual driver" (StateVisualizer) observes that state to produce feedback; and the interactor's attachTransform "IS the interaction logic," so the manipulated object never cares which input produced the pose. Unity XRI formalizes a Manager-as-intermediary running a deterministic per-frame loop (validate -> exit events -> enter events, interactor-before-interactable) over the four states Hover/Select/Focus/Activate. Meta's Interaction SDK splits the same way (Interactor+Interactable pair; Grabbable/Pointable carry transform logic). ECS-for-UI experience (Bevy/Leafwing, Flecs, Unity DOTS) shows retained-mode entities + change detection + a clean layout/render split scale well, but one-off UI callbacks, hierarchy traversal, and multi-entity widgets need first-class abstractions or they become boilerplate. The decisive finding: RealityKit is itself an ECS, and visionOS 26 (the deploy target) ships exactly the primitives that dissolve both known bugs — ManipulationComponent (visionOS 2.6+) owns the transform during a drag so the data model stops fighting it; Observable entities plus the rule "never mutate observed state inside RealityView.update" remove the reconcile feedback loop; and the hierarchical InputTargetComponent/CollisionComponent model plus per-entity HoverEffectComponent fix the swallowed-hover "x" control. The synthesis: stop driving transforms imperatively from a monolithic reconcile pass; move continuous concerns into RealityKit Systems keyed by custom Components, let ManipulationComponent own live drag, let SwiftUI attachments render visuals as state observers, and derive edges in a System instead of rebuilding them every update.

### Core principles
- **Separate interaction logic from visual presentation from data — the most repeated lesson across MRTK3, XRI, and Meta ISDK** — MRTK3 states it bluntly: interactables are pure state and interaction; the interactable doesn't render any visual changes based on its internal state. Visual feedback is produced by a SEPARATE visual driver (StateVisualizer) that observes the interactable's state (IsGazeHovered, IsGrabSelected, Selectedness). The same PressableButton logic can drive a squishy ball, a flat trackpad, or a network event with zero changes. For our app: the card visuals (glass, jitter-on-hover, highlight) must be a function of an observed interaction-state value, NOT logic baked into reconcile. Today reconcile both decides state and mutates visuals — the exact MRTK2 mistake MRTK3 retired. _(src: MRTK3 Interactables architecture + State Visualizer (Microsoft Learn))_
- **Exactly one authority owns an entity's transform at any instant; ownership transfers explicitly between data-model, live-gesture, and animation** — This is the root of bug #1. In XRI/MRTK the interactor's attachTransform owns the pose during manipulation; the data model only reads it. In visionOS 26, ManipulationComponent makes this native: during manipulation the component owns transform updates. Constraints are applied by subscribing to ManipulationEvents.DidUpdateTransform and writing the clamped value back inside that callback; the data model never re-asserts position mid-drag. General rule (WWDC25 274 boundary guidance): a transform is owned by a System/ManipulationComponent during interaction, by SwiftUI-driven animation during a transition, or by the data projection at rest — and writes from one owner must be suppressed while another owns it. The current code has no ownership concept: reconcile re-asserts the stored position on EVERY pass, fighting the live gesture. _(src: WWDC25 Session 274; Step Into Vision Manipulation deep dive; MRTK3 Interactor architecture (attachTransform))_
- **Never mutate observed state inside the reactive update closure — the canonical feedback loop, and precisely our drag snap-back** — Apple (WWDC25 274): to avoid an infinite loop, don't modify observed state within your update closure; you may modify entities you're not observing. Safe write sites: custom RealityKit Systems, gesture closures (run on input, not body evaluation), and the make/setup closure. RealityView.update is NOT a game tick — it runs only when referenced state changes, so writing observed state there schedules another update. Our reconcile() does exactly the forbidden thing: it runs on every Observable change AND re-sets card.position from the stale view-model each pass. The @State draggingID guard fails because gesture-set @State isn't reliably visible to the next update closure. The fix is structural: move transform authority out of update entirely. _(src: WWDC25 Session 274; Step Into Vision update-closure; Apple Developer Forums 733020)_
- **The layered pipeline state -> interaction -> layout -> scene-graph -> render, with an explicit handoff at each boundary** — Bevy's ECS GUI (Leafwing) decouples layout (taffy/flexbox produces size/position) from a separate render pass from a separate picking system that maps pointer events to targets — enabling swapping layout strategies without touching rendering. This validates our existing pure SpatialLayoutEngine and argues for keeping it strictly nonisolated, downstream of state and upstream of placement. Boundary discipline: the layout engine outputs target poses for AUTO-FLOW entities only; hand-moved/manipulated entities are excluded (owned by the gesture/manipulation layer); placement writes layout output to an entity only when no higher-priority owner holds it. _(src: Leafwing ECS-backed GUI framework; MRTK3; Unity XRI architecture)_
- **A Manager/loop as deterministic intermediary between input sources and targets, with ordered phases and a four-state model (Hover, Select, Focus, Activate)** — Unity XRI's Interaction Manager acts as an intermediary between Interactors and Interactables, running a deterministic per-frame loop: validate existing states -> dispatch exit events -> dispatch enter events, with interactors always notified before interactables. Targets are prioritized (closest first) via pluggable Target filters, and Interaction Groups allow only one member to interact at once with continuity bias. The four states are a proven vocabulary. Our app currently disambiguates interaction by PARSING entity names (ctl::role::ownerId, edge::) inside gesture handlers — fragile and unscalable. The mature pattern is typed components/roles + a small mediator, not string parsing. _(src: Unity XR Interaction Toolkit Architecture @3.3; Meta Interaction SDK lifecycle)_
- **Retained-mode scene entities + change detection (dirty flags/observers) beat immediate-mode rebuild-everything for complex persistent spatial scenes** — Retained mode lets the library transfer to the renderer only data that changed and perform occlusion culling. RealityKit is retained-mode ECS: a System calls a single update method per frame over entities matched by EntityQuery, reading per-entity Component state. The anti-pattern our code commits is immediate-mode-in-a-retained-world: it deletes and re-adds ALL edges + ports from scratch every reconcile pass and rebuilds the x control each pass. Retained approach: create entities once, mutate only changed components, query-and-update via Systems, and let RealityKit diff. visionOS 26 Observable entities make the reverse direction (entity -> SwiftUI) reactive without polling. _(src: Microsoft Learn Retained vs Immediate Mode; RealityKit ECS; MRTK3 StateVisualizer (polling noted as a perf compromise))_
- **Make input/collision hierarchy explicit; hover/highlight/focus must drive something the renderer actually shows** — Root of bug #2. InputTargetComponent and CollisionComponent are HIERARCHICAL: descendants with collision components can be targeted for gestures — but a parent collision box that encloses a child (our x is only +0.006 z inside a depth max(z,0.03) parent box) means the parent's hit region overlaps the child, so input/hover resolves ambiguously and the parent swallows it. Fix: give the child its own non-overlapping collision shape that protrudes beyond the parent (or carve the parent's collision), and give each interactive sub-entity its own InputTargetComponent + HoverEffectComponent. A SwiftUI .hoverEffect inside an attachment rootView highlights only if that attachment entity is itself the eye-tracking target; the reliable native highlight is HoverEffectComponent on the entity. For playful jitter, drive a transform/shader effect from an observed hover state in a System or via Entity.animate, not a SwiftUI scaleEffect the parent swallows. _(src: visionOS component docs (AppleInsider, Step Into Vision); WWDC24 Explore game input in visionOS (10094); AGENTS.md visionOS 26)_
- **attachTransform / inversion of control: write new INPUT sources, not new per-object interaction code** — MRTK3: philosophically the attachTransform IS the interaction logic; always prefer writing a new interactor with new attachTransform logic over rewriting interactables. Result: every interactable automatically supports every input modality. For us: a card should not contain modality-specific drag math; it should expose a single grabbed-pose input that a drag gesture, a future hand-tracking grab, or a programmatic move all feed. ManipulationComponent already does this on visionOS 26, so adopting it is the highest-leverage move. _(src: MRTK3 Interactor architecture (AttachTransform and Inversion of Control))_
- **Unidirectional data flow (MVU/MVVM) for the SCENE, with a single source of truth and messages as the only mutation path** — MVU/Elm: Model -> View renders -> user input -> Update produces new Model — unidirectional flow makes state predictable and reduces bugs. For a 3D scene: SpatialEntity rows (data) -> layout/derive -> entities (view) -> gestures emit intents -> view-model reduces intents to data writes -> persistence. The key is that gestures emit INTENTS (.moved(id,pose), .dismissed(id), .connected(a,b)) rather than directly mutating both entity and stored position in two places. Our Observable view-model is the right vehicle; it needs a disciplined intent/reduce boundary so transform writes have one path and one owner. _(src: Model-View-Update / Elm Architecture writeups; Andre Staltz Unidirectional UI Architectures)_

### Anti-patterns
- **Re-asserting every entity's stored transform on every reactive update/reconcile pass (immediate-mode in a retained-mode engine)** — Direct cause of drag snap-back. Violates transform-ownership (data model overwrites the live gesture) and the don't-mutate-observed-state-in-update rule (feedback loop). Retained-mode engines expect you to mutate only what changed; re-asserting everything each pass also destroys frame-coherent state (selection, in-flight animation) and scales O(n) per change. Replace with: create once, mutate changed components, and let ManipulationComponent/Systems own transforms.
- **Disambiguating interaction by parsing entity names (ctl::role::ownerId, edge::) inside global .targetedToAnyEntity gestures** — String-typed dispatch is unscalable and error-prone; every new control kind grows a parser branch with no compiler help. Mature stacks use typed Interactor/Interactable roles or per-entity GestureComponents (visionOS 26) so identity and behavior are intrinsic. It also couples all interaction into two megafunctions, the opposite of the single-responsibility seams MRTK3/XRI rely on.
- **Deleting and re-adding ALL edges + ports from scratch every update pass** — Pure immediate-mode rebuild — discards retained-mode benefits (diffing, culling, stable identity), thrashes the entity graph, and scales badly as the graph grows. Edges are derived data with stable endpoints; model them as entities owned by an EdgeSystem that updates only geometry whose endpoint poses changed (EntityQuery on EdgeEndpointComponent).
- **Baking visual feedback (highlight, jitter, color, press) into the same code that computes interaction state** — The MRTK2 mistake MRTK3 explicitly retired (interactables are pure state and interaction, render no visuals). Coupling makes visuals un-reskinnable, un-testable, and forces interaction edits whenever design changes. Split into a state layer (observed) and a visual-driver layer (functions of state). Directly relevant to the requested scalable design/component system.
- **Relying on a SwiftUI .hoverEffect{ scaleEffect } inside a ViewAttachmentComponent rootView for a child control while a parent CollisionComponent encloses the child** — Bug #2's root: the parent's hit region overlaps the child (only +0.006 z inside a depth max(z,0.03) box), so eye-tracking/hover resolves to the parent and the child's SwiftUI hover never fires. Native HoverEffectComponent on the child entity (with its own non-overlapping CollisionComponent + InputTargetComponent) is the reliable mechanism; SwiftUI-internal hover effects don't substitute for RealityKit input targeting.
- **Using a view @State flag (draggingID) checked inside the RealityView update closure to skip the dragged entity** — State mutated inside a RealityKit gesture .onChanged is not reliably visible to the update closure before the next reconcile pass, so the guard races and fails (already observed). The problem is architectural, not a missing flag: remove transform-writing from update entirely and let the gesture/ManipulationComponent own the transform, so there is nothing to guard.
- **Putting continuous/per-frame behavior in the RealityView update closure (treating it like a game tick)** — update fires on observed-state change, not per frame, so continuous logic there is both incorrect and a feedback-loop hazard. Guidance: for continuous behaviors implement a custom System, not the update closure. Systems run once per frame over EntityQuery matches with proper isolation.

### Recommendations for our app
- Adopt a 5-layer pipeline with explicit ownership at each seam: (1) Data: SpatialEntity rows from tRPC; (2) Derive/Layout: the existing pure nonisolated SpatialLayoutEngine produces target poses for AUTO-FLOW entities only; (3) Interaction: per-entity roles + ManipulationComponent/GestureComponent emit INTENTS; (4) Placement/Scene: a small set of RealityKit Systems write transforms, but a transform is written by exactly one owner at a time (manipulation > animation > layout > data); (5) Render: SwiftUI attachments as pure observers of interaction state. This is the MRTK3/XRI/Bevy consensus applied to RealityKit.
- Fix bug #1 at the root by deleting position-writing from reconcile/update and adopting ManipulationComponent (visionOS 2.6+) for card movement. Clamp to VolumeMetrics in a ManipulationEvents.DidUpdateTransform subscription; persist on WillEnd. Keep releaseBehavior=.stay. This makes grab-and-drag the near-trivial thing the developer expects.
- Fix bug #2 at the root via hierarchical collision shaping: give the x control its own CollisionComponent that protrudes clearly beyond the card's collision box (or carve the parent box), its own InputTargetComponent, and a native HoverEffectComponent (not a SwiftUI-internal hover effect). Make jitter-on-highlight a transform animation driven from an observed HoverStateComponent via Entity.animate or a HighlightSystem.
- Replace the monolithic reconcile(content:attachments:) with: a one-time entity factory (create card/control/edge entities once), an EntitySyncSystem that adds/removes entities when SpatialEntity rows change (diff, don't rebuild), an EdgeSystem that updates only moved edges (EntityQuery on EdgeEndpointComponent), and a LayoutApplySystem that writes auto-flow poses only to entities no other authority owns. Register custom Components/Systems once at setup.
- Introduce a typed interaction/visual seam instead of name-parsing: define entity roles as components (TaskNodeRole, DismissControlRole, EdgeRole, etc.), attach per-entity GestureComponents (visionOS 26) so taps route by identity, and build an MRTK3-style visual driver where each card's glass/opacity/highlight is a pure function of an observed InteractionState. New entity kinds declare state->style and never touch dispatch code.
- Enforce unidirectional flow in the @Observable view-model: gestures emit intents (.moved, .dismissed, .connected, .linked, .toggledVolume); a single reduce path turns intents into data writes and persistence. No code path may write both an entity transform and the stored position in two places — that duplication is what made the snap-back possible.
- Keep the layout engine pure and nonisolated (Foundation/simd only) and call it FROM a MainActor System; do not let Systems' default MainActor isolation leak into the engine. This preserves the unit-tested deterministic layout while satisfying SWIFT_DEFAULT_ACTOR_ISOLATION=MainActor.
- Use Entity.animate/content.animate with interactiveSpring/easeInOut for all transitions (snap-to-slot, dismiss-to-tray, volume expand/collapse), and treat each animation as a temporary transform-ownership claim that suppresses layout/data writes for that entity until it completes — preventing a new class of snap-back during animations.
- Adopt the four-state vocabulary (Hover, Select, Focus, Activate) for entity interaction even though RealityKit has no Manager — a tiny mediator/reducer suffices. This gives a stable, testable model for highlight, selection, connect/link modes, and future near/far hand-tracking interactors without per-modality branching, mirroring XRI's attachTransform inversion-of-control.
- Validate everything with xcodebuild -scheme TaskPlannerVision -destination 'generic/platform=visionOS Simulator' (trust it over SourceKit). Gate ManipulationComponent behind a fallback path due to the known intermittent translation-stops-working bug, and unit-test the layout engine and the intent->reduce logic with Swift tests where the VM logic is nonisolated/extractable.

## Scalable Spatial Design System and Reusable Component Library for visionOS 3D UI (Task-Planner Cards, Panels, Controls, Edges)

The current scene's two root problems are architecture gaps, not gesture bugs. (1) It re-asserts every entity transform inside the RealityView update closure each pass, fighting live drags and SwiftUI Observation. WWDC25 is explicit that the update closure is an extension of the view body and must not own continuous transforms; transforms belong to gestures, a custom RealityKit System, or Entity.animate(), and on visionOS 26 entities are Observable so SwiftUI can read state without re-pushing it. (2) The custom x control does not highlight because its scale comes from a SwiftUI .hoverEffect inside a ViewAttachment whose parent card has an enclosing CollisionComponent only +0.006z away; hierarchical hover on a RealityKit entity is the job of HoverEffectComponent (spotlight/highlight/shader, visionOS 2+) plus a non-overlapping child collision shape, not SwiftUI content-hover inside an occluded attachment. The single biggest lever is visionOS 26 ManipulationComponent (and SwiftUI .manipulable), giving system-owned grab/translate/rotate/scale with WillBegin/DidUpdateTransform/WillEnd events for persistence, making grab-and-drag near-trivial. For the design system, Apple HIG plus a Unity MRTK3 style token/theme model maps cleanly to Swift: one nonisolated token enum (meters spacing/elevation, corner radius, glass material levels, tint, motion durations/easings, hover styles) feeding composable SwiftUI card primitives, with a per-entity state machine (rest/hover/selected/dragging/disabled) expressed as a RealityKit Component driven by a System rather than parsed from entity names. Edges/ports become a derived, diffed System, not a per-pass teardown. Tokens + component state machine + ManipulationComponent + HoverEffectComponent + Systems-driven reconcile + Observation is the production-grade foundation that scales.

### Core principles
- **SwiftUI renders content; RealityKit owns transforms; the RealityView update closure owns neither continuously** — WWDC25 session 274 states the update closure is treated as an extension of the view body, so mutating observed state or re-asserting transforms there causes loops and fights gestures. Safe places to mutate an entity transform: the gesture closure (live drag), a custom System.update(context:), the one-shot make closure, or Entity.animate(). Root fix for drag snap-back: stop re-writing card.position from the stale view-model value every pass. _(src: WWDC25 Better together: SwiftUI and RealityKit (session 274))_
- **On visionOS 26 entities are Observable; read entity state into SwiftUI, do not push SwiftUI state onto entities every frame** — entity.observable exposes position/scale/rotation/children/components to Swift Observation, so SwiftUI re-renders when an entity changes. The data flow inverts: the gesture/System mutates the entity transform (single source of truth for placement), persistence reads it on release, SwiftUI chrome observes it. Removes the stale-position-during-drag bug class. _(src: Apple docs Entity.Observable; WWDC25 session 274)_
- **Continuous/derived scene behavior belongs in a RealityKit System, not the update closure** — Primary and community sources converge: do not use the RealityView update closure for continuous behavior; use a custom System conforming to RealityKit.System, running one update(context: SceneUpdateContext) per frame over EntityQuery+QueryPredicate matches, reading/writing components directly. Correct home for edge/port redraw (diff vs teardown-rebuild) and the per-entity visual state machine. _(src: WWDC23 session 10080; swiftorbit.io ECS articles; tomkrikorian visionOS 26 AGENTS.md)_
- **visionOS 26 ManipulationComponent makes grab/drag/rotate/scale a system-owned, single-line capability** — ManipulationComponent.configureEntity(entity, hoverEffect:allowedInputTypes:collisionShapes:) adds Manipulation + Collision + InputTarget + HoverEffect. During manipulation the component owns the transform (no ownership conflict); persist via ManipulationEvents.WillEnd/WillRelease. releaseBehavior=.stay keeps the final pose vs .reset. It auto-handles user re-orientation mid-drag, the exact coordinate-space complexity the manual DragGesture+convert code wrestles with. SwiftUI equivalent: .manipulable(operations:inertia:). _(src: WWDC25 session 274; stepinto.vision Deep Dive into Manipulation on visionOS)_
- **Hover/highlight/focus is a RealityKit component concern; parent collision must not occlude child controls** — HoverEffectComponent (visionOS 1+) supports .spotlight(SpotlightHoverEffectStyle(color:strength:)), .highlight(HighlightHoverEffectStyle(color:strength:)) uniform mesh tint (good for playful jitter/glow on select), and .shader(...) driven by a ShaderGraph HoverState node whose intensity (0 to 1 on gaze) animates emissive/scale (highlight and shader: visionOS 2+). The x control fails because its parent card CollisionComponent (depth max(z,0.03)) encloses the child only +0.006z away, so gaze/taps resolve to the parent. Fix: own forward-offset non-overlapping collision + own HoverEffectComponent. _(src: WWDC24 sessions 10103 + 10152; HoverEffectComponent docs)_
- **Centralize design decisions as tokens (MRTK3/Material-3 pattern, adapted to meters)** — MRTK3 data-binding/theming themes whole hierarchies from one profile; Material 3 formalizes tokens as single source of truth. Apple HIG seeds spatial token values: ~60pt min interactive target, 44pt standard / 28pt mini buttons, 16pt min button spacing, 4pt padding to avoid hover-effect overlap, ornament 20pt window overlap, concentric corner radius = innerRadius + padding, glass primary material with vibrancy tiers and white text by default. In Swift: one nonisolated enum SpatialToken (Foundation/simd only), testable and shared with the layout engine. _(src: Apple HIG / WWDC23 session 10076; MRTK3 Data Binding and Theming docs; Material Design 3 tokens)_
- **Composition over inheritance; every interactive entity carries Collision + InputTarget + Hover; theme by variant not name-parsing** — visionOS 26 AGENTS.md: prefer composition over inheritance, create custom components implementing Component and Codable; all draggable entities must have CollisionComponent and InputTargetComponent; always add a hover effect to custom interactive elements. Gesture disambiguation should read typed components (a SpatialRoleComponent with kind and ownerId), not parse ctl::role::id strings; this scales and is testable. _(src: tomkrikorian visionOS 26 and Swift 6.2 AGENTS.md; WWDC25 session 274 GestureComponent)_
- **Animate transforms declaratively with Entity.animate()/move(to:); do not snap** — visionOS 26 Entity.animate(.bouncy) { entity.position = ... } implicitly animates Transform/Audio volume/Model and Light color via a SwiftUI animation curve; pre-26 fallback is entity.move(to:relativeTo:duration:timingFunction:) or FromToByAnimation + AnimationResource.generate + playAnimation. Use for return-to-tray, expand/collapse pop-out, and selection jitter, with durations/easings tokenized. _(src: WWDC25 session 274; createwithswift.com Play an animation in RealityKit)_

### Key APIs
- **`ManipulationComponent / ManipulationComponent.configureEntity(_:hoverEffect:allowedInputTypes:collisionShapes:)`** _(min visionOS 26)_ — System-owned grab/translate/rotate/scale with handoff, inertia, audio; replaces manual DragGesture + value.convert coordinate math.
  - When: Make cards, notes, workflow volumes draggable. Set releaseBehavior=.stay; persist transform in ManipulationEvents.WillEnd. Disable scaling/rotation per-kind via dynamics for translate-only cards.
  - ⚠️ During manipulation the COMPONENT owns the transform; do not also write position from the view model (this is the snap-back cause). Subscribe to DidUpdateTransform only to clamp to VolumeMetrics. Reported sim quirk: translation can stop app-wide until device restart.
- **`.manipulable(operations:inertia:) (SwiftUI)`** _(min visionOS 26)_ — SwiftUI-level manipulation for Model3D/attachment-backed views, mirroring ManipulationComponent for the content-renders-in-SwiftUI paradigm.
  - When: When a card is presented primarily as a SwiftUI view and you want manipulation without entity-level component wiring.
  - ⚠️ Operations: .translation, .primaryRotation, .secondaryRotation, .scaling. Prefer entity-level ManipulationComponent when you need WillEnd events for tRPC persistence and clamping.
- **`HoverEffectComponent(.spotlight/.highlight/.shader)`** _(min visionOS 1 (component); .highlight and .shader styles: visionOS 2)_ — Gaze-driven visual feedback owned by RealityKit per entity: spotlight (default), highlight (uniform mesh tint, color/strength), shader (ShaderGraph HoverState intensity).
  - When: Every interactive entity: cards (subtle spotlight), x and port controls (highlight/scale), selected state (highlight tint via tokens). Drives playful jitter via shader intensity or animate-on-hover.
  - ⚠️ SwiftUI .hoverEffect inside a ViewAttachment will not visibly scale when the parent CollisionComponent occludes the child; that is bug #2. Use entity-level HoverEffectComponent and give the child its own non-overlapping collision shape offset enough in +z to clear the parent box.
- **`RealityKit System + EntityQuery (System protocol, SceneUpdateContext)`** _(min visionOS 1 (System); register via registerSystem())_ — Per-frame query-driven processing for all entities of interest in one update call; correct home for derived edges/ports and the visual state machine.
  - When: Replace drawEdges() teardown-rebuild with an EdgeSystem that diffs/repositions existing edge entities against live positions; add a NodeStateSystem mapping a StateComponent to hover/selection visuals.
  - ⚠️ Apple guidance: do not do continuous work in the RealityView update closure; Systems run off the render loop and do not entangle with SwiftUI body evaluation.
- **`Entity.observable (Entity.Observable)`** _(min visionOS 26)_ — Observation-framework view of an entity transform/components so SwiftUI re-renders on change without pushing state back into the entity.
  - When: Any SwiftUI chrome that must reflect live entity position (minimap, selection HUD, persistence trigger) reads entity.observable instead of a mirrored @Observable position field.
  - ⚠️ Lets you delete position-mirroring in the view model that causes stale-position fights. Pre-26 you must mirror manually but still must not re-assert during drag.
- **`Entity.animate(_:body:) / content.animate { } / Entity.move(to:relativeTo:duration:timingFunction:)`** _(min Entity.animate / content.animate: visionOS 26. move(to:timingFunction:): visionOS 1.)_ — Declarative implicit animation of entity Transform (and audio/model/light color) using SwiftUI curves; move(to:) is the pre-26 fallback with explicit easing.
  - When: Return-to-tray, expand/collapse pop-out, selection jitter, snap-to-clamp. Centralize curve/duration in the token file.
  - ⚠️ move(to:) timing functions: .linear/.easeIn/.easeOut/.easeInOut (plus cubicBezier). Returns AnimationPlaybackController. Do not animate inside the update closure unless wrapped in content.animate within a state-gated branch.
- **`ViewAttachmentComponent / PresentationComponent / GestureComponent (visionOS 26)`** _(min visionOS 26 (component-form attachments, PresentationComponent, GestureComponent); ViewAttachment builder: visionOS 1)_ — ViewAttachmentComponent embeds a SwiftUI view on an entity (no builder); PresentationComponent presents popovers/sheets from an entity; GestureComponent attaches a SwiftUI gesture to a specific entity.
  - When: Cards/notes/panels as attachments; per-entity tap (edit) and dismiss as GestureComponent, removing the global tap gesture + name-parsing. Edit sheets via PresentationComponent anchored to the card.
  - ⚠️ GestureComponent still requires InputTargetComponent + CollisionComponent. Per-entity gestures eliminate the brittle prefix-parsing dispatcher.
- **`Glass materials: .glassBackgroundEffect() and Material levels (.ultraThin/.thin/.regular/.thick)`** _(min Materials: SwiftUI 1; .glassBackgroundEffect(): visionOS 1)_ — Adaptive semi-transparent glass surfaces with vibrancy for elegant cards/panels; material level = an elevation token.
  - When: Token material level per tier: panels = regular, cards = regular/thin, transient controls = thin/ultraThin. Use .glassBackgroundEffect() for spatially-aware glass on attachments.
  - ⚠️ HIG: avoid stacking lighter materials on lighter; prefer white text/symbols and vibrancy tiers (primary/secondary/tertiary) over custom colors for legibility on glass.

### Anti-patterns
- **Re-asserting card.position from the view-model in the RealityView update closure every pass** — Root cause of drag snap-back. The update closure is an extension of the view body; the stored VM position is stale until release, so it yanks the card back. Transform ownership must be the gesture/ManipulationComponent during interaction; persistence reads the entity on release.
- **Using @State draggingID checked inside reconcile to skip the dragged card** — @State mutated in a RealityKit gesture .onChanged is not reliably visible to the update closure before the next pass (already observed). The fix is architectural: do not have an update-pass that owns transforms at all.
- **SwiftUI .hoverEffect { effect.scaleEffect } inside a ViewAttachment for a control under a larger parent collision box** — Root cause of the x not highlighting. The parent card CollisionComponent (depth max(z,0.03)) encloses the child only +0.006z away, so gaze/hover resolve to the parent and the child SwiftUI hover never activates. Use entity-level HoverEffectComponent + a non-overlapping child collision shape.
- **Tearing down and re-adding ALL edges + ports from scratch every reconcile pass** — Wasteful and causes flicker/GC churn as the graph grows; prevents animating edges. Replace with an EdgeSystem that diffs the derived edge set and only creates/removes/repositions changed line/port entities against live positions.
- **Encoding control/edge identity in entity names (ctl::role::owner, edge::) and parsing them in the gesture handler** — Stringly-typed dispatch does not scale and is not testable. Replace with typed components (SpatialRoleComponent/StateComponent) and per-entity GestureComponent so role/owner are first-class data.
- **Scattering meters/colors/durations/corner radii as magic numbers across views and the reconcile (0.006, 0.03, 0.07, 16, opacity 0.9)** — No single source of truth means inconsistency as components multiply. MRTK3/Material-3 practice and the existing VolumeMetrics precedent argue for one nonisolated token file feeding every component.
- **Generating CollisionComponent from card.visualBounds(recursive:) every pass** — Documented in project memory as a recursive visualBounds feedback loop (bounds grow as children are added, inflating the box each pass). Compute collision from token-defined card dimensions, not measured recursive bounds.
- **Hand-building primitive-mesh / 3D-text UI for cards and controls** — Already tried and rejected as unusable; violates the locked SwiftUI-renders-content paradigm. Keep all content as SwiftUI attachments; RealityKit only places, animates, and routes gestures.

### Recommendations for our app
- Create one token source of truth: a nonisolated enum SpatialTokens (Foundation/simd only, same isolation rule as SpatialLayoutEngine) holding meters spacing, z-elevation layers (card, control, edge, selection-lift), corner radii, material levels per tier, tint roles, motion durations/easings, and hover styles. Every card view, the collision sizing, and the layout engine read from it. Directly testable and ends magic numbers.
- Adopt visionOS 26 ManipulationComponent for all movable kinds via ManipulationComponent.configureEntity(card, hoverEffect:.spotlight(...), collisionShapes:[tokenSizedBox]). Persist in ManipulationEvents.WillEnd by reading the entity transform and calling viewModel.endDrag; clamp to VolumeMetrics in DidUpdateTransform. Delete the manual DragGesture + value.convert + dragStart/draggingID logic. This single change fixes bug #1 at the root and makes grab-and-drag trivial.
- Stop owning transforms in the update closure. Move scene reconcile to: (a) a one-shot make closure for initial add, (b) an entity-diffing add/remove step gated on the entity SET actually changing (not every observed change), and (c) RealityKit Systems for continuous/derived work. Read live positions via entity.observable instead of mirroring position in the view model; the VM keeps canonical data + persistence, not per-frame placement.
- Build a reusable SpatialCard component abstraction with an explicit state machine: a NodeStateComponent with state rest/hover/selected/dragging/disabled plus a NodeStateSystem that maps state to visuals (HoverEffectComponent highlight tint from tokens, a small Entity.animate jitter/scale on hover, selection lift in +z). Cards, panels, controls, and workflow volumes are variants of the same primitive differing only by tokens/content, mirroring MRTK3 theming-by-profile.
- Fix the x control as its own first-class entity: give it a token-sized collision box offset far enough in +z to clear the parent card box (and/or shrink the parent box depth), attach HoverEffectComponent(.highlight) + GestureComponent(TapGesture) so it gets its own gaze/tap, and render its visual as a SwiftUI attachment. This fixes bug #2 and removes the need for hierarchical collision filtering hacks. Apply the 4pt-padding HIG rule (in meters) so control and card hover regions do not overlap.
- Replace name-parsing gesture dispatch with typed components: SpatialRoleComponent with kind and ownerId on controls/edges and per-entity GestureComponent for tap/edit/dismiss. The global TapGesture + controlRole(name:) string parsing goes away, making the interaction layer testable and scalable to new control types.
- Convert edges/ports to an EdgeSystem that diffs the derived edge set (from TaskStep.dependsOn + links) against existing edge entities and only mutates deltas, repositioning via entity.observable positions. Token edge thickness/port radius/colors. Removes per-pass teardown and lets edges animate as nodes move.
- Centralize glass/material as elevation tokens: panels = .regular glass, cards = .regular/.thin, transient controls = .thin/.ultraThin, with vibrancy tiers and white symbols per HIG. Animate state transitions (return-to-tray, expand/collapse pop-out, selection) with Entity.animate using token durations/easings (pre-26 fallback: move(to:timingFunction:)). Verify only via xcodebuild -scheme TaskPlannerVision (SourceKit false errors), and gate visionOS-26-only APIs since the target is 26.2.
