## Proposals (full)

### RealityKit-ECS-native. The RealityKit Scene is the single source of RUNTIME truth; the @Observable view model and tRPC store are the source of CANONICAL/PERSISTED truth, and they only SEED and RENDER. Every per-entity runtime fact (who owns the transform right now, where layout wants it, hover/selection state, drag deltas, jitter phase, edge endpoints) lives in a custom RealityKit Component on the entity. Every behavior over time (translating during a drag, gliding to a layout slot, scaling on hover, jittering, repositioning edges) lives in a custom RealityKit System ticked off the render loop. The RealityView `update:` closure shrinks to a pure, identity-only data->entity SYNC that is forbidden by construction from writing any transform a System owns. SwiftUI keeps its locked job — it renders all card CONTENT via ViewAttachmentComponent — and the pure, nonisolated SpatialLayoutEngine keeps its job: it produces TARGET poses that a MainActor LayoutSystem applies, never touching entities itself. This inverts today's design where the reconcile pass is the de-facto transform owner fighting both the gesture and SwiftUI Observation. The mental model: data flows DOWN into components once (seed/sync); behavior flows FORWARD each frame in Systems (own/animate); intents flow UP from gestures into components, and only commit to the store on release. One authority owns a transform at any instant, and that authority is always a System reading a component — never the update closure.
> Make components the runtime state and Systems the behavior so exactly one System owns each entity's transform at any instant; the reconcile pass becomes an identity-only data sync that never writes a System-owned transform, which kills the drag snap-back, and a HoverSystem driven by collision-group-isolated child colliders + a dedicated HoverEffectComponent fixes the x-highlight.

- **Transform ownership:** A single TransformAuthorityComponent enum {.layout, .gesture, .animating} is the ownership token on every movable entity, and exactly one System may write a given entity's transform per instant based on it. (1) Default authority is .layout. LayoutSystem queries entities with authority==.layout and a LayoutTargetComponent, and for those whose target changed glides position via Entity.animate(.easeOut(SpatialMotion.reflow)); it never touches .gesture or .animating entities. (2) On grab, ManipulationComponent owns the transform natively for its lifetime; ManipulationEvents.WillBegin sets authority=.gesture (and disarms LayoutSystem). RealityKit itself drives the transform during manipulation, so there is no app-side write to race. For the fallback path, a DragGesture writes DragComponent{startTransform, active=true} and DragSystem (authority .gesture) applies the clamped delta — still a single writer. (3) On release, ManipulationEvents.WillEnd reads the final entity.transform, sets authority=.animating, hands (id, pose) UP to the VM to persist (the VM writes the SpatialEntity row but NEVER pushes it back onto the entity), and after a short settle Entity.animate completion sets authority back to .layout. Crucially, SceneSync (the update closure) is structurally forbidden from writing ANY transform: it only seeds a transform once at creation (when card.parent==nil) and otherwise writes only to LayoutTargetComponent (a request, not a position). Because the layout target and the live transform are separate values, the canonical store can change freely mid-drag without yanking the card. This is the textbook target-vs-presented split: the store/layout own the TARGET; a System owns the PRESENTED transform; the update closure owns NEITHER. There is therefore no instant at which two authorities write the same transform.
- **Reconciliation:** SceneSync replaces the 543-line reconcile with an identity-only diff that runs in update: and is pure of transform writes. Steps each pass: (1) Build liveIDs from the VM's renderedEntities snapshot. (2) Removed: entities present in the scene (matched by SpatialRoleComponent.entityId, not name) but absent from liveIDs get LifecycleComponent.phase=.exiting; LifecycleSystem animates them out and removes — never an instant content.remove. (3) Added: ids in liveIDs with no scene entity are built by SpatialEntityFactory, seeded with all components (authority=.layout, role, lifecycle=.entering, hover/jitter/selection), added once, and given an initial transform ONLY here (the single legal transform write in the sync path). (4) Changed identity/content: if a card's underlying Task/step/type content changed, refresh its ViewAttachmentComponent rootView (content, not transform); if its canonical/computed slot changed, write the new value to LayoutTargetComponent (a request the LayoutSystem fulfils only when authority==.layout). (5) Selection/mode: write SelectionComponent from VM mode state so the HoverSystem/visual driver react. Edges are NOT rebuilt here — EdgeComponent-bearing entities are created once by a diff over the derived edge set (keyed by sourceId+targetId+semantic) and the EdgeSystem follows live endpoints every frame. Because the sync only ever writes components and never a transform owned by a System, and because change detection is by stable id + dirty target rather than re-stamping everything, an unrelated @Observable change can fire update: harmlessly: nothing moves unless a target actually changed. This also satisfies the WWDC25 274 rule — no observed entity transform is mutated in the update closure.
- **Interaction:** Routing moves from one global gesture pair + string parsing to typed, per-entity dispatch. Drag: ManipulationComponent.configureEntity(card, hoverEffect:.highlight, allowedInputTypes:.indirect, collisionShapes:[tokenSizedBox]) with releaseBehavior=.stay and dynamics set translate-only for cards (free 6DoF for notes/volumes). RealityKit owns the move; subscribe once to ManipulationEvents.WillBegin (authority=.gesture), DidUpdateTransform (clamp to VolumeMetrics by constraining the value), and WillEnd (authority=.animating, read final transform, viewModel.endDrag to persist + mark manuallyMoved). Tap: each card carries a GestureComponent(TapGesture()) whose handler reads the entity's SpatialRoleComponent to dispatch (taskNode/stepNode -> edit; workflowVolume -> toggle; note -> note edit) — no name parsing. The × control carries its OWN GestureComponent(TapGesture()) that reads DismissControlComponent.ownerId, so dismiss is intrinsic, not routed through the card. Merge/Link modes become explicit VM interaction-mode state (an enum + sourceId), reduced from intents; entering a mode flips card taps to 'pick' behavior and writes SelectionComponent.pickedMerge/pickedLink to candidates so the visual driver highlights them; entering a mode also sets ManipulationComponent.isEnabled=false on cards so a pick can't be misread as a drag. CoordinateSpace3D / GeometryProxy3D.coordinateSpace3D() replaces the ad-hoc value.convert(.local,.scene) math in any remaining manual path. Edges/ports carry no InputTargetComponent and sit in a CollisionGroup excluded from all input masks, so they are structurally non-interactive.
- **Hover/highlight:** Hover/highlight/selection are three distinct, decoupled concerns. (1) System gaze hover: every interactive entity gets a real RealityKit HoverEffectComponent(.highlight(HighlightHoverEffectStyle(color: tokenTint, strength:))) — NOT a SwiftUI .hoverEffect buried in an attachment. The card and the × badge each get their OWN HoverEffectComponent with DISTINCT HoverEffectComponent.GroupID so the badge activates independently of the card (hover does not cascade to children, and a shared group would couple them). (2) Playful pop + jitter: a HoverStateComponent is the app-readable hover flag; because the app cannot read system gaze directly, the HoverStateComponent is toggled from the SwiftUI .onHover inside the attachment OR from a hover-state shader path — the HoverSystem then arms JitterComponent and runs a low-amplitude Entity.animate(.spring) scale-pop. JitterSystem integrates context.deltaTime into phase and offsets a CLEAN base position (so jitter never drifts the persisted transform), gated by accessibilityReduceMotion (crossfade fallback). (3) App selection (merge/link picked, edit target): SelectionComponent drives the visual driver's tokenized border/lift — kept entirely separate from hover, never derived from it. The × badge's highlight now works because (a) it has its own non-enclosed collider in its own input/hover group and (b) its highlight is a RealityKit HoverEffectComponent, not a 2D SwiftUI scaleEffect that the parent swallowed.
- **Collision:** Hierarchical collision is designed, not accidental, on three axes. (1) Geometry: the card collider stops being inflated to depth max(z,0.03). It is a THIN token-defined box (~0.006 m) sized from SpatialTokens card dimensions (computed once at creation, cached — no per-pass visualBounds regeneration, killing the recursive-visualBounds drift trap). The × badge is offset a REAL gap (~0.025-0.03 m, not 0.006 m) clearly in front of the card so the parent box cannot enclose it; per Apple DTS an enclosing parent with InputTargetComponent is always hit first, which is exactly why the badge was dead. (2) CollisionGroup/CollisionFilter: define .node, .control, .edge groups. Cards = .node, controls = .control, edges/ports = .edge with an empty input mask (and no InputTargetComponent). Manipulation/drag targets .node; control taps target .control. Routing by mask/role replaces isControlOrEdge name checks. (3) Input precedence: cards set InputTargetComponent(allowedInputTypes:.indirect); the control gets its own InputTargetComponent so it is independently hittable. If any control must overlap its parent, the wrapper's InputTargetComponent.isEnabled=false plus a dedicated child body target makes siblings reachable. Collision shapes derive from tokens, so the visual and the hit region can never disagree, and the valid/invalid visualBounds branch disappears. Where precise frontmost resolution is needed (connect/link targeting) use scene.raycast(..., mask:.node).
- **Animation:** All motion is target-vs-presented with one animation authority. Entity.animate(_:) (in gesture/event closures and Systems) and content.animate { } drive transforms with SwiftUI curves; pre-26 fallback is entity.move(to:relativeTo:duration:timingFunction:). LayoutSystem glides .layout entities to LayoutTargetComponent via Entity.animate(.easeOut(SpatialMotion.reflow)) so relayout, return-to-tray, and column repack all become smooth flights rather than teleports. LifecycleSystem animates entrance (scale 0.85->1, opacity 0->1 via the attachment) on .entering and exit (shrink/fade, or fly-to-tray for returnToTray) on .exiting before removal. Drag release runs a short settle spring then completion flips authority back to .layout (treating the animation as a temporary ownership claim so nothing fights it). Workflow expand spawns step entities AT the volume position then Entity.animate's them with a staggered spring to SpatialLayoutEngine.stepGraph slots; collapse reverses. Hover pop and jitter are Entity.animate(.spring) + a deltaTime System as above. Every duration/easing/spring/jitter-amplitude is a SpatialMotion token, and every animation reads accessibilityReduceMotion to substitute a crossfade. Edges animate their opacity in/out on add/remove and follow nodes continuously via EdgeSystem — never recreated, so in-flight animation/hover state survives.
- **Material/design system:** A nonisolated SpatialTokens enum is the single source of truth (mirroring VolumeMetrics' precedent), holding: per-kind card dimensions (in BOTH points for the SwiftUI .frame and meters for collision, bridged so they can't disagree), corner radii, padding, glass material level per elevation tier (panel=section, node/note=interactive, volume=recede), tint roles, border weights per SelectionState, semantic edge colors (one token bridged to both SwiftUI Color and UIColor), and SpatialMotion (durations/easings/spring params/jitter amplitude+frequency). All four card views collapse into one SpatialCard primitive that takes kind+tint+interactionState and applies tokenized glass via .glassBackgroundEffect() (used consistently, replacing the .regularMaterial/.thinMaterial/color-under-material mix) with vibrancy tiers (.primary/.secondary/.tertiary) and white text — no colored text, no solid color washes. Edge/port meshes use UnlitMaterial(applyPostProcessToneMap:false) so their token colors match the SwiftUI palette exactly (tone mapping is otherwise on by default and shifts UI color). Note color, dependency-edge teal, and link orange become named semantic tokens, eliminating the literal .yellow/.systemTeal/.systemOrange. Accessibility: the glass token swaps to an opaque/high-contrast surface under accessibilityReduceTransparency/increaseContrast, and selection adds a non-color cue (lift + thicker tokenized border) so state is never color-only — important for the ADHD audience.
- **Fixes drag bug:** Root cause today: reconcile runs in update: on every @Observable change and re-asserts card.position from the VM's STORED position (stale until drag-release), guarded only by a SwiftUI @State draggingID that the update closure can't reliably see mid-gesture — so the card snaps to the stale slot. The fix removes the cause entirely on three levels. (1) The update closure (now SceneSync) is structurally forbidden from writing any transform a System owns; it writes only LayoutTargetComponent (a request) — there is no card.position re-assert line to fight the drag. (2) Transform ownership is an explicit TransformAuthorityComponent: during a drag authority==.gesture and LayoutSystem skips the entity; on release it goes .animating then .layout. Ownership lives in a Component the Systems read, not in view @State the update closure can't observe in time — so the failed draggingID race is gone. (3) Adopting ManipulationComponent (releaseBehavior=.stay, persist in WillEnd) makes RealityKit itself own the transform during the drag, so there is literally no app-side writer to race, and .stay prevents the framework's own .reset snap-back. The target (store/layout) and the presented transform (System) are separate values, so the store changing mid-drag can never yank the card. Net: no two authorities ever write the same transform; the snap-back cannot occur by construction.
- **Fixes hover bug:** Root cause today: the × is a child only +0.006 m in front of a parent card collider inflated to depth max(z,0.03) (~3 cm), so the parent ENCLOSES the badge and, per Apple DTS, the parent (which has InputTargetComponent) is hit first — the badge never receives gaze/hover/tap; and its highlight is a SwiftUI .hoverEffect{scaleEffect} inside the attachment, which scales 2D content and isn't a reliable RealityKit hover for a placed entity. The fix attacks all three. (1) Geometry: the card collider becomes a THIN token box (~0.006 m, not 0.03) and the badge is pushed a real ~0.025-0.03 m gap forward so the parent box no longer contains it — the badge becomes the frontmost collider and is independently hit-testable. (2) Isolation: the badge gets its own CollisionComponent in CollisionGroup .control plus its own InputTargetComponent, and a HoverEffectComponent with a DISTINCT HoverEffectComponent.GroupID so it activates independently of the card (hover doesn't cascade to children). (3) Visible highlight: highlight is now a real RealityKit HoverEffectComponent(.highlight) on the badge entity (plus an optional HoverSystem-driven Entity.animate scale pop), not a SwiftUI scaleEffect; and the dismiss tap is its own per-entity GestureComponent reading DismissControlComponent — no name routing through the swallowing parent. With the badge frontmost, isolated, and RealityKit-driven, it both highlights and dismisses reliably.
- **Scalability:** Adding a new entity kind = add a case to SpatialRole + SpatialTokens dimensions + a SpatialCard content branch; the factory wires collision/input/hover/manipulation/components uniformly and every System already operates on it via its components. Adding a new behavior (e.g. magnetic snapping, breathing-on-selected, auto-arrange) = add one Component + one System keyed by EntityQuery; no edits to the sync, gestures, or existing Systems. Adding a new control (port handle, context menu) = the factory's Control recipe (group .control, own collider/hover/GroupID, own GestureComponent) — no central parser to grow. Adding a new edge semantic = a token color + an EdgeComponent semantic case; EdgeSystem already follows it. Replacing tap routing or input modality = swap the per-entity GestureComponent / ManipulationComponent config; nothing else changes because dispatch reads typed components, not names. Performance scales because edges follow in place (no O(E) teardown) and the sync writes only deltas (no O(N) transform re-stamp). The monolith's two megafunctions are replaced by small single-responsibility units, so growth is additive, not edit-in-lockstep.
- **Testability:** The pure layer expands and stays unit-tested with no simulator: SpatialLayoutEngine (type-cluster + new stepGraph strategy) and VolumeMetrics remain nonisolated Foundation/simd, and SpatialTokens/SpatialMotion become nonisolated value types with tested derivations (point<->meter bridge, normalized scale, color bridge). ### Treat the RealityKit scene as a RETAINED render target that a SwiftUI-driven declarative reconciler keeps in sync with the @Observable data model — exactly the React-Three-Fiber discipline, ported to visionOS 26. The reconciler is a keyed differ that writes ONLY the properties it owns; the single property whose ownership is contested (an entity's transform) is governed by an explicit, per-entity OWNERSHIP TOKEN that lives in the @Observable model (never in SwiftUI @State, never inferred from the scene). At any instant exactly one authority — data, gesture, or animation — owns each transform, and the reconciler defers to whoever holds the token. We split a "target transform" (what the data/layout says) from the "presented transform" (what is drawn) and put a thin interpolation layer between them so layout changes glide instead of snapping and never fight a live drag. Content stays 100% SwiftUI (ViewAttachmentComponent); RealityKit only places entities, routes gestures, and runs the interpolation/jitter/edge Systems. The pure nonisolated SpatialLayoutEngine and VolumeMetrics are untouched — they remain the only source of target positions. Everything else (collision, hover, animation, glass) is derived from one nonisolated SpatialTokens table so the system scales by adding token rows and entity kinds, not by editing two megafiles.
> A keyed declarative reconciler that writes only what it owns, arbitrated by per-entity transform-ownership tokens in the @Observable model, with a target-vs-presented interpolation layer — so the data model, live gestures, and animations never fight over an entity's transform, and hover/highlight is a presentation concern decoupled from layout.

- **Transform ownership:** Each entity carries an ownership TOKEN — TransformOwnership { case data, gesture, animating } — stored in the @Observable model (viewModel.ownershipByID[id]) AND mirrored on the entity as TransformTargetComponent.ownership for Systems to read cheaply. Critically the AUTHORITATIVE token lives in the model, NOT in SwiftUI @State, which is exactly why the current draggingID guard fails: @State mutated inside a RealityKit gesture .onChanged is not reliably visible to the next RealityView.update pass, but an @Observable model field set inside the gesture closure IS observed and IS visible to the reconciler. Rules: (1) DATA owns at rest and on creation/relayout — only then may the reconciler write the presented transform (indirectly, by setting target + letting TransformFollowSystem glide). (2) GESTURE takes ownership the instant a drag/manipulation begins (ManipulationEvents.WillBegin → viewModel.claim(id, .gesture)); while held, ManipulationComponent owns the live transform, TransformFollowSystem skips the entity, and the reconciler skips it. (3) ANIMATING owns during an explicit Entity.animate transition (entrance, return-to-tray, expand/collapse), set by whoever starts the animation and cleared on completion. Handoff is explicit and unidirectional: WillEnd persists the final transform, writes it as the new target, then releases to .data. There is never a moment where two authorities write the same transform, because the reconciler's write path is gated `guard ownershipByID[id] == .data`. This single guard, reading observed model state, is the root-cause fix.
- **Reconciliation:** A keyed differ (React-Three-Fiber applyProps discipline) run from RealityView.update, but it touches ONLY properties it owns and never re-asserts contested transforms. Algorithm per pass: (1) Compute liveIDs = Set(renderedEntities.id) and existing scene ids. (2) ADD: for each new id, build the entity once via SpatialEntityFactory, attach its ViewAttachmentComponent, set name=id, wire InputTarget/Collision/Hover/role components ONCE, set ownership=.animating and run an entrance Entity.animate (scale 0.85→1, opacity ramp), then set target transform. (3) REMOVE: for ids gone from liveIDs, run an exit animation then content.remove on completion (no instant pop). (4) RETAIN: for surviving entities, write the TARGET transform into TransformTargetComponent.target — but ONLY if ownershipByID[id] == .data; gesture/animating entities are skipped so a live drag is never overwritten. The reconciler NEVER sets entity.position directly and NEVER regenerates CollisionComponent or HoverEffectComponent per pass (they are created once; collision is sized from SpatialTokens card dimensions, not visualBounds, killing the recursive-bounds drift and per-pass churn). Edges are diffed by stable composite key (sourceID+targetID+kind) via EdgeComponent: add only new edge entities, remove only gone ones; endpoints are repositioned every frame by EdgeFollowSystem reading live transforms — replacing the current delete-and-re-add-all teardown. Because the reconciler is now a small pure-ish diff (previous live set + new model → add/remove/target-write list), the diff function is extracted as a nonisolated testable reducer (DiffResult) and unit-tested without the simulator.
- **Interaction:** Replace the two top-level .targetedToAnyEntity() gestures + entity.name string parsing with typed, per-entity routing. Every entity carries SpatialRoleComponent {kind, role, ownerId}; entity.name is demoted to being ONLY the stable id. Movable cards (taskNode/stepNode/note/workflowVolume) get ManipulationComponent via ManipulationComponent.configureEntity(entity, hoverEffect:.highlight, allowedInputTypes:.indirect, collisionShapes:[tokenBox]) with releaseBehavior=.stay (default .reset IS a built-in snap-back). Lifecycle: subscribe to ManipulationEvents.WillBegin → viewModel.claim(id,.gesture); DidUpdateTransform → clamp to VolumeMetrics.clamp in place (constrain, don't replace); WillEnd → read final transform, viewModel.endDrag(persist + mark hand-moved), set it as the new target, release to .data. This deletes dragGesture, dragStart, draggingID, and the manual value.convert(.local,.scene) math (ManipulationComponent handles re-orientation and coordinate space). Tap-to-edit, the × dismiss, and connect/link picks are per-entity GestureComponent(TapGesture()) handlers that read SpatialRoleComponent — no central switch, no prefix parsing. Connect/link become an explicit interaction-mode on the @Observable model (idle/connect/link + source id) so entering a mode disables manipulation and tap-to-edit, and candidate highlight is driven by observed state, not a transient @State bool. New control kinds (ports, context menus) attach their own GestureComponent — the dispatcher never grows.
- **Hover/highlight:** Hover/highlight is a PRESENTATION concern that lives in the presented layer and is fully decoupled from layout/target transform — so it composes with (never overwrites) position. Three mechanisms, used deliberately: (1) System gaze hover = RealityKit HoverEffectComponent(.highlight(HighlightHoverEffectStyle(color: typeTint, strength:))) on the entity, set once at creation, with a distinct HoverEffectComponent.GroupID per role so a card and its × badge activate INDEPENDENTLY (hover does not cascade to children and does not bleed between them). The × badge gets its own GroupID — this plus the collision fix is the root cause fix for bug #2. (2) App selection = an InteractionState enum {rest,hover,selected,picked,dragging,disabled} sourced from observed model state (selection set, interaction mode), passed into SpatialCard, which renders a tokenized border/lift — never derived from system hover (which is privacy-applied and unreadable by the app). The current merge-only isHighlighted bool becomes one case of this enum. (3) Playful jitter = NOT a SwiftUI .scaleEffect inside the attachment (that scales flat 2D content and is unreliable for a placed entity). Instead JitterSystem reads HoverStateComponent.isHovered and applies a low-amplitude spring scale/position offset on top of the presented transform via deltaTime integration over a clean base, so jitter and layout coexist. The DismissBadge .hoverEffect is removed. All jitter is gated behind @Environment(\\.accessibilityReduceMotion) with a tint/opacity crossfade fallback (ADHD audience).
- **Collision:** Make hierarchy explicit instead of accidental. Root cause of bug #2: the card's CollisionComponent is inflated to depth max(z,0.03)≈3cm and the × child sits only +0.006m in front, so the parent box geometrically encloses the child and (per Apple DTS forum 747256) the parent with an InputTargetComponent is always hit first — the badge never gets hover/tap. Fixes, all token-driven: (a) Size the card collider from SpatialTokens card dimensions (NOT visualBounds — eliminating the per-pass recursive-bounds drift) with a SHALLOW depth (~0.006m) matching a flat SwiftUI quad, so it cannot enclose forward controls. (b) Offset the × control to a real gap (~+0.025–0.03m, not +0.006m) clearly in FRONT of the card box, with its own small non-enclosed collider, OR move it just outside the card's xy footprint. (c) Define named CollisionGroups (.node, .control, .edge) with CollisionFilter so hit-testing routes by category and decorative edges/ports carry NO InputTargetComponent and are excluded from input masks — gesture correctness stops depending on string-prefix exclusion. (d) Set collision ONCE in the factory; never rebuild per reconcile pass. For precise frontmost resolution (connect/link targeting) use scene.raycast(..., mask:.node) instead of gesture target order.
- **Animation:** Two clearly separated animation paths. (1) Implicit transitions own the transform briefly via ownership=.animating: entrance (scale 0.85→1 + opacity, on add), exit (shrink/fade before content.remove), return-to-tray (fly from current to new column slot), expand/collapse pop-out (steps emerge from the volume position to their stepGraph slots with a staggered spring). Each runs Entity.animate(.spring/.easeInOut, duration: SpatialTokens.motion.X) inside the reconciler/event closure (never inside the update body re-reading observed state — that is the WWDC25 274 infinite-loop trap), claims .animating, and releases to .data on completion. (2) Continuous interpolation is the standing layer: TransformFollowSystem glides presented→target with a frame-rate-independent critically-damped follow (deltaTime-based, Rory Driscoll/spring-roll-call), so any change to a data-owned target animates automatically and layout reflow is never a teleport. The wire scaleProperty→sizeMultiplier feature gets wired here (animate scale changes) or deleted as dead code. All durations/easings/spring params come from one SpatialTokens.Motion table; every animation reads reduceMotion and degrades to a crossfade.
- **Material/design system:** One nonisolated SpatialTokens enum (Foundation/SwiftUI only, mirroring the VolumeMetrics 'single authority, no scattered magic numbers' precedent) is the single source of truth for: per-kind card dimensions (meters AND points, bridging the two domains so collision and layout derive from the same width), corner radii, paddings, border widths (rest vs selected), glass material level per elevation tier (HIG: typePanel=.regular section, node/note=.thin interactive, workflowVolume=.thick recede), tint roles, motion durations/easings, hover styles, and semantic edge colors (dependency, link) derived ONCE so the RealityKit UnlitMaterial(applyPostProcessToneMap:false) edges match the SwiftUI palette instead of today's disconnected .systemTeal/.systemOrange/.yellow literals. A single SpatialCard<Content> container applies glass + tint + radius + InteractionState border/lift; the four card views become content-only and stop hardcoding 16/18/14/16 radii and 210/240/190/200 widths. Glass uses .glassBackgroundEffect()/Liquid Glass consistently with vibrancy tiers (.primary/.secondary/.tertiary) instead of solid color washes; respects accessibilityReduceTransparency/increaseContrast (swap to opaque high-contrast surface) and conveys state redundantly (lift + symbol, not color alone).
- **Fixes drag bug:** Root cause today: reconcile() runs from RealityView.update on every @Observable change and unconditionally re-asserts card.position from the STALE stored model position (written only on release), gated only by a SwiftUI @State draggingID that the update closure cannot reliably see mid-gesture (the developer's failed fix). Two compounding faults: the model fights the live gesture, and the guard races. The new architecture eliminates both at the root: (1) Transform ownership is an @Observable model token. The instant a drag begins, ManipulationEvents.WillBegin calls viewModel.claim(id, .gesture) — set in the gesture/event closure, this mutation IS observed, so the very next reconcile pass sees ownershipByID[id]==.gesture and SKIPS the entity (the reconciler's write is `guard ownershipByID[id]==.data`). There is nothing to race because the authority is observed model state, not view @State. (2) The reconciler no longer writes transforms at all during a drag — ManipulationComponent owns the live transform and TransformFollowSystem also skips .gesture entities, so presented==what your hand is doing. (3) On release, WillEnd persists the final transform, writes it as the new TARGET, and releases to .data; the follow system is already at that target so there is no jump. - **Fixes hover bug:** Root cause today: the × badge sits +0.006m inside a card collider inflated to depth max(z,0.03)≈3cm, so the parent encloses the child and the parent (with InputTargetComponent) is hit first — the badge never receives gaze, so its SwiftUI .hoverEffect scaleEffect never fires; and even if it did, a SwiftUI scale inside a ViewAttachment is not a dependable RealityKit hover for a placed entity. (2) The × control is offset to a real ~+0.025m gap clearly in front of (and/or outside) the card box, with its own small non-enclosed collider in CollisionGroup .control, so gaze resolves to the badge. (3) Highlight is driven by a RealityKit HoverEffectComponent(.highlight(...)) on the badge entity with its OWN HoverEffectComponent.GroupID distinct from the card's, so it activates independently and hover does not cascade from/swallow the parent. ### HYBRID LAYERED: MVU for state, a thin ECS only for genuinely runtime concerns, and a design-token system for visuals — with one inviolable rule that dissolves both bugs: EXACTLY ONE AUTHORITY OWNS AN ENTITY'S TRANSFORM AT ANY INSTANT, and that authority is encoded in a RealityKit Component (TransformAuthorityComponent { .data | .gesture | .animating }) that a System reads, never in SwiftUI @State and never in the RealityView update closure. Data flows DOWN unidirectionally: the Domain store (@Observable, data only) → a Spatial Scene Model that reconciles data into a KEYED set of SpatialComponent view-models, each carrying an explicit interaction state machine (rest/hover/selected/dragging/disabled) → a thin reconciler that only adds/removes/re-attaches entities and writes a *target* transform + intent into components (never the live transform of a gesture/animation-owned entity) → a small set of MainActor RealityKit Systems that own all continuous transform evolution (DragFollowSystem, LayoutTweenSystem, HoverJitterSystem, EdgeSystem). Intents flow UP: gestures and ManipulationComponent events emit typed intents the Scene Model reduces into store writes. Visuals are a PURE FUNCTION of interaction state via a SpatialTokens design system. The pure SpatialLayoutEngine stays nonisolated/Foundation+simd-only and produces TARGETS, never touching entities. This is MRTK3's 'interactables are pure state, a separate visual driver renders' fused with WWDC25 274's 'mutate transforms only in gestures, Systems, or Entity.animate — never in update'.
> Make transform ownership a typed Component that one System honors at a time, shrink the RealityView update closure to identity-only reconciliation, render every visual as a pure function of an explicit per-entity interaction state machine fed by typed intents — and grab-drag plus child-control hover become trivial because nothing fights the gesture and nothing is geometrically swallowed.

- **Transform ownership:** Single-writer ownership encoded as TransformAuthorityComponent { owner: .data | .gesture | .animating } on every movable entity. RULE: a layer may write an entity's TransformComponent ONLY when it is the current owner. (1) .data (rest/at-target): the reconciler may set LayoutTargetComponent; LayoutTweenSystem interpolates the entity toward the target via Entity.animate or a critically-damped per-frame follow using context.deltaTime. The reconciler NEVER writes the live transform directly — it writes a target component, and only for .data-owned entities. (2) .gesture (active drag): ManipulationComponent.configureEntity(entity, releaseBehavior: .stay, dynamics: translate-biased, collisionShapes:[tokenBox]) owns the transform; on ManipulationEvents.WillBegin DragFollowSystem flips owner→.gesture; DidUpdateTransform clamps to VolumeMetrics by constraining (not replacing) the value; WillEnd reads the final entity transform, emits Intent.dragEnded(id, transform) which the SceneModel reduces into a DomainStore persist + manuallyMovedIds insert, then flips owner→.data. (3) .animating (entrance/exit/return-to-tray/pop-out): Entity.animate(.spring) owns the transform; owner→.animating for the animation duration, restored to .data on completion. Because ownership lives in a Component the update closure can read but the gesture sets directly on the entity, there is no @State visibility race and no two-writer conflict. The committed position in DomainStore is written ONLY on WillEnd, so it is never stale-read mid-drag.
- **Reconciliation:** Identity-only, diff-driven, idempotent. The RealityView update closure calls SceneModel to compute a SceneDiff (pure: previousKeyedVMs + currentDomainSnapshot → {added, removed, contentChanged, targetChanged}) and then performs ONLY structural mutations: (1) removed → animate-out then content.remove on completion; (2) added → one-time EntityFactory.makeCard(vm) wiring ViewAttachmentComponent + a token-sized CollisionComponent (computed ONCE from SpatialTokens dimensions, never from per-pass visualBounds) + InputTargetComponent(allowedInputTypes:.indirect) + HoverEffectComponent(.highlight, GroupID) + SpatialRoleComponent + TransformAuthorityComponent(.data) + LayoutTargetComponent, then entrance-animate; (3) contentChanged → swap only the attachment rootView / re-key its content (no transform touch); (4) targetChanged → write LayoutTargetComponent.target ONLY for .data-owned entities (LayoutTweenSystem does the motion). The closure NEVER writes a live TransformComponent and NEVER re-asserts position every pass — eliminating the WWDC25-274 observed-state feedback loop. Edges are NOT rebuilt here: EdgeSystem maintains persistent, stably-keyed (sourceId+targetId+semantic) edge/port entities and repositions endpoints in place each frame from live positions, gated by an edgesDirty flag the SceneModel sets only on topology change.
- **Interaction:** Typed, per-entity, intent-based — no name parsing. SpatialRoleComponent { kind, role, ownerId } replaces the 'ctl::role::owner'/'edge::' string protocol; entity.name is ONLY the stable id. Movement is owned by ManipulationComponent (releaseBehavior:.stay) so there is no hand-rolled DragGesture, no value.convert math, no dragStart/draggingID @State. Discrete actions use visionOS 26 GestureComponent attached to specific entities: a card carries TapGesture→Intent.tapped(id); the × control carries its OWN TapGesture→Intent.dismissTapped(id); edges/ports carry no input. Hits route by CollisionGroup mask (.node/.control/.edge) as a backstop. All gesture closures emit a typed Intent (never mutate data or transform directly); SceneModel.reduce(state, intent) is a pure function that updates InteractionMode/InteractionState and dispatches DomainStore actions. Merge/Link cease to be view @State booleans — they become InteractionMode in the DomainStore, reduced from intents, so candidate highlighting is driven by observed selectionRole (not a transient flag) and entering a mode disables tap-to-edit/manipulation so a pick can't be misread.
- **Hover/highlight:** Two strictly separated systems, both RealityKit-visible. (A) SYSTEM HOVER (gaze, applied out-of-process for privacy): every interactive entity gets HoverEffectComponent(.highlight(HighlightHoverEffectStyle(color: typeTint, strength:))) with a distinct HoverEffectComponent.GroupID per entity so a card and its × activate INDEPENDENTLY (hover does not cascade to children). The × control is its own entity with its own non-enclosed collider and its own GroupID — so it actually receives gaze. The SwiftUI .hoverEffect{scaleEffect} buried in DismissBadge is DELETED (it scales flat 2D content, not the placed entity, and is gated by a hit the parent swallowed). (B) APP SELECTION/FOCUS (fully app-owned): InteractionState (rest/hover/selected/dragging/disabled) on each SpatialComponentVM drives the SwiftUI card style (border weight, tint, elevation lift) via SpatialTokens — selectionRole replaces the merge-only isHighlighted bool and now expresses picked-for-merge/picked-for-link/selected/editing uniformly across all four card kinds. The two never read each other (you cannot read system hover in app code).
- **Collision:** Hierarchical hit-testing designed, not accidental. ROOT CAUSE of bug #2 fixed three ways together: (1) the card's CollisionComponent depth is sized to the card's real thickness (~0.006–0.01m) from SpatialTokens, NOT inflated to max(z,0.03) — so it no longer encloses anything in front of it; (2) the × control is promoted to a SIBLING-level entity (not a child buried at +0.006z) positioned with a real z-gap clear of the card box, with its OWN token-sized CollisionComponent + InputTargetComponent + GestureComponent; (3) CollisionGroup/CollisionFilter categories (.node/.control/.edge) plus per-entity GestureComponent give the control intrinsic, type-safe targeting. Card colliders are generated ONCE at creation from token dimensions (no per-pass visualBounds → kills the recursive-bounds drift/feedback trap) and cached. Edges/ports get no InputTargetComponent and live in the .edge group excluded from all input masks, so they are structurally non-interactive rather than excluded by name prefix. allowedInputTypes:.indirect on cards/controls matches the comfortable gaze+pinch modality.
- **Animation:** All motion is owned by the .animating authority or by Systems, never by direct assignment in update. (1) Transitions use visionOS 26 Entity.animate(.spring/.easeInOut) { entity.transform = target } in System/event closures — entrance (scale 0.85→1.0 + opacity ramp on add), exit (shrink/fade before content.remove), return-to-tray (fly current→new column slot), workflow expand/collapse (steps emerge from the volume, staggered, with WorkflowVolumeCardView's now-real isExpanded), drag release settle. (2) LayoutTweenSystem glides .data-owned entities toward LayoutTargetComponent.target each frame (frame-rate-independent damped follow with context.deltaTime) so layout reflow animates instead of teleporting. (3) HoverJitterSystem reads HoverStateComponent and applies a low-amplitude sinusoidal offset + small spring pop over a CLEAN base transform (jitter never persisted, no drift) — this is the playful highlight. All durations/easings/amplitudes come from SpatialTokens.Motion; every animation is gated behind @Environment(\\.accessibilityReduceMotion) with a crossfade/static fallback, and JitterSystem is suppressed when reduce-motion is on (important for the ADHD audience). scaleProperty/sizeMultiplier is either wired through LayoutTweenSystem (animated scale on change) or deleted as dead code.
- **Material/design system:** SpatialTokens: a single nonisolated enum (Foundation/SwiftUI only, mirroring the VolumeMetrics 'no scattered magic numbers' precedent) that is the source of truth for visuals, replacing the per-view hardcoded 16/18/14 radii, 210/240/190/200 widths, 0.25/0.35/0.5/0.6 opacities, and literal .yellow. It holds: per-kind card dimensions (in points AND derived meters, bridging L4↔L5), corner radii, glass material LEVEL per elevation tier (HIG semantics: typePanel=.regular section, node/note=.thin interactive, volume=.thick recede, all via .glassBackgroundEffect() for spatial grounding), semantic colors (selection, dependencyEdge, linkEdge, noteAccent) exposed as both SwiftUI Color and RealityKit UIColor (edges use UnlitMaterial(color:, applyPostProcessToneMap:false) so mesh colors match cards), border weights per InteractionState, elevation lift per state, and Motion (durations/easings/spring params/jitter amplitude). A pure InteractionState→CardStyle function makes the SpatialCard primitive render any state consistently. Honors accessibilityReduceTransparency (swap to opaque/high-contrast surface) and conveys state redundantly (lift + border, never color alone).
- **Fixes drag bug:** Root cause today: reconcile() re-asserts card.position from the stale stored position on EVERY update pass, guarded only by a @State draggingID that the update closure cannot reliably see mid-gesture (SpatialSceneView.swift:246–250). Three layers kill it: (L3) ManipulationComponent owns the transform during the drag (authority=.gesture) so RealityKit, not the model, drives position — and releaseBehavior=.stay means no built-in reset either. (L2/L3) the reconciler is forbidden from writing a live transform; it writes LayoutTargetComponent only for .data-owned entities, and a .gesture-owned entity is structurally skipped by ownership, not by a racy flag — so there is nothing to fight the gesture. (L1) the committed DomainStore position is written ONLY on ManipulationEvents.WillEnd, so it is never stale-read during the drag and the on-release re-assert/feedback path is gone entirely. Snap-back is eliminated by construction: at every instant exactly one writer owns the transform.
- **Fixes hover bug:** Root cause today: the × badge is a child only +0.006m in front of a card whose CollisionComponent is inflated to depth max(z,0.03)≈3cm, so the parent collider geometrically encloses the badge and (per Apple DTS forum 747256) the parent is always hit first — the badge never receives gaze/tap, so its SwiftUI .hoverEffect scaleEffect never fires. Fixed at the root: (collisionModel) the card collider is sized to real card thickness (~0.006–0.01m) so it stops enclosing anything; the × is promoted to a sibling-level entity with a real z-gap and its OWN token-sized non-enclosed CollisionComponent + InputTargetComponent. (hoverHighlightModel) the × gets a RealityKit HoverEffectComponent(.highlight) with its OWN HoverEffectComponent.GroupID so it activates independently of the card (hover does not cascade), and the unreliable SwiftUI .hoverEffect inside the attachment is deleted. (interactionModel) the × carries its own GestureComponent(TapGesture)→Intent.dismissTapped(id), routed by CollisionGroup .control mask — so its tap is intrinsic, not dependent on the parent or on name parsing. It now both highlights and dismisses reliably.
- **Scalability:** Adding a new entity kind or control is declarative, not surgical. A new kind = one SpatialComponentVM case + one SwiftUI content view wrapped in SpatialCard (tokens give it consistent glass/state visuals for free) + one EntityFactory recipe; it automatically gets collision/input/hover/authority/manipulation wiring, participates in the diff reconciler, and is laid out by adding a strategy to the pure engine. A new control = a sibling entity with a SpatialRoleComponent(role:.control) + GestureComponent emitting a new Intent case — no edits to a megafunction or a name parser. New interaction modes are InteractionMode cases reduced purely. New runtime behavior is a new System + marker Component, queried by EntityQuery, with zero changes to reconcile. Because behavior is sliced by lifetime (data→store, structure→reconciler, continuous→Systems, visuals→tokens) and dispatch is typed, complexity grows linearly with features instead of concentrating in two 600-line files. Edges scale via in-place EdgeSystem updates (O(changed) not O(all) per pass).
- **Testability:** High, because the load-bearing logic is pure and nonisolated. (1) SceneReducer.diff(old,new) and reduce(state,intent) are pure functions (keyed VMs + snapshot → SceneDiff; state + Intent → state + actions) unit-tested with no RealityKit/simulator — the dirty-diff and the merge/link/dismiss/drag-end state transitions all get regression tests. (2) SpatialLayoutEngine (type-cluster + new stepGraph topological layout) and VolumeMetrics stay nonisolated/Foundation+simd-only and get full coverage (currently zero in-target). (3) InteractionState transitions and InteractionState→CardStyle token mapping are pure and tested. (4) DomainStore actions are tested against a mock SpatialSceneService. RealityKit Systems and visuals are verified via xcodebuild -scheme TaskPlannerVision -destination 'generic/platform=visionOS Simulator' + Simulator (trusting xcodebuild over SourceKit), since Systems/hover/manipulation are not Vitest-able. Every new pure function gets a test per CLAUDE.md; the bug fixes get regression tests on the reducer (e.g. 'dragEnded persists exactly once and flips authority back to .data', 'reconciler never targets a .gesture-owned id').
- **Tradeoffs:** More types and files than the current two-file approach (DomainStore, SceneModel, 4 Systems, ~6 Components, SpatialTokens, SpatialCard, EntityFactory) — deliberate: the cost buys single-writer transform safety, typed dispatch, and pure testability, and the research is unanimous that the monolith is the root of both bugs and the scaling wall. Choosing ManipulationComponent over a hand-rolled gesture trades some fine-grained control (its translation/clamping is partly internal) for correctness and triviality, mitigated by DidUpdateTransform clamping and the DragGesture fallback flag. Using a thin ECS only for genuinely-runtime concerns (not a full ECS-for-everything) keeps MVU's predictable unidirectional flow while getting per-frame Systems where they're actually needed (jitter/edges/tween) — avoiding the opposite failure of pushing identity/data into Systems. Tokens add an indirection layer over raw .background calls, but that indirection is exactly what makes global glass/accessibility/elevation changes one-line instead of four-file. Keeping the layout engine pure and nonisolated under SWIFT_DEFAULT_ACTOR_ISOLATION=MainActor requires explicit nonisolated annotations and plain-projection inputs (a small ergonomic tax) but preserves deterministic unit testing — a constraint we honor rather than fight.
- **Layers:**
  - *L1 — Domain Store (DomainStore, @Observable, MainActor)* — Pure data. Owns the SpatialScene, [SpatialEntity], [SpatialLink], tasksById/stepsById/userTaskTypes, manuallyMovedIds, and active interaction MODE (none/merge/link with picked source). The single source of truth for DATA — it never knows about RealityKit entities, transforms-in-flight, or rendering. All tRPC persistence (createEntity, batchUpdateEntityTransforms, connect, link, collapseWorkflow, setRendered) lives behind async actions here. This is today's SpatialSceneViewModel split in two: the data/CRUD/persist half. Crucially it stores positions only as the *committed* (persisted) transform; the live/animated transform never round-trips through it mid-interaction (kills the snap-back feedback path at the source). (owns: SpatialScene, [SpatialEntity] (committed transforms only), [SpatialLink], task/step/type indices, manuallyMovedIds, InteractionMode, all tRPC writes.)
  - *L2 — Spatial Scene Model (SceneModel, @Observable, MainActor)* — The MVU 'view' of the scene: reconciles DomainStore data into a stable, KEYED [String: SpatialComponentVM] (id → view-model). Each SpatialComponentVM is the per-entity record holding: kind, refId, content snapshot (title/type/duration/note), the *target* transform (from layout engine or committed position), the InteractionState state machine value, selectionRole, and a contentVersion. SceneModel diffs old vs new keyed set and emits a SceneDiff { added, removed, contentChanged, targetChanged } — a PURE, unit-testable reducer (previous keyed set + new domain snapshot → diff). It also reduces typed Intents (from L3) back into DomainStore actions and InteractionState transitions. It runs the layout engine (L5) to compute targets. Nothing here touches RealityKit. (owns: [String: SpatialComponentVM] keyed view-models, the InteractionState state machines, SceneDiff computation (pure), Intent reduction, layout-target computation.)
  - *L3 — RealityKit Systems + Components (thin ECS, MainActor)* — The ONLY code that writes a live entity transform. A tiny component vocabulary + 4 Systems own all continuous/runtime behavior. TransformAuthorityComponent is the linchpin: it marks who owns this entity's transform RIGHT NOW. The reconciler sets a LayoutTargetComponent and SystemS move the entity toward it ONLY when authority == .data. ManipulationComponent (visionOS 26) owns the transform during a drag (authority flips to .gesture on WillBegin, back to .data on WillEnd). HoverJitterSystem applies a low-amplitude spring offset on hover over a clean base. EdgeSystem repositions persistent, stably-keyed edge/port entities in place from live positions (no teardown). All Systems query by marker components via EntityQuery and tick in update(context:) with context.deltaTime. (owns: Live entity TransformComponent (during .gesture/.animating), edge/port entity positions, hover jitter offsets, layout tween interpolation. Component registration + System registration at app launch.)
  - *L4 — SwiftUI Content + Design Tokens (SwiftUI, MainActor)* — Renders ALL content (locked paradigm). Every card is one SwiftUI primitive — SpatialCard(kind:tint:state:content:) — whose glass material level, tint, corner radius, border, elevation lift, and hover/selection treatment are PURE FUNCTIONS of (kind, InteractionState) read from SpatialTokens. The four card bodies (Node/TypePanel/Note/WorkflowVolume) become content-only and wrap in SpatialCard. The × control becomes its OWN attachment entity (sibling-level, not buried in the card collider) styled by a Control token. SpatialTokens is a nonisolated enum (Foundation/SwiftUI only) holding meters dimensions, radii, material levels per tier, semantic colors (one source of truth bridged to both SwiftUI Color and RealityKit UIColor with applyPostProcessToneMap:false), motion durations/easings, jitter amplitude, and reduce-motion/reduce-transparency fallbacks. (owns: All SwiftUI views, SpatialCard primitive, DismissControl visual, SpatialTokens design system, edit/create sheets via PresentationComponent.)
  - *L5 — Pure Layout Engine (SpatialLayoutEngine + VolumeMetrics, nonisolated)* — Unchanged contract, extended scope. Stays nonisolated, imports only Foundation/simd, takes plain projections and returns [Placement] (TARGETS). Gains a stepGraph(strategy:) function for workflow pop-out (topological levels) so the hand-rolled stepRowPositions leaves the view model and becomes a pure, unit-tested engine strategy. VolumeMetrics also gains per-kind card dimensions (bridged from SpatialTokens) so layout spacing, collision sizing, and rendered size all derive from one place. The engine produces targets; L3's LayoutTweenSystem applies them. The engine NEVER touches an entity, satisfying SWIFT_DEFAULT_ACTOR_ISOLATION=MainActor by being explicitly nonisolated. (owns: Type-cluster layout, workflow step-graph layout, clamp/bounds math, target computation. Zero entity/RealityKit/MainActor dependency.)
- **Risks:** ManipulationComponent has a reported intermittent visionOS-26 bug where translation can stop app-wide until device restart — keep the hand-rolled DragGesture+ownership path behind a feature flag as a fallback, sharing the same TransformAuthorityComponent so the reconciler treats both uniformly.; visionOS 26-only APIs (ManipulationComponent, GestureComponent, PresentationComponent, ViewAttachmentComponent-as-component, Entity.animate, HoverEffect .highlight/GroupID, CoordinateSpace3D) are correct for the 26.2 target but newer/less battle-tested; the ECS foundations (System/Component/EntityQuery/HoverEffectComponent/InputTarget/Collision) are stable since visionOS 1, so anchor on those and isolate the 26-only pieces behind small adapters.; SourceKit reports false errors against the wrong SDK — every build claim must be validated with xcodebuild, not the editor.; Promoting the × to a sibling entity adds entity count and a second attachment per card; keep controls lightweight and pooled, and only attach when dismissEligible to avoid scene churn as the graph grows.; Migrating from the current two-file monolith is sizable; sequence it (Phase A: TransformAuthority + ManipulationComponent + identity-only reconcile to kill both bugs; Phase B: SpatialTokens + SpatialCard + state machine; Phase C: Systems for jitter/edges/tween + engine stepGraph) so each phase is independently shippable and verifiable.; Entity.animate ownership windows must restore authority to .data on completion or a crashed/cancelled animation could strand an entity as .animating; add a watchdog in LayoutTweenSystem that reclaims stale .animating entities past a max duration.
